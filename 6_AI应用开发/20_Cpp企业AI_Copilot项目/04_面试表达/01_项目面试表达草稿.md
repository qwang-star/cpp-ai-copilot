# 项目面试表达草稿

## 1. 三分钟介绍

```text
我做的是一个 C++ 实现的企业级 AI Copilot 系统，主要解决企业内部文档难查、流程问题反复问人工的问题。

系统分成两条主线：第一条是文档入库链路，用户上传制度文档后，后端先保存文件并在 MySQL 记录状态，然后通过 MQ 异步交给 Worker 解析、清洗、切分 chunk，再调用 Embedding 模型生成向量并写入向量数据库。第二条是在线问答链路，用户提问时，系统先做 JWT 鉴权和知识库权限过滤，然后对问题做向量化，进行向量检索和关键词混合检索，召回候选 chunk 后做 Rerank，把最相关的资料拼进 Prompt，再通过模型网关调用大模型，并用 SSE 流式返回带引用的答案。

除了知识库问答，我还做了 Tool Calling，比如用户问“我的报销单审批到哪了”，模型只负责判断要调用哪个工具和生成参数，真正的参数校验、权限判断和数据库查询都在后端完成。工程上我封装了模型网关，统一处理超时、重试、降级、限流、token 成本统计和调用日志。最后我用 Golden Dataset 做评测，对比不同检索策略的 Recall@K、答案正确性和引用准确率。
```

## 2. 项目亮点

```text
1. 用 C++ 做 AI 应用后端，体现高性能服务、网络、并发和工程能力。
2. 文档入库通过 MQ 异步化，避免上传接口被耗时解析阻塞。
3. RAG 链路包含权限过滤、混合检索、Rerank、引用和拒答，不是简单向量搜索。
4. SSE 实现流式输出，优化用户等待体验。
5. Tool Calling 查询真实业务表，体现 AI 和后端系统结合。
6. 模型网关统一治理模型调用，覆盖稳定性、成本和日志。
7. 用 Golden Dataset 评测，而不是凭感觉说效果变好。
```

## 3. 高频追问

### 为什么用 C++ 做这个项目？

```text
因为我的主要语言是 C++，我希望项目既能体现 AI 应用能力，也能体现后端基础能力。C++ 适合做高性能服务、并发控制和系统级优化。这个项目里我用 C++ 实现 API Server、SSE 流式返回、模型网关、数据库和缓存访问，把 C++ 后端能力和 RAG、Tool Calling 这些 AI 应用能力结合起来。
```

### 为什么文档处理要异步？

```text
文档解析、切分、Embedding 和写入向量库都比较耗时。如果上传接口同步等待，用户体验差，也容易占满 Web 线程。所以我用 MQ 做异步任务，上传时只记录文档状态并投递消息，后台 Worker 慢慢处理。这样既能提升接口响应速度，也能通过重试、死信队列和扩容 Worker 提高稳定性。
```

### 为什么向量库不能替代 MySQL？

```text
两者职责不同。MySQL 管结构化业务数据和事务，比如用户、文档状态、会话消息、模型调用日志；向量库管 chunk embedding 的语义检索。向量库适合根据语义找相似内容，但不适合承担复杂事务和业务状态管理，所以项目里二者是配合关系。
```

### RAG 为什么不能完全消除幻觉？

```text
RAG 能把相关资料提供给模型，降低模型凭空编造的概率，但不能保证完全没有幻觉。原因包括检索可能漏召回、召回资料可能不相关、Prompt 可能约束不够强、模型可能误读资料。所以我做了引用来源、资料不足拒答、Rerank 和评测集回归来降低风险。
```

### Tool Calling 怎么保证安全？

```text
我不让模型直接操作数据库，也不让模型决定权限。模型只输出工具名和参数，后端会根据工具 Schema 做参数校验，再根据当前用户身份做鉴权，只允许查询当前用户有权限的数据。工具调用还会记录审计日志。也就是说，模型是建议调用什么，真正执行由可信后端控制。
```
