1. 为什么有两个 AppConfig？
AppConfig AppConfigload(const stdstring& path)
第一个 AppConfig 是返回值类型：
AppConfig 函数名(...)
意思是：这个函数最后会返回一个 AppConfig 对象。
第二个 AppConfigload 是在说：
我现在实现的是 AppConfig 这个结构体里面声明过的 load 函数
 叫作用域解析运算符，可以理解成“属于谁”。
所以整句人话是：
实现 AppConfig 里面的 load 函数，这个函数返回一个 AppConfig 配置对象。
类似你写：
int add(int a, int b)
这里 int 是返回值。只不过现在函数属于 AppConfig，所以要写成：
AppConfigload
2. ifstream 是函数吗？input 是什么？
不是。stdifstream 是一个类，意思是 input file stream，输入文件流。
这句：
stdifstream input(path);
你可以类比成：
stdstring name(hello);
stdstring 是类型，name 是对象。
所以：
stdifstream input(path);
意思是：
创建一个叫 input 的文件读取对象，让它去打开 path 这个文件。
后面：
stdgetline(input, line)
就是从 input 这个文件流里一行一行读内容。
3. config.hpp 和 config.cpp 什么关系？
config.hpp 负责声明，也就是告诉别人“我有什么”。
struct AppConfig {
    stdstring host;
    int port;
    stdstring log_level;

    static AppConfig load(const stdstring& path);
};
它告诉外界：
AppConfig 有 host、port、log_level
AppConfig 还有一个 load 函数
但它没有写 load 具体怎么读文件。
config.cpp 负责定义，也就是真正把函数写出来：
AppConfig AppConfigload(const stdstring& path) {
    ...
}
可以理解成：
.hpp 是说明书目录
.cpp 是具体实现正文
为什么要分开？
因为别的文件，比如 main.cpp，只需要知道：
AppConfigload(...)
能用就行，不需要把读取配置文件的所有细节都塞进 main.cpp。
4. 为什么 include 要写 copilotconfig.hpp？
因为项目目录是这样的：
include
    copilot
    config.hpp
Makefile 里有：
INCLUDES = -Iinclude
这句话告诉编译器：
你找头文件时，可以从 include 目录开始找。
所以：
#include copilotconfig.hpp
实际找的是：
includecopilotconfig.hpp
这里的 copilot 是项目自己的头文件目录，和命名空间 namespace copilot 配合起来，表示“这是 copilot 项目的公共接口”。
5. 为什么函数只在头文件声明，定义放到源文件？
这是 C++ 项目里非常常见的写法。
头文件里：
static AppConfig load(const stdstring& path);
意思是：
有这个函数，你可以调用它。
源文件里：
AppConfig AppConfigload(const stdstring& path) {
    ...
}
意思是：
这个函数具体这样执行。
这样做的好处是：
main.cpp 不需要知道 load 里面怎么解析文件
config.cpp 可以独立修改实现
config.hpp 保持稳定接口
编译结构更清楚
这个 load 函数整体做的事其实很朴素：
创建默认配置
打开配置文件
一行一行读取
跳过空行和注释
按等号切成 keyvalue
把 APP_HOST、APP_PORT、LOG_LEVEL 写进 config
检查端口是否合法
返回 config
也就是把这种文件：
APP_HOST=127.0.0.1
APP_PORT=18080
LOG_LEVEL=info
变成 C++ 里的：
config.host = 127.0.0.1;
config.port = 18080;
config.log_level = info;
