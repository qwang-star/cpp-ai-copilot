# 08 模型网关、流式输出与稳定性

## 1. 为什么需要模型网关

如果业务代码到处直接调用模型 API，会出现：

- 超时策略不一致。
- 日志格式不一致。
- 成本无法统计。
- 模型切换困难。
- 降级困难。
- Prompt 和模型版本难管理。

模型网关的作用：

```text
统一模型调用入口，统一做路由、限流、重试、降级、日志、成本、监控。
```

## 2. 模型网关能力清单

必须掌握：

- 模型路由。
- 多供应商适配。
- API Key 管理。
- 超时控制。
- 重试和指数退避。
- 熔断。
- 降级。
- 流式响应转发。
- Token 统计。
- 成本统计。
- 请求日志。
- Prompt 版本记录。
- 输出格式校验。
- 敏感信息脱敏。
- 灰度发布。

## 3. 模型路由

路由维度：

- 任务类型：问答、摘要、抽取、分类、代码。
- 用户等级：免费、付费、内部测试。
- 成本预算。
- 延迟要求。
- 上下文长度。
- 模型可用性。
- 灰度策略。

例子：

```text
简单分类 -> 小模型
复杂推理 -> 强模型
长文档总结 -> 长上下文模型
结构化抽取 -> 支持结构化输出的模型
```

## 4. 流式输出

为什么需要：

- 大模型生成慢。
- 用户不想等完整答案。
- 流式输出能更早看到内容。

常见方式：

- SSE。
- WebSocket。

SSE 更适合：

- 服务端单向推送。
- 文本生成。
- 普通聊天。

WebSocket 更适合：

- 双向实时通信。
- 语音对话。
- 协作编辑。

## 5. SSE 后端链路

```text
客户端建立 HTTP 连接
  -> 后端设置 text/event-stream
  -> 调用模型 stream
  -> 收到 token delta
  -> 立即 flush 给客户端
  -> 结束时发送 done
  -> 保存完整回答
```

注意：

- 连接断开要停止上游模型调用或停止转发。
- 要处理代理服务器缓冲。
- 要设置心跳。
- 要限制最大连接时长。
- 要记录完整输出，不能只依赖前端拼接。

## 6. 超时与重试

超时分层：

- 网关总超时。
- 模型连接超时。
- 首 token 超时。
- 总生成超时。
- 工具调用超时。
- RAG 检索超时。

重试原则：

- 只对可重试错误重试。
- 使用指数退避。
- 设置最大重试次数。
- 注意幂等。
- 流式输出已经开始后，重试会变复杂。

## 7. 熔断与降级

熔断：

```text
当某个模型错误率或超时率过高，暂时停止调用它。
```

降级：

- 切备用模型。
- 切小模型。
- 返回缓存答案。
- 返回检索结果摘要。
- 提示稍后重试。
- 转人工。

## 8. 限流

限流维度：

- 用户。
- IP。
- 租户。
- 知识库。
- 模型。
- 接口。
- Token 数。
- 成本预算。

算法：

- 固定窗口。
- 滑动窗口。
- 漏桶。
- 令牌桶。

AI 应用特别要做：

```text
按 token 或费用限额，而不仅仅按请求次数限流。
```

## 9. 成本优化

策略：

- 模型分级。
- Prompt 压缩。
- 控制 RAG TopK。
- 缓存高频问题。
- Embedding 批处理。
- 只在必要时调用 Rerank。
- 失败重试有上限。
- 对用户设置预算。
- 记录 input_tokens 和 output_tokens。

## 10. 可观测性

一次 AI 请求至少记录：

```text
trace_id
user_id
tenant_id
request_id
model_name
prompt_version
input_tokens
output_tokens
latency_ms
first_token_latency_ms
retrieval_latency_ms
rerank_latency_ms
tool_call_count
error_code
cost
```

面试表达：

```text
我会把一次问答拆成检索、重排、模型调用、流式输出几个 span，这样能定位到底是向量库慢、模型慢，还是网络慢。
```
