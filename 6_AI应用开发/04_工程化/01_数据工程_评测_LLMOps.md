# 09 数据工程、评测与 LLMOps

## 1. 数据工程为什么重要

RAG 的上限常常不是模型，而是数据质量。

如果文档解析错、chunk 切坏、metadata 缺失，后面的向量检索和大模型生成都会被拖垮。

## 2. 数据处理流程

```text
采集
  -> 解析
  -> 清洗
  -> 去重
  -> 切分
  -> 标注 metadata
  -> 向量化
  -> 索引
  -> 质量检查
```

## 3. 数据版本管理

需要版本化的对象：

- 原始文档。
- 解析结果。
- chunk。
- embedding 模型。
- 向量索引。
- Prompt。
- 评测集。
- 模型版本。

为什么：

- 方便回滚。
- 方便复现实验。
- 方便对比效果。
- 方便定位线上问题。

## 4. 评测体系

AI 应用必须评测，否则优化只靠感觉。

评测分三类：

```text
离线评测
  -> 固定测试集，版本上线前跑

在线评测
  -> 真实用户反馈、A/B Test、点击、点赞点踩

人工评测
  -> 高质量但成本高，用于关键样本
```

## 5. Golden Dataset

Golden Dataset 是固定高质量评测集。

每条样本建议包含：

```text
question
expected_answer
relevant_doc_ids
relevant_chunk_ids
must_have_points
forbidden_points
difficulty
category
```

来源：

- 真实用户问题。
- 业务专家整理。
- 历史客服工单。
- 产品 FAQ。
- 线上 bad case。

## 6. RAG 评测指标

检索指标：

- Recall@K：正确 chunk 是否出现在前 K 个结果。
- Precision@K：前 K 个结果中有多少是相关的。
- MRR：第一个正确结果排得靠不靠前。
- NDCG：排序质量。

生成指标：

- Correctness：答案是否正确。
- Faithfulness：是否忠于检索材料。
- Completeness：是否完整。
- Citation Accuracy：引用是否准确。
- Hallucination Rate：幻觉率。
- Refusal Accuracy：该拒答时是否拒答。

工程指标：

- P50 / P95 / P99 延迟。
- 错误率。
- 超时率。
- Token 成本。
- 单次请求成本。
- 缓存命中率。
- 用户反馈率。

## 7. LLM-as-a-Judge

用大模型做裁判可以提高评测效率，但不能完全替代人工。

适合：

- 初筛。
- 大规模回归。
- 对比两个版本。
- 发现明显幻觉。

风险：

- 裁判模型也会错。
- 评分标准可能漂移。
- 对细粒度事实不一定可靠。

建议：

- 使用明确 rubric。
- 关键样本人工复核。
- 固定裁判 Prompt 和模型版本。
- 保留评分理由。

## 8. LLMOps

LLMOps 是围绕大模型应用的工程生命周期管理。

包括：

- Prompt 管理。
- 模型管理。
- 数据集管理。
- 评测流水线。
- 灰度发布。
- 监控告警。
- 成本分析。
- 用户反馈闭环。
- Bad case 分析。

闭环：

```text
线上日志
  -> 用户反馈
  -> Bad case 标注
  -> 加入评测集
  -> 优化 Prompt / RAG / 模型
  -> 离线评测
  -> 灰度发布
  -> 在线观察
```

## 9. 面试高频题

### 如何证明 RAG 优化有效？

回答：

- 建立固定评测集。
- 分别看检索指标和生成指标。
- 对比优化前后的 Recall@K、答案正确率、幻觉率、P95 延迟和成本。
- 对线上流量做灰度或 A/B Test。
- 收集用户反馈和 bad case。

### 为什么不能只看用户点赞点踩？

回答：

- 用户反馈稀疏。
- 反馈有主观性。
- 很多错误用户不会反馈。
- 无法覆盖边界场景。
- 所以需要离线评测集 + 在线反馈结合。
