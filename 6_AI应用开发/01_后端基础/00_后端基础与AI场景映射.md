# 02 后端基础与 AI 场景映射

## 1. 为什么后端基础仍然是主菜

AI 应用看起来新，但请求进入系统后，仍然要经过普通后端链路：

```text
HTTP 请求
  -> 鉴权
  -> 参数校验
  -> 业务逻辑
  -> 缓存 / 数据库
  -> 外部服务调用
  -> 日志
  -> 返回响应
```

区别在于：外部服务从支付、短信、地图，变成了 LLM、Embedding、OCR、向量数据库。

## 2. 计算机网络

必须掌握：

- TCP 三次握手和四次挥手。
- TCP 可靠传输、滑动窗口、拥塞控制。
- HTTP 请求方法和状态码。
- HTTPS 加密和证书。
- HTTP 长连接。
- HTTP/2 多路复用。
- RESTful API。
- RPC / gRPC。
- SSE。
- WebSocket。

AI 场景映射：

- 聊天接口要支持长文本返回，常用 SSE 做流式输出。
- 如果需要双向实时交互，可以用 WebSocket。
- 模型调用链路长，要设置连接池、超时、重试。
- 大文件上传要考虑分片、大小限制、断点续传。

高频追问：

```text
SSE 和 WebSocket 怎么选？
```

回答要点：

- SSE 是服务端到客户端的单向流，基于 HTTP，适合模型流式输出。
- WebSocket 是全双工，适合实时协作、语音对话、多人互动。
- 普通聊天生成答案，SSE 通常更简单。

## 3. 操作系统

必须掌握：

- 进程、线程、协程。
- 用户态和内核态。
- 上下文切换。
- 锁、死锁、线程安全。
- IO 模型：阻塞、非阻塞、多路复用、异步 IO。
- 文件系统和缓冲区。
- 内存管理。

AI 场景映射：

- 模型调用慢，不能让工作线程无限阻塞。
- 文档解析可能占 CPU，上传下载占 IO，要分清任务类型。
- 大文件不能一次性读进内存。
- 后台向量化适合线程池、协程或任务队列。

高频追问：

```text
为什么文档解析要异步？
```

回答要点：

- 文件解析和 Embedding 可能耗时数秒到数分钟。
- 同步处理会占用请求线程，导致接口超时和吞吐下降。
- 异步化后可以立即返回 task_id，后台处理，前端轮询或订阅状态。

## 4. MySQL

必须掌握：

- InnoDB。
- B+ 树索引。
- 聚簇索引和二级索引。
- 最左前缀。
- 覆盖索引。
- explain。
- ACID。
- 隔离级别。
- MVCC。
- 行锁、间隙锁、临键锁。
- redo log、undo log、binlog。
- 主从复制。
- 分库分表。

AI 场景映射：

- MySQL 存业务元数据，不适合直接做语义检索。
- 知识库、文档、chunk、会话、消息、模型调用日志都需要关系型表。
- 向量库里的 metadata 要能回到 MySQL 查原文和权限。

典型表：

```text
users
knowledge_bases
documents
document_chunks
conversations
messages
model_call_logs
tool_call_logs
evaluation_cases
evaluation_results
```

设计重点：

- document_chunks 保存 chunk_id、document_id、content、position、token_count。
- 向量库保存 chunk_id 和 metadata。
- 删除文档时，MySQL 和向量库要一致。
- 日志表要考虑按时间分区或冷热分离。

## 5. Redis

必须掌握：

- String、Hash、List、Set、ZSet、Stream。
- 过期策略。
- 淘汰策略。
- RDB / AOF。
- 主从、哨兵、集群。
- 缓存穿透、击穿、雪崩。
- 分布式锁。
- Lua 脚本。
- 限流。

AI 场景映射：

- 缓存热点知识库检索结果。
- 缓存高频问答结果。
- 保存用户短期会话状态。
- 做用户级、接口级、模型级限流。
- 防止重复提交文档解析任务。

常见问题：

```text
AI 问答结果能不能缓存？
```

回答：

- 高频、确定性、权限一致的问题可以缓存。
- 强个性化、强实时、涉及敏感权限的问题慎重缓存。
- 缓存 key 要包含 user_id、kb_id、query_hash、prompt_version、model_version 等影响结果的因素。

## 6. 消息队列

必须掌握：

- 异步、解耦、削峰。
- Topic、Partition、Consumer Group。
- 消息可靠性。
- 重复消费。
- 顺序消息。
- 延迟队列。
- 死信队列。
- 幂等。

AI 场景映射：

- 文档上传后异步解析。
- 文档解析后异步切分。
- chunk 批量 Embedding。
- 离线评测任务。
- 用户反馈分析。
- 生成长报告。

典型链路：

```text
POST /documents/upload
  -> 保存文件到对象存储
  -> 写 document 状态 uploaded
  -> 发送 parse_document 消息
  -> worker 解析
  -> 发送 embed_chunks 消息
  -> worker 向量化
  -> 写入向量库
  -> document 状态 ready
```

面试重点：

- 消息失败要重试。
- 重试多次失败进入死信队列。
- 消费者用 document_id + task_type 做幂等。
- 状态机要能表达 uploaded、parsing、embedding、ready、failed。

## 7. 分布式与稳定性

必须掌握：

- CAP、BASE。
- 负载均衡。
- 服务注册发现。
- 限流。
- 熔断。
- 降级。
- 分布式 ID。
- 链路追踪。
- 灰度发布。

AI 场景映射：

- 模型服务慢或挂，要降级。
- 大模型成本高，要限流。
- 一次问答跨多个模块，要 trace_id 串起来。
- Prompt 或模型版本上线要灰度。

典型 trace：

```text
trace_id
  -> api_request
  -> auth_check
  -> redis_cache
  -> vector_search
  -> rerank
  -> model_call
  -> stream_response
```

## 8. 后端基础和 AI 的关系

```text
RAG 准不准：AI 问题 + 数据工程问题
接口快不快：后端问题 + 模型延迟问题
系统稳不稳：分布式问题 + 第三方依赖问题
成本高不高：架构问题 + 模型选择问题
会不会泄露：权限问题 + Prompt 安全问题
```
