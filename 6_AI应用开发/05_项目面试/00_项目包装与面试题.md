## 5. 项目难点包装

### 难点 1：文档处理耗时

方案：

- 上传接口只保存文件和记录。
- 通过 MQ 投递解析任务。
- Worker 异步解析、切分、向量化。
- 用状态机记录进度。
- 失败进入重试和死信队列。

### 难点 2：检索不准

方案：

- 优化文档清洗和 chunk 切分。
- 使用向量 + 关键词混合检索。
- 召回后 Rerank。
- 控制 TopK。
- 建立评测集，用 Recall@K 和答案正确率验证。

### 难点 3：模型慢且贵

方案：

- 流式输出优化体感。
- 缓存高频问题。
- 模型分级调用。
- Prompt 压缩。
- 记录 token 成本。
- 对用户和租户设置限额。

### 难点 4：权限安全

方案：

- chunk metadata 写入 tenant、owner、role。
- 检索时做 metadata filter。
- 返回引用前校验权限。
- Tool Calling 前做后端鉴权。
- 日志脱敏。

### 难点 5：线上稳定性

方案：

- 模型网关统一超时、重试、熔断、降级。
- 第三方模型异常时切备用模型。
- RAG 召回为空时拒答。
- JSON 输出做 schema 校验。
- 全链路 trace_id。

## 6. 三分钟项目介绍

```text
我做的是一个企业知识库问答系统，目标是让员工可以基于公司内部文档进行自然语言问答。

系统分为文档处理链路和在线问答链路。文档上传后先存对象存储，MySQL 记录文档状态，然后通过 MQ 异步解析、清洗、切分并生成 Embedding，最后写入向量数据库。在线问答时，后端先做鉴权和限流，然后对用户问题做向量检索和关键词检索，合并后进行 Rerank，把最相关的文档片段拼入 Prompt，再通过模型网关调用大模型，使用 SSE 流式返回答案并附带引用来源。

项目重点解决了几个工程问题：文档处理异步化、RAG 检索准确率、权限过滤、模型调用稳定性和成本监控。我还设计了评测集，用 Recall@K、答案正确率、幻觉率和 P95 延迟来观察优化效果。
```

## 7. 高频项目追问

### 为什么用 MQ？

文档解析、OCR、Embedding 都是耗时任务。同步执行会导致接口超时和请求线程被占满。用 MQ 可以异步化、削峰、失败重试，也方便扩展 Worker。

### 为什么用 Redis？

用于缓存热点问答、保存短期会话状态、做限流、做任务幂等和分布式锁。

### 为什么向量库不能替代 MySQL？

向量库负责语义检索，MySQL 负责业务关系、事务、权限、状态和日志。两者职责不同，向量库里的 chunk_id 需要回到 MySQL 查业务元数据。

### 模型挂了怎么办？

模型网关做超时、重试、熔断和降级。可以切备用模型、返回缓存答案、返回检索材料摘要，或者提示用户稍后重试。

### RAG 如何评估？

分检索和生成两层。检索看 Recall@K、MRR；生成看答案正确率、忠实性、引用准确率、幻觉率；工程上看 P95 延迟、错误率和成本。
