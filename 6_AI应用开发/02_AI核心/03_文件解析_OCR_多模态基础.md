# 03 文件解析、OCR 与多模态基础

## 1. 为什么要学这一章

企业 AI 应用里，知识通常不是干净的纯文本，而是：

- PDF。
- Word。
- Excel。
- PPT。
- 图片。
- 扫描件。
- 网页。
- 代码仓库。
- 邮件。
- 工单。

RAG 做不好，很多时候不是模型不行，而是文件解析和数据结构处理太粗糙。

## 2. 文件解析目标

文件解析不是简单把文字扒出来，而是尽量保留：

- 标题层级。
- 段落。
- 表格。
- 列表。
- 页码。
- 图片说明。
- 代码块。
- 脚注。
- 文档来源。

理想中间格式：

```text
结构化 Markdown + metadata
```

metadata 包括：

- document_id。
- page。
- section_title。
- block_type。
- position。
- source。

## 3. PDF 解析

难点：

- PDF 本质更像版式文件，不是语义文档。
- 多栏排版会打乱阅读顺序。
- 表格容易变成散乱文本。
- 页眉页脚会重复出现。
- 扫描 PDF 没有文本层。

处理策略：

- 优先提取文本层。
- 对扫描件使用 OCR。
- 表格单独抽取。
- 保留页码。
- 去掉重复页眉页脚。
- 用标题规则恢复层级。

面试表达：

```text
PDF 解析质量直接影响 chunk 质量。如果表格、标题和段落结构丢失，向量检索召回的 chunk 就会语义不完整。
```

## 4. Word / Markdown / HTML

Word：

- 保留标题样式。
- 保留表格。
- 保留列表。
- 图片可抽取说明或 OCR。

Markdown：

- 最适合 RAG。
- 标题、列表、代码块结构清晰。

HTML：

- 去导航栏、广告、脚本。
- 保留正文。
- 保留链接。
- 提取标题。

## 5. Excel / 表格

表格处理是高频难点。

问题：

- 单个单元格脱离表头后没有语义。
- 合并单元格会影响结构。
- 多 sheet 要保留上下文。

处理方式：

- 表格转 Markdown。
- 每一行携带表头。
- 大表按行块切分。
- 保留 sheet 名和标题。

例子：

```text
Sheet: 差旅标准
表头: 城市级别 | 住宿标准 | 交通标准
行: 一线城市 | 600 元/晚 | 高铁二等座
```

## 6. OCR

OCR 用于图片和扫描件文字识别。

流程：

```text
图片预处理
  -> 方向矫正
  -> 版面分析
  -> 文字识别
  -> 表格识别
  -> 结果后处理
```

常见问题：

- 识别错字。
- 表格错位。
- 阅读顺序错。
- 图片清晰度低。
- 手写体困难。

工程建议：

- OCR 任务异步化。
- 记录识别置信度。
- 低置信度结果人工复核。
- 原图和识别文本都保留。

## 7. 多模态应用

多模态指模型能处理文本、图片、音频、视频等多种输入。

常见能力：

- 图片理解。
- 文档截图问答。
- OCR + 视觉理解。
- 语音转文字 ASR。
- 文字转语音 TTS。
- 视频摘要。

后端关注点：

- 文件上传。
- 对象存储。
- 异步任务。
- 文件大小限制。
- 转码。
- 任务状态。
- 成本控制。
- 内容安全。

## 8. 多模态 RAG

常见做法：

```text
图片 / PDF
  -> OCR 提取文字
  -> 图片生成 caption
  -> 文本和 caption 入向量库
  -> 原图位置作为引用
```

更高级：

- 文本向量和图片向量分别索引。
- 多路召回。
- 结果融合。
- 多模态模型生成答案。

## 9. 面试高频题

### 为什么文档解析会影响 RAG 效果？

因为 RAG 检索的是 chunk。如果解析阶段把标题、表格、段落顺序破坏了，chunk 语义就不完整，召回和生成都会受影响。

### 表格怎么做 RAG？

可以把表格转成 Markdown 或结构化行记录，每个 chunk 保留表头、sheet 名、标题和页码。对于大表可以按行块切分，并在 metadata 中记录位置。
