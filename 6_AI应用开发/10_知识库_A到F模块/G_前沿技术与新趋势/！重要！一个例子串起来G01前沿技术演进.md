# ！重要！一个例子串起来 G01 前沿技术演进

## 场景：用户在地铁里用企业 AI Copilot 问报销问题

用户手机网络很差，一会儿 Wi-Fi，一会儿 5G。

他问：

```text
我的报销单审批到哪了？顺便告诉我差旅报销还缺什么材料。
```

这个问题会同时触发：

```text
网络传输
RAG 检索
Tool Calling
模型调用
流式返回
可观测性
安全审计
```

这就是一个能把前沿技术全部串起来的例子。

---

## 读之前先把这篇的名词说清楚

### 1. QUIC

**新手理解**：QUIC 像一条更适合移动网络的新高速路。它基于 UDP，但自己实现可靠传输、加密、多路复用和连接迁移。

**在这个例子里**：用户从 Wi-Fi 切到 5G 时，QUIC 可以通过 Connection ID 尽量保持连接连续。

**面试说法**：QUIC v1 是基于 UDP 的安全多路复用传输协议，集成 TLS 1.3，减少 TCP 队头阻塞，并支持连接迁移。

### 2. HTTP/3

**新手理解**：HTTP/3 不是换掉 HTTP 的意思，而是 HTTP 的语义还在，底层从 TCP 换成 QUIC。

**在这个例子里**：手机弱网访问 AI Copilot 时，HTTP/3 可以改善丢包和切网下的体验。

**面试说法**：HTTP/3 把 HTTP 语义映射到 QUIC 上，解决 HTTP/2 仍受 TCP 队头阻塞影响的问题。

### 3. 0-RTT

**新手理解**：0-RTT 像老客户进店，老板已经认识你，可以更快开始服务。

**在这个例子里**：用户再次打开知识库查询时，读取类请求可以更快开始发送。

**面试说法**：0-RTT 能降低重连延迟，但有重放风险，所以更适合幂等读请求，不适合支付、审批、扣费等写操作。

### 4. WebTransport

**新手理解**：WebTransport 像 WebSocket 的更现代版本之一，既能可靠传数据，也能允许部分低价值数据丢掉来换低延迟。

**在这个例子里**：普通文字聊天用 SSE 就够；如果后续做语音、多模态实时协作，可以考虑 WebTransport。

**面试说法**：WebTransport 面向低延迟双向通信，支持 stream 和 datagram，适合实时多模态场景，但生态成熟度和基础设施支持要评估。

### 5. Hybrid Search

**新手理解**：Hybrid Search 是“语义搜索 + 关键词搜索”一起用。

**在这个例子里**：用户问“差旅报销材料”，向量检索能懂同义表达；报销单号、条款号、金额则需要关键词检索兜底。

**面试说法**：企业 RAG 里纯向量检索容易漏金额、编号和专有名词，dense + sparse hybrid search 能提升召回稳定性。

### 6. GraphRAG

**新手理解**：普通 RAG 是找几段话，GraphRAG 是先把文档里的实体和关系画成地图。

**在这个例子里**：如果用户问“整个报销流程涉及哪些部门”，GraphRAG 比只找几个 chunk 更适合。

**面试说法**：GraphRAG 适合跨文档、全局总结、实体关系推理，但构建和评测成本更高。

### 7. Structured Outputs

**新手理解**：Structured Outputs 是让模型按规定格式交作业，不要“差不多 JSON”。

**在这个例子里**：模型判断要调用 `get_reimbursement_status` 时，必须输出合法的工具名和参数。

**面试说法**：结构化输出能提高后端解析稳定性，但后端仍要做字段校验、权限校验和审计。

### 8. Prompt Caching

**新手理解**：固定的系统说明、工具说明别每次重新算，能复用就复用。

**在这个例子里**：AI Copilot 的 system prompt、工具 schema、输出格式长期不变，可以放在 Prompt 前缀提升缓存命中。

**面试说法**：Prompt Caching 通过复用稳定前缀降低延迟和成本，但要把动态内容放后面，并监控缓存命中率。

### 9. MCP

**新手理解**：MCP 像给模型接外部工具定了一套插头标准。

**在这个例子里**：报销系统、请假系统、知识库系统以后都可以用标准方式暴露给 Agent。

**面试说法**：MCP 能标准化模型和工具/数据源连接，但安全仍要靠后端鉴权、参数校验、最小权限和审计。

### 10. OpenTelemetry

**新手理解**：OpenTelemetry 像给一次请求装 GPS，能看到它经过哪些服务、每段花多久。

**在这个例子里**：用户说“回答好慢”，你可以看到慢在向量检索、Rerank、模型首 token 还是 SSE 返回。

**面试说法**：OTel 统一 traces、metrics、logs，适合 AI 应用做端到端可观测性。

### 11. eBPF

**新手理解**：eBPF 像 Linux 内核里的安全探针，不改业务代码也能看底层发生了什么。

**在这个例子里**：如果 OTel 显示应用层没问题，但网络仍然慢，可以用 eBPF 观察 TCP 重传、系统调用耗时、容器网络等。

**面试说法**：eBPF 适合内核级网络、性能、安全观测，但需要 Linux 内核支持，权限和复杂度较高。

### 12. io_uring

**新手理解**：io_uring 像用户态和内核态之间的两个任务托盘，一边放任务，一边拿结果，减少来回跑窗口。

**在这个例子里**：如果系统有大量文件上传、日志落盘、向量索引文件 IO，未来可以考虑 io_uring 优化。

**面试说法**：io_uring 用提交队列和完成队列组织异步 IO，能减少系统调用和上下文切换，但复杂度较高，适合高性能 Linux IO 热点。

## 0. 总流程

```text
用户手机弱网提问
  -> HTTP/3 / QUIC 建立低延迟连接
  -> 如果支持 0-RTT，幂等读请求更快开始
  -> 后端收到问题
  -> Hybrid Search 召回制度 chunk
  -> GraphRAG 可选处理全局关系问题
  -> Structured Outputs 判断是否要 Tool Calling
  -> 后端安全调用报销系统工具
  -> Prompt Caching 降低固定提示词成本
  -> LLM 生成答案
  -> SSE / 未来 WebTransport 返回
  -> OpenTelemetry 记录全链路
  -> eBPF 辅助排查底层网络和系统问题
```

---

## 1. 网络层：从 HTTP/2 演进到 HTTP/3

旧技术不是没用。

HTTP/2 已经解决了 HTTP/1.1 的很多问题：

```text
不用开很多 TCP 连接
一个连接上可以复用多个 stream
Header 可以压缩
```

但是 HTTP/2 仍然跑在 TCP 上。

TCP 的问题是：

```text
它只认一条有序字节流。
如果前面的包丢了，后面的包即使到了也要等。
```

于是 HTTP/3 把底层换成 QUIC：

```text
HTTP 语义还在
TCP 换成 QUIC
TLS 1.3 集成进 QUIC
stream 更独立
移动端连接迁移更友好
```

项目里怎么讲：

```text
我的第一版项目用 HTTP/1.1 + SSE，因为简单稳定。生产上如果移动端弱网场景明显，可以通过网关/CDN 支持 HTTP/3，并保留 HTTP/2 fallback。
```

---

## 2. 检索层：从纯向量 RAG 到 Hybrid RAG

用户问：

```text
差旅报销需要什么材料？
```

向量检索很擅长找：

```text
出差回来要交哪些凭证？
报销材料有哪些？
```

但如果用户问：

```text
BX2026001 审批到哪了？
第 3.2.1 条是什么意思？
住宿补贴 200 元的标准是什么？
```

纯向量检索可能不如关键词稳定。

所以要升级：

```text
dense embedding
  +
sparse/BM25 关键词
  +
合并去重
  +
Rerank
```

这就是 Hybrid Search。

项目里怎么讲：

```text
我不会只做普通向量检索。企业制度里金额、条款、编号很多，所以我会引入 hybrid search，再用 Rerank 选最终上下文。
```

---

## 3. 工具层：从普通 Tool Calling 到 MCP 思维

第一版 Tool Calling 是：

```text
模型输出：
{
  "tool_name": "get_reimbursement_status",
  "arguments": {
    "order_id": "BX2026001"
  }
}
```

后端做：

```text
参数校验
用户鉴权
查 MySQL
记录审计
把结果交给模型总结
```

如果后续工具越来越多：

```text
报销系统
请假系统
采购系统
代码仓库
知识库
工单系统
```

就可以考虑 MCP：

```text
工具发现标准化
参数 schema 标准化
资源访问标准化
上下文传递标准化
```

但要记住：

```text
MCP 不是安全本身。
安全仍然在后端鉴权、最小权限、参数校验、审计日志。
```

---

## 4. 模型层：从普通 Prompt 到结构化输出和缓存

普通 Prompt 可能让模型输出：

```text
好的，这是 JSON：
answer: ...
```

这对后端很难受。

生产系统要让模型输出稳定结构：

```text
answer
citations
need_tool_call
tool_name
tool_arguments
```

这就是 Structured Outputs 的价值。

同时，AI Copilot 的 Prompt 里有很多固定内容：

```text
角色说明
安全规则
工具 schema
输出格式
Few-shot 示例
```

这些应该放在 Prompt 前缀，动态内容放后面，提高 Prompt Caching 命中。

项目里怎么讲：

```text
我会把稳定系统提示词和工具 schema 放在前缀，把用户问题和检索 chunk 放在后面，既利于结构化输出，也利于 prompt cache。
```

## 5. 可观测性：从日志到 OpenTelemetry + eBPF

用户说：

```text
这次回答怎么这么慢？
```

只看日志，你可能只知道：

```text
总耗时 12 秒
```

但你不知道：

```text
embedding 慢？
向量库慢？
Rerank 慢？
模型首 token 慢？
生成 tokens/s 慢？
SSE 网络慢？
```

OpenTelemetry 把一次请求拆成 trace：

```text
chat.request
  auth.check
  rag.embedding
  rag.vector_search
  rag.rerank
  prompt.build
  model.call
  sse.stream
```

如果应用层看不出问题，再用 eBPF 看更底层：

```text
TCP 重传
系统调用耗时
容器网络
内核丢包
```

项目里怎么讲：

```text
AI 应用的慢可能出现在 RAG、模型、网络任意一段，所以我会用 OTel 做应用层全链路追踪，用 eBPF 辅助排查内核和网络层问题。
```

## 6. 最终面试总结版

```text
我理解新技术不是为了背新名词，而是看它解决了旧技术什么瓶颈。比如网络层，HTTP/3 保留 HTTP 语义，但用 QUIC 替代 TCP 作为传输层，改善 HTTP/2 仍受 TCP 队头阻塞影响的问题，并支持 TLS 1.3、0-RTT 和连接迁移。AI 应用层，普通 RAG 在企业制度场景下会漏金额、编号、条款，所以我会从纯向量检索升级到 dense + sparse 的 Hybrid Search，再接 Rerank；如果要处理跨文档关系，可以考虑 GraphRAG。模型输出方面，我会用 Structured Outputs 保证后端可解析，用 Prompt Caching 降低固定系统提示词和工具 schema 的成本。工程上，我会用 OpenTelemetry 做一次请求从网关、RAG、向量库到模型的 trace，必要时用 eBPF 排查内核和网络问题。
```
