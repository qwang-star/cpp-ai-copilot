# G00 前沿技术总览

这部分不是为了追概念热闹，而是为了回答一个面试里越来越常见的问题：

```text
你除了会基础八股，知不知道这些技术为什么还在演进？
```

你可以把技术演进理解成一条很朴素的主线：

```text
旧技术不是错了
只是旧技术在新场景下暴露了新的瓶颈
于是新技术在旧技术的基础上改一块关键短板
```

比如：

```text
HTTP/2 没有废掉 HTTP
它是在 HTTP/1.1 上解决多请求复用问题

HTTP/3 没有废掉 HTTP 语义
它是在 HTTP/2 的基础上把底层 TCP 换成 QUIC，减少队头阻塞和弱网建连成本
```

---

## 1. 前沿技术总地图

| 方向 | 旧技术 | 新技术 | 主要改进点 | 项目里怎么用 |
|---|---|---|---|---|
| 传输协议 | TCP + TLS + HTTP/2 | QUIC v1 + HTTP/3 | 减少队头阻塞，集成 TLS 1.3，支持连接迁移和 0-RTT | AI 聊天、移动端弱网、跨地域访问 |
| 实时通信 | WebSocket | WebTransport | 支持可靠流和不可靠 datagram，天然适合 HTTP/3/QUIC | 多模态实时协作、低延迟状态同步 |
| 高性能 IO | select/poll/epoll | io_uring | 提交队列和完成队列，减少系统调用和拷贝开销 | Linux 高并发文件/网络 IO 优化 |
| 可观测性 | 日志 + Prometheus + Zipkin/Jaeger 各管各的 | OpenTelemetry | 统一 traces、metrics、logs，厂商中立 | 模型调用链路追踪、RAG 延迟分析 |
| 内核观测 | 应用埋点 | eBPF | 不改应用代码也能观测内核、网络、安全事件 | 定位网络慢、系统调用慢、容器网络问题 |
| RAG 检索 | 纯向量检索 | Hybrid Search | dense 语义 + sparse 关键词，召回更稳 | 企业制度里的金额、条款、专有名词检索 |
| 深层 RAG | 普通 chunk RAG | GraphRAG | 用实体和关系图理解全局结构 | 长文档、多制度、多部门关系分析 |
| 模型输出 | 普通 JSON Prompt | Structured Outputs | 用 JSON Schema 约束输出结构 | Tool Calling 参数、抽取结果、评测结果 |
| 模型成本 | 每次完整算 Prompt | Prompt Caching | 复用稳定前缀，降低延迟和成本 | 长 system prompt、固定工具说明、固定 RAG 模板 |
| Agent 集成 | 每个工具单独接 API | MCP | 标准化模型/Agent 连接外部工具和数据 | IDE、企业工具、知识库、数据库统一接入 |
| 向量检索 | 默认向量库 + 单路召回 | HNSW 调参、量化、RRF、Late Interaction | 在召回、延迟、成本之间做工程平衡 | 大规模企业知识库、代码检索、制度条款检索 |
| LLM 推理 | 简单 HTTP 包一层模型 | vLLM、PagedAttention、Continuous Batching、TensorRT-LLM、Triton | 提升 GPU 利用率、吞吐和显存效率 | 私有化模型服务、Embedding/Rerank/LLM 统一治理 |
| 实时多模态 | 文本请求/整段音频上传 | WebRTC、Realtime API、Streaming ASR/TTS、WebGPU | 低延迟语音、多模态交互、端侧预处理 | 语音 Copilot、截图问答、端云协同 |
| 数据治理 | 登录鉴权 + 普通日志 | PII 脱敏、ABAC/ReBAC、RAG metadata filter、Confidential Computing | 防止越权检索和敏感信息泄露 | 企业私有知识库、工具调用、审计合规 |

## 2. 学这些新技术时的正确姿势

不要这样学：

```text
HTTP/3 很新，所以我背一下。
QUIC 很快，所以我背一下。
OpenTelemetry 很火，所以我背一下。
```

要这样学：

```text
旧技术遇到了什么问题？
新技术具体改了哪一层？
它真的解决了什么？
它又带来了什么新成本？
我项目里有没有一个场景能用它解释？
```

面试官真正想听的不是“我知道一个新名词”，而是：

```text
我知道这个技术为什么出现
我知道它和旧方案的差异
我知道什么时候该用，什么时候不该用
我知道落地时有哪些坑
```

## 3. 和 C++ 企业 AI Copilot 项目的关系

你的项目是：

```text
C++ 企业级 AI Copilot
```

它可以自然带出这些前沿技术：

```text
用户移动端弱网访问 -> HTTP/3 / QUIC
聊天流式输出 -> SSE，现在可讲 HTTP/2/HTTP/3 下的流式传输差异
多模态实时交互 -> WebTransport
高并发文件上传和解析 -> io_uring
线上性能排查 -> OpenTelemetry + eBPF
RAG 检索不准 -> Hybrid Search + Rerank + GraphRAG
模型输出不稳定 -> Structured Outputs
模型调用成本高 -> Prompt Caching
工具接入越来越多 -> MCP
向量库变大变慢 -> HNSW 调参 + 量化 + RRF + Late Interaction
私有化模型又贵又慢 -> vLLM / TensorRT-LLM / Triton + KV Cache 优化
语音和多模态实时交互 -> WebRTC / Realtime API / WebTransport
企业数据不能乱看乱传 -> PII 脱敏 + ABAC/ReBAC + metadata filter + 审计
```

也就是说，你的主项目不是只停留在“会做后端 + 调模型”，它可以升级成：

```text
会用现代协议、现代检索、现代可观测性，把 AI 应用做成生产系统
```

## 4. 面试回答模板

如果面试官问：

```text
你了解一些新的后端或 AI 应用技术吗？
```

你可以这样答：

```text
我会把新技术放在原有技术演进里理解。比如网络层，HTTP/2 已经解决了 HTTP/1.1 多连接和请求复用问题，但它仍然跑在 TCP 上，所以一个 TCP 包丢失时会影响同一连接上的多个 HTTP/2 stream。HTTP/3 把 HTTP 语义映射到 QUIC 上，QUIC 基于 UDP 实现多路复用、TLS 1.3 握手、连接迁移和 0-RTT，弱网和移动端体验会更好，但也带来 UDP 可达性、基础设施支持和排查成本。

在 AI 应用层，我不会只做普通向量 RAG。企业制度问答里有金额、条款号、专有名词，纯 dense embedding 容易漏，所以我会用 dense + sparse 的 Hybrid Search，再接 Rerank。对于长文档和跨文档关系，可以考虑 GraphRAG。工程上我会用 OpenTelemetry 记录一次问题从网关、RAG、向量库到模型的 trace，再用指标看 token、延迟、错误率和缓存命中率。
```
