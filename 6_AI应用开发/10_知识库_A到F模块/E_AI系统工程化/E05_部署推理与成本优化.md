# E05 部署推理与成本优化

## 1. 第三方 API

### 是什么

通过云厂商或模型平台 API 调用模型。

### 优点

- 接入快。
- 模型能力强。
- 无需 GPU 运维。

### 缺点

- 成本受供应商影响。
- 数据出域风险。
- 可控性弱。

## 2. 自部署模型

### 是什么

自己部署开源或私有模型，对外提供推理服务。

### 优点

- 数据可控。
- 适合私有化。
- 高调用量下可能更经济。

### 缺点

- 需要 GPU。
- 运维复杂。
- 推理优化难。

## 3. 推理服务

### 是什么

把模型封装成可调用服务。

常见：

- vLLM。
- TGI。
- Triton。
- Ollama。
- llama.cpp。

### 解决

- 模型加载。
- 请求调度。
- 批处理。
- KV Cache。
- 流式输出。
- GPU 显存管理。

## 4. 首 token 延迟

### 是什么

从请求发出到收到第一个 token 的时间。

### 为什么重要

决定用户体感是否“有反应”。

## 5. tokens/s

### 是什么

模型每秒生成 token 数。

### 影响

长答案生成速度。

## 6. Batch 推理

### 是什么

把多个请求合并成一批推理，提高吞吐。

### AI 场景

Embedding 非常适合 batch。

## 7. KV Cache

### 是什么

缓存 Transformer 已计算过的 Key / Value，避免生成每个新 token 时重复计算全部历史。

### 作用

加速自回归生成。

## 8. 量化

### 是什么

用更低精度表示模型权重。

### 优点

- 降低显存。
- 提高速度。

### 代价

- 可能损失效果。

## 9. Prompt Cache

### 是什么

缓存重复 prompt 前缀的计算结果。

### AI 场景

系统 prompt、工具 schema、固定上下文很长时有价值。

## 10. 成本优化

### 方法

- 模型分级。
- 缓存高频答案。
- Prompt 压缩。
- 控制 TopK。
- 控制 max_tokens。
- 批量 Embedding。
- 只在必要时 Rerank。
- 失败重试有上限。

## 11. Docker

### 是什么

容器化技术，把应用和运行环境打包。

### AI 场景

- 部署 API 服务。
- 部署 Worker。
- 部署模型服务。

### 注意

API Key 不写进镜像，模型文件可能很大。

## 12. Kubernetes

### 是什么

容器编排平台。

核心：

- Pod。
- Deployment。
- Service。
- Ingress。
- ConfigMap。
- Secret。
- HPA。

### AI 场景

- Chat Service 水平扩容。
- Worker 按队列扩容。
- GPU 模型服务单独调度。

## 13. 灰度和回滚

### AI 应用灰度对象

- 代码。
- Prompt。
- 模型。
- Embedding。
- Rerank。
- 检索策略。

### 注意

Embedding 版本回滚更复杂，因为涉及向量索引。

## 14. 面试回答模板

```text
第三方 API 适合快速验证，自部署模型适合私有化、强数据安全或高调用量场景。推理服务要关注首 token 延迟、tokens/s、吞吐、GPU 利用率和显存。成本优化可以从模型分级、缓存、Prompt 压缩、TopK 控制、批量 Embedding 和重试上限几个方面做。
```

## 15. 前沿升级：Prompt Caching、Structured Outputs、MCP

### Prompt Caching

旧做法：

```text
每次都完整处理 system prompt、工具说明、输出格式、Few-shot 示例
```

新做法：

```text
稳定前缀放前面
动态问题和 RAG chunk 放后面
尽量命中 Prompt Cache
```

优点：

- 降低首 token 延迟。
- 降低输入 token 成本。
- 适合长 system prompt 和大量工具 schema。

代价：

- 依赖供应商缓存策略。
- 动态内容放错位置会降低命中率。
- 仍要监控 cached tokens。

### Structured Outputs

旧做法：

```text
Prompt 里说“请输出 JSON”
```

问题：

```text
模型可能输出非法 JSON、缺字段、多解释文本
```

新做法：

```text
用 schema 约束 answer、citations、tool_name、arguments 等字段
```

但后端仍然要：

```text
JSON 解析
字段校验
参数校验
权限校验
审计日志
```

### MCP

当工具越来越多：

```text
报销系统
请假系统
采购系统
知识库
代码仓库
```

可以考虑 MCP 标准化工具和上下文接入。

但 MCP 不是安全本身：

```text
最小权限
后端鉴权
参数校验
审计日志
仍然必须保留
```

## 16. 前沿升级：vLLM、PagedAttention、Continuous Batching、TensorRT-LLM

如果项目从第三方模型 API 走向私有化部署，重点不只是“把模型跑起来”，而是：

```text
首 token 延迟低不低？
tokens/s 高不高？
GPU 利用率高不高？
KV Cache 会不会爆显存？
高并发请求能不能排队调度？
成本能不能降下来？
```

现代 LLM 推理服务常见升级：

```text
vLLM：OpenAI 兼容服务，适合快速私有化验证
PagedAttention：像操作系统分页一样管理 KV Cache
Continuous Batching：请求动态进出 batch，提高 GPU 利用率
Prefix Cache：复用稳定 system prompt 和工具 schema
Speculative Decoding：小模型打草稿，大模型验证，降低生成延迟
TensorRT-LLM：NVIDIA GPU 上更极致的推理优化
Triton：统一服务化 Embedding、Rerank、分类、OCR 等模型
SGLang：复杂 LLM 程序和结构化生成的推理编排方向
```

项目里推荐路线：

```text
第一版：第三方 API
第二版：C++ Model Gateway 抽象模型调用
第三版：接 vLLM OpenAI-compatible endpoint
第四版：监控 TTFT、TPOT、tokens/s、GPU 利用率、缓存命中
第五版：再评估量化、Speculative Decoding、TensorRT-LLM、Triton
```
