# D01 模型调用基础

## 1. 大模型调用是什么

### 是什么

大模型调用是后端通过 API 或本地推理服务，把用户输入、系统指令、上下文发送给模型，并接收模型输出。

### 基本链路

```text
用户请求
  -> 后端参数校验
  -> 组装 messages / prompt
  -> 调用模型
  -> 解析输出
  -> 保存日志
  -> 返回前端
```

### 为什么重要

这是所有 AI 应用的入口能力。RAG、Agent、摘要、抽取、客服都要调用模型。

## 2. System / User / Assistant

### 是什么

聊天模型通常用多角色消息表达上下文。

- System：系统级规则和角色。
- User：用户输入。
- Assistant：模型历史回复。

### 示例

```json
[
  {"role": "system", "content": "你是企业知识库问答助手。"},
  {"role": "user", "content": "报销流程是什么？"}
]
```

### AI 场景

System 用来放助手职责、回答边界、格式要求；User 放用户问题；Assistant 放历史回复。

## 3. 模型参数

### temperature

控制随机性。

- 低：稳定。
- 高：发散。

知识库问答、信息抽取、客服建议低 temperature。

### top_p

控制候选 token 的累计概率范围。

### max_tokens

限制最大输出 token 数。

### stop

遇到指定停止符停止输出。

### response_format

约束输出格式，例如 JSON。

## 4. 上下文窗口

### 是什么

模型一次能处理的最大 token 数。

### 包括

- System prompt。
- 历史消息。
- RAG 检索资料。
- 工具定义。
- 用户问题。
- 模型输出。

### 常见问题

上下文太长会：

- 成本增加。
- 延迟增加。
- 超过限制。
- 重点被稀释。

## 5. 流式调用

### 是什么

模型边生成边返回 token delta。

### 优点

- 首字更快。
- 用户体验更好。
- 长回答不用一直白屏。

### 后端处理

```text
调用模型 stream
  -> 接收 delta
  -> 转发给前端
  -> 追加到完整 answer
  -> 结束后保存 message
```

### 常见坑

- 客户端断开未取消上游。
- 后端没有保存完整答案。
- 流式中途错误没有事件通知。

## 6. 超时

### 为什么重要

模型调用可能慢，不能无限等。

### 超时类型

- 连接超时。
- 首 token 超时。
- 总生成超时。
- 工具调用超时。

### AI 场景

知识库问答可设置：

```text
检索 1s
Rerank 1s
模型首 token 5s
总生成 30s
```

## 7. 重试

### 是什么

模型调用失败后再次尝试。

### 可重试

- 网络抖动。
- 429 限流后等待。
- 5xx。
- 临时超时。

### 不宜重试

- 参数错误。
- 权限错误。
- 已经开始流式输出。
- 非幂等工具调用。

## 8. 模型调用日志

### 必须记录

```text
trace_id
user_id
tenant_id
model_name
prompt_version
input_tokens
output_tokens
latency_ms
first_token_latency_ms
cost
status
error_code
```

### 用途

- 排查慢请求。
- 统计成本。
- 评估模型效果。
- 审计。
- 回放 bad case。

## 9. 模型客户端封装

### 为什么需要

不要在业务代码里到处直接调模型 API。

推荐：

```text
ModelClient
  -> chat()
  -> streamChat()
  -> embed()
  -> rerank()
```

统一处理：

- API Key。
- 超时。
- 重试。
- 日志。
- 成本。
- 错误码。
- 降级。

## 10. 面试回答模板

```text
我会把模型调用封装到统一 ModelClient 或模型网关中，而不是散落在业务代码里。调用时要处理 messages 组装、模型参数、流式输出、超时重试、错误码、日志和 token 成本。这样后续切模型、做降级和统计成本都更方便。
```
