# A01 网络

## 1. TCP/IP 分层

### 是什么

TCP/IP 分层是计算机网络的分层模型，用来把复杂的网络通信拆成不同职责。

常见四层模型：

```text
应用层：HTTP、DNS、WebSocket、SSE、gRPC
传输层：TCP、UDP
网络层：IP、ICMP
链路层：以太网、Wi-Fi
```

### 为什么重要

后端服务处理的每个请求，本质都是网络通信。AI 应用里模型调用、向量库查询、文件上传、流式输出都离不开网络。

### 核心机制

- 应用层定义数据格式和语义，比如 HTTP 请求。
- 传输层解决端到端传输，比如 TCP 保证可靠。
- 网络层负责主机到主机的路由。
- 链路层负责局域网内的数据帧传输。

### 后端 AI 应用场景

```text
用户聊天请求 -> HTTP
模型流式返回 -> SSE / WebSocket
后端调用模型服务 -> HTTP / gRPC
向量库查询 -> HTTP / gRPC
文件上传 -> HTTP multipart
```

### 面试回答

可以说：

```text
网络分层的意义是职责拆分。应用层关心协议语义，传输层关心可靠传输，网络层关心路由，链路层关心物理网络。后端排查问题时也会按层定位，比如 HTTP 状态码、TCP 连接、DNS 解析、负载均衡等。
```

## 2. TCP 三次握手

### 是什么

TCP 三次握手是建立 TCP 连接的过程。

```text
客户端 -> SYN -> 服务端
服务端 -> SYN + ACK -> 客户端
客户端 -> ACK -> 服务端
```

### 为什么重要

TCP 是可靠连接协议，HTTP/1.1、HTTP/2、HTTPS、数据库连接、Redis 连接都依赖 TCP。

### 核心机制

三次握手要确认两件事：

- 双方发送能力正常。
- 双方接收能力正常。

两次不够，因为服务端无法确认客户端是否能收到自己的响应。

### 后端 AI 应用场景

- 模型 API 请求需要建立连接。
- 连接池复用 TCP 连接，减少握手开销。
- 模型调用慢时，如果连接池配置不合理，会导致请求排队。

### 常见坑

- 每次请求都新建连接会增加延迟。
- 连接池太小会排队。
- 连接池太大可能压垮下游服务。

### 面试回答

```text
三次握手是为了确认双方的收发能力。第一次客户端发 SYN，服务端知道客户端能发；第二次服务端回 SYN+ACK，客户端知道服务端能收也能发；第三次客户端回 ACK，服务端知道客户端能收。这样双方才能建立可靠连接。
```

## 3. TCP 四次挥手

### 是什么

TCP 四次挥手是关闭连接的过程。

```text
主动关闭方 -> FIN
被动关闭方 -> ACK
被动关闭方 -> FIN
主动关闭方 -> ACK
```

### 为什么是四次

TCP 是全双工。一个方向关闭，不代表另一个方向也立刻关闭。

### TIME_WAIT

主动关闭方会进入 TIME_WAIT。

作用：

- 确保最后一个 ACK 能到达对方。
- 等待旧连接中的残留报文过期，避免影响新连接。

### AI 场景

流式输出时，如果客户端断开连接，后端要及时感知并关闭上游模型调用，避免继续烧 token。

## 4. TCP 可靠传输

### 是什么

TCP 通过一系列机制保证数据可靠、有序、不重复地到达。

核心机制：

- 序列号。
- 确认应答 ACK。
- 超时重传。
- 流量控制。
- 拥塞控制。
- 校验和。

### 后端 AI 应用场景

大模型流式输出是连续数据传输。如果网络抖动，TCP 会尽力重传，但应用层仍然要处理连接断开、超时、客户端取消等情况。

### 面试回答

```text
TCP 的可靠性主要靠序列号、ACK、重传、滑动窗口和拥塞控制。序列号保证有序，ACK 确认收到，超时重传处理丢包，滑动窗口提高吞吐，拥塞控制避免把网络打满。
```

## 5. 滑动窗口

### 是什么

滑动窗口是 TCP 用来提高传输效率和做流量控制的机制。

如果每发一个包都等 ACK，吞吐会很低。滑动窗口允许发送方一次发送多个未确认的数据包。

### 核心机制

- 接收方通过窗口大小告诉发送方自己还能接收多少数据。
- 发送方在窗口范围内连续发送。
- 收到 ACK 后窗口向前滑动。

### AI 场景

文件上传、下载、模型输出、日志传输都受 TCP 窗口影响。后端更多从应用层处理，比如限制上传大小、分片上传、流式传输。

## 6. 拥塞控制

### 是什么

拥塞控制是 TCP 防止网络过载的机制。

核心阶段：

- 慢启动。
- 拥塞避免。
- 快重传。
- 快恢复。

### 为什么重要

它保护的是整个网络，不只是单个接收方。

### 面试回答

```text
流量控制关注接收方处理不过来，拥塞控制关注网络链路被打满。TCP 通过慢启动、拥塞避免、快重传、快恢复动态调整发送速率。
```

## 7. HTTP 请求与响应

### 是什么

HTTP 是应用层协议，定义客户端和服务端如何交换数据。

请求包含：

- Method。
- URL。
- Header。
- Body。

响应包含：

- Status Code。
- Header。
- Body。

### 常见方法

- GET：查询。
- POST：创建或提交。
- PUT：整体更新。
- PATCH：局部更新。
- DELETE：删除。

### AI 场景 API

```text
POST /api/v1/chat/stream
POST /api/v1/documents/upload
GET  /api/v1/documents/{id}/status
POST /api/v1/knowledge-bases/{id}/search
POST /api/v1/agents/{id}/runs
```

## 8. HTTP 状态码

### 常见状态码

- 200：成功。
- 201：创建成功。
- 400：参数错误。
- 401：未认证。
- 403：无权限。
- 404：资源不存在。
- 409：冲突，比如重复提交。
- 429：限流。
- 500：服务端错误。
- 502：网关或下游错误。
- 503：服务不可用。
- 504：网关超时。

### AI 场景

- 模型超时可以映射为 504。
- 用户额度耗尽可以返回 429 或业务错误码。
- 文档未解析完成可以返回 202 / task status。

## 9. HTTP 长连接

### 是什么

HTTP 长连接是在一个 TCP 连接上复用多个 HTTP 请求，减少频繁建立连接的开销。

### 为什么重要

模型调用、向量库查询、微服务调用都可能很频繁。连接复用能降低延迟和资源消耗。

### 常见坑

- 连接池泄露。
- 空闲连接被服务端关闭。
- 连接池太小排队。
- 下游慢导致连接被占满。

## 10. HTTPS

### 是什么

HTTPS = HTTP + TLS。

它提供：

- 加密。
- 身份认证。
- 完整性校验。

### 核心机制

- 非对称加密用于协商密钥和身份认证。
- 对称加密用于传输数据。
- 证书用于证明服务端身份。

### AI 场景

模型 API Key、用户文档、企业知识库都属于敏感数据，必须通过 HTTPS 传输。

## 11. HTTP/1.1、HTTP/2、HTTP/3

### HTTP/1.1

特点：

- 长连接。
- 管线化支持有限。
- 队头阻塞问题明显。

### HTTP/2

特点：

- 多路复用。
- 二进制帧。
- Header 压缩。
- 单连接并发多个请求。

### HTTP/3

特点：

- 基于 QUIC。
- 使用 UDP。
- 改善 TCP 层队头阻塞。
- 连接迁移更好。

### 面试回答

```text
HTTP/2 通过多路复用减少了 HTTP/1.1 多连接和队头阻塞问题。HTTP/3 基于 QUIC，进一步解决 TCP 层队头阻塞，并改善弱网和连接迁移。
```

## 12. RESTful

### 是什么

RESTful 是一种基于资源的 API 设计风格。

例子：

```text
GET    /documents
POST   /documents
GET    /documents/{id}
DELETE /documents/{id}
```

### AI 场景

知识库、文档、会话、消息、任务、评测都适合建模为资源。

### 常见坑

- 把所有接口都写成 `/doSomething`。
- 状态码混乱。
- 删除、重试等操作没有幂等设计。

## 13. RPC / gRPC

### 是什么

RPC 是远程过程调用，让调用远程服务像调用本地函数一样。

gRPC 基于 HTTP/2 和 Protocol Buffers。

### 适合场景

- 微服务内部调用。
- 高性能服务间通信。
- 强类型接口。
- 流式通信。

### AI 场景

- Chat Service 调 RAG Service。
- RAG Service 调 Embedding Service。
- Model Gateway 调本地推理服务。

## 14. SSE

### 是什么

SSE 是 Server-Sent Events，服务端向客户端单向推送事件。

### 特点

- 基于 HTTP。
- 单向推送。
- 浏览器原生支持 EventSource。
- 适合文本流式输出。

### AI 场景

大模型生成答案时，每生成一小段就推给前端。

```text
data: 你好

data: ，这是

data: 答案

data: [DONE]
```

### 常见坑

- 代理缓冲导致前端不能实时收到。
- 连接断开没有取消上游模型。
- 没有心跳，长连接被断。
- 只在前端拼接答案，后端没有保存完整输出。

## 15. WebSocket

### 是什么

WebSocket 是全双工长连接协议，客户端和服务端可以互相主动发送消息。

### 适合场景

- 实时聊天。
- 语音对话。
- 协作编辑。
- 游戏。
- 多人互动。

### SSE vs WebSocket

```text
SSE：服务端单向推送，简单，适合普通模型流式输出。
WebSocket：双向通信，复杂，适合实时互动。
```

## 16. DNS

### 是什么

DNS 把域名解析成 IP 地址。

### 核心流程

```text
浏览器缓存
  -> 系统缓存
  -> 本地 DNS
  -> 根域名服务器
  -> 顶级域名服务器
  -> 权威域名服务器
```

### AI 场景

第三方模型 API 域名解析失败，会导致模型调用失败。生产中要监控 DNS、连接错误、超时错误。

## 17. CDN

### 是什么

CDN 把静态资源缓存到离用户更近的节点。

### AI 场景

- 文档原文件下载。
- 图片、音频、视频访问。
- 前端静态资源。

不适合：

- 私密文档直接公开缓存。
- 强权限控制资源不做鉴权就上 CDN。

## 18. 负载均衡

### 是什么

负载均衡把请求分发到多个后端实例。

策略：

- 轮询。
- 加权轮询。
- 最少连接。
- 一致性哈希。
- 基于延迟。

### AI 场景

- Chat Service 水平扩容。
- Embedding Worker 扩容。
- 模型推理服务多实例路由。

### 常见坑

- 长连接需要考虑连接保持。
- WebSocket 可能需要会话粘滞。
- 下游模型服务负载不均。

## 19. 连接池

### 是什么

连接池复用已建立的连接，避免频繁创建和销毁连接。

常见对象：

- 数据库连接池。
- Redis 连接池。
- HTTP 连接池。
- gRPC Channel。

### AI 场景

模型网关需要 HTTP 连接池。向量库查询也需要连接复用。

### 常见配置

- 最大连接数。
- 最大空闲连接数。
- 连接超时。
- 读取超时。
- 空闲连接回收。

### 常见坑

- 连接池太小导致排队。
- 没有读取超时导致线程卡死。
- 下游慢时连接耗尽。

## 20. 超时与重试

### 是什么

超时是给一次远程调用设置最大等待时间。重试是在失败后再次尝试。

### 为什么重要

AI 应用链路长：

```text
鉴权 -> Redis -> MySQL -> 向量库 -> Rerank -> LLM -> 流式返回
```

任何一步没有超时，都可能拖垮整个请求。

### 重试原则

- 只重试可恢复错误。
- 设置最大次数。
- 指数退避。
- 加随机抖动。
- 保证幂等。
- 避免重试风暴。

### 面试回答

```text
我会给模型调用、向量检索、工具调用分别设置超时，并在模型网关统一处理可重试错误。重试要有上限和指数退避，同时结合熔断，避免下游已经异常时被大量重试打垮。
```

## 21. 前沿升级：HTTP/3、QUIC、WebTransport

### 为什么要补这些

传统网络八股讲到 HTTP/2、SSE、WebSocket 还不够。现在移动端、弱网、跨地域访问越来越多，HTTP/3 和 QUIC 已经是必须知道的演进方向。

### 从旧技术怎么演进

```text
HTTP/1.1
  -> 多连接、队头阻塞明显

HTTP/2
  -> 一个 TCP 连接上多路复用多个 stream
  -> 但仍然受 TCP 层队头阻塞影响

HTTP/3
  -> HTTP 语义保留
  -> 底层从 TCP 换成 QUIC
  -> QUIC 基于 UDP，实现多路复用、TLS 1.3、连接迁移、0-RTT
```

### 优点

- 弱网和移动网络更友好。
- 减少 TCP 队头阻塞影响。
- TLS 1.3 集成进 QUIC 握手。
- Connection ID 支持网络切换后的连接迁移。

### 代价

- UDP 可能被企业网络、防火墙、代理限制。
- 抓包和排查更难。
- CDN、负载均衡、反向代理都要支持。
- 0-RTT 有重放风险，只适合幂等读请求。

### 在 AI Copilot 项目里怎么用

第一版：

```text
HTTP/1.1 + SSE
```

生产演进：

```text
网关层支持 HTTP/2
移动端弱网场景启用 HTTP/3 / QUIC
普通文字流式仍可 SSE
低延迟多模态实时交互再考虑 WebTransport
```
