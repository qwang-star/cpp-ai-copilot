# A04 Redis 缓存

## 1. Redis 是什么

### 是什么

Redis 是基于内存的高性能 Key-Value 数据库。

它常用于：

- 缓存。
- 分布式锁。
- 限流。
- 排行榜。
- 会话。
- 消息流。
- 计数器。

### 为什么快

主要原因：

- 数据在内存中。
- 单线程事件循环避免大量锁竞争。
- IO 多路复用。
- 数据结构设计高效。
- C 语言实现。

注意：

```text
Redis 单线程主要指命令执行单线程。新版本中网络 IO、持久化等部分可能有多线程优化。
```

### AI 场景

- 缓存高频问答。
- 缓存 RAG 检索结果。
- 用户级限流。
- 模型调用额度计数。
- 文档处理任务幂等。
- SSE 流式状态暂存。
- 分布式锁防止重复索引。

## 2. String

### 是什么

String 是 Redis 最基础的数据结构，可以存字符串、数字、JSON、序列化对象。

### 常用命令

```text
set key value ex 60
get key
incr key
decr key
setnx key value
```

### AI 场景

- 缓存问答结果。
- 用户 token 用量计数。
- 接口调用次数计数。
- 分布式锁。

例子：

```text
qa_cache:{tenant_id}:{kb_id}:{query_hash}:{prompt_version}
```

## 3. Hash

### 是什么

Hash 类似一个小型 Map，适合存对象字段。

### AI 场景

- 会话摘要。
- 用户配额信息。
- 文档处理状态。

例子：

```text
HSET task:{task_id} status embedding progress 80 updated_at 1780000000
```

## 4. List

### 是什么

List 是双端链表，可以从头尾插入或弹出。

### AI 场景

- 简单任务队列。
- 最近消息列表。

注意：

正式生产任务队列更推荐 MQ 或 Redis Stream。

## 5. Set

### 是什么

Set 是无序不重复集合。

### AI 场景

- 用户已访问文档集合。
- 去重 content_hash。
- 记录某知识库在线用户。

## 6. ZSet

### 是什么

ZSet 是带分数的有序集合。

### AI 场景

- 热门问题排行榜。
- 用户模型调用成本排行。
- 任务优先级队列。
- 最近活跃会话排序。

例子：

```text
ZINCRBY hot_queries:{kb_id} 1 {query_hash}
```

## 7. Bitmap

### 是什么

Bitmap 用 bit 位表示状态，适合大量布尔值统计。

### AI 场景

- 用户每日是否使用 AI 功能。
- 评测样本是否执行过。

## 8. HyperLogLog

### 是什么

HyperLogLog 用于大规模基数估算，占用内存很小，但结果近似。

### AI 场景

- 统计每日独立使用 AI 用户数。
- 统计知识库独立访问用户数。

## 9. Stream

### 是什么

Redis Stream 是 Redis 的消息流结构，支持消息 ID、消费者组、ACK。

### AI 场景

- 轻量级文档处理任务队列。
- 模型调用日志流。
- 用户反馈事件流。

注意：

大规模复杂消息系统仍然更常用 Kafka / RocketMQ / RabbitMQ。

## 10. 过期策略

### 是什么

Redis key 可以设置 TTL，到期后删除。

删除策略：

- 惰性删除：访问 key 时发现过期再删除。
- 定期删除：后台定期抽样删除过期 key。

### AI 场景

- 问答缓存设置短 TTL。
- 会话短期状态设置 TTL。
- 分布式锁必须设置过期时间。

## 11. 内存淘汰策略

### 是什么

当 Redis 内存达到上限，需要按策略淘汰 key。

常见策略：

- noeviction：不淘汰，写入报错。
- allkeys-lru：所有 key 中淘汰最近最少使用。
- volatile-lru：只从设置过期时间的 key 中淘汰。
- allkeys-lfu：淘汰使用频率低的。
- volatile-ttl：淘汰即将过期的。

### AI 场景

问答缓存通常可淘汰；分布式锁、任务状态等关键 key 不应被随意淘汰。

## 12. RDB

### 是什么

RDB 是 Redis 的快照持久化，把某一时刻的数据保存到磁盘。

### 优点

- 文件紧凑。
- 恢复快。

### 缺点

- 可能丢失最近一次快照后的数据。

## 13. AOF

### 是什么

AOF 记录每次写命令，重启时重放恢复。

### 优点

- 数据更安全。

### 缺点

- 文件可能更大。
- 恢复可能更慢。

## 14. 缓存穿透

### 是什么

查询的数据在缓存和数据库都不存在，请求每次都打到数据库。

### 解决

- 缓存空值。
- 布隆过滤器。
- 参数校验。

### AI 场景

用户反复查询不存在的 document_id，如果不缓存空结果，会打爆数据库。

## 15. 缓存击穿

### 是什么

热点 key 过期瞬间，大量请求同时打到数据库或下游服务。

### 解决

- 互斥锁重建缓存。
- 逻辑过期。
- 热点 key 不过期。
- 后台刷新。

### AI 场景

热门问题答案缓存过期，大量用户同时提问，可能同时触发向量检索和模型调用，成本非常高。

## 16. 缓存雪崩

### 是什么

大量 key 同时过期，或者 Redis 故障，导致请求集中打到数据库或下游服务。

### 解决

- TTL 加随机值。
- 多级缓存。
- 限流降级。
- Redis 高可用。

## 17. 分布式锁

### 是什么

分布式锁用于多个服务实例之间互斥访问共享资源。

Redis 常见实现：

```text
SET lock:key value NX EX 30
```

释放锁要校验 value，防止误删别人的锁。

### AI 场景

- 防止同一文档被多个 Worker 重复向量化。
- 防止同一用户额度并发扣减异常。
- 防止同一知识库重复重建索引。

### 常见坑

- 没有过期时间导致死锁。
- 业务执行超过锁过期时间。
- 删除锁时没有校验 owner。
- Redis 主从切换导致锁安全性问题。

## 18. Lua 脚本

### 是什么

Redis Lua 脚本可以把多个命令打包成一个原子操作。

### AI 场景

- 限流计数 + 设置过期时间。
- 校验锁 value + 删除锁。
- 用户额度扣减。

## 19. 限流

### 是什么

限流用于控制请求速率，保护系统和成本。

限流维度：

- 用户。
- IP。
- 租户。
- 接口。
- 模型。
- token。
- 金额。

常见算法：

- 固定窗口。
- 滑动窗口。
- 令牌桶。
- 漏桶。

### AI 场景

AI 应用必须关注 token 成本，不能只按请求次数限流。

例子：

```text
user:{id}:daily_tokens <= 100000
tenant:{id}:monthly_cost <= 5000
```

## 20. Redis 集群

### 是什么

Redis 集群通过分片存储数据，提高容量和吞吐。

核心：

- hash slot。
- 主从复制。
- 故障转移。

### 常见问题

- 跨 slot 多 key 操作受限。
- 热 key 仍然可能压垮单节点。
- 大 key 会影响性能。

### AI 场景

热门问答缓存可能形成热 key，需要本地缓存、复制缓存或拆分 key。
