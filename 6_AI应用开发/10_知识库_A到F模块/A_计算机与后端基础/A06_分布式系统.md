# A06 分布式系统

## 1. 分布式系统是什么

### 是什么

分布式系统是多个独立节点通过网络协作，对外提供一个整体服务。

### 为什么重要

后端 AI 应用通常包含多个服务：

- API 服务。
- 文档服务。
- RAG 服务。
- 模型网关。
- Worker。
- 向量数据库。
- MySQL。
- Redis。
- MQ。

任何一个节点都可能慢、挂、超时。

## 2. CAP

### 是什么

CAP 指分布式系统中的三个性质：

- Consistency：一致性。
- Availability：可用性。
- Partition Tolerance：分区容错性。

网络分区存在时，系统通常只能在 C 和 A 之间取舍。

### 面试回答

```text
CAP 不是说三个永远只能选两个，而是在网络分区发生时，系统需要在一致性和可用性之间取舍。实际业务会按场景选择，比如核心交易偏一致性，推荐和缓存偏可用性。
```

## 3. BASE

### 是什么

BASE 是一种最终一致性思想：

- Basically Available：基本可用。
- Soft State：软状态。
- Eventually Consistent：最终一致。

### AI 场景

文档上传后不是立即可问答，而是状态从 uploaded 到 ready，最终完成索引。

这就是最终一致：

```text
MySQL 文档记录已存在
向量索引稍后完成
用户看到状态 processing
最终 ready 后可检索
```

## 4. 一致性

### 常见一致性

- 强一致性。
- 弱一致性。
- 最终一致性。

### AI 场景

- 用户权限：必须强一致或尽量强。
- 文档索引：可以最终一致。
- 模型调用日志：可以异步最终一致。
- 缓存：通常允许短暂不一致。

## 5. 服务注册发现

### 是什么

服务启动后注册自己的地址，调用方通过注册中心发现可用实例。

### AI 场景

RAG Service 需要发现 Embedding Service、Rerank Service、Model Gateway。

## 6. 负载均衡

### 是什么

把请求分发到多个实例。

### AI 场景

- 多个 Chat Service 实例。
- 多个 Worker。
- 多个模型推理实例。

### 注意

模型服务负载和普通服务不同，GPU 显存和队列长度也要纳入路由。

## 7. 限流

### 是什么

限制请求速率，保护系统。

### 算法

- 固定窗口。
- 滑动窗口。
- 令牌桶。
- 漏桶。

### AI 场景

限流维度：

- user_id。
- tenant_id。
- model_name。
- token_count。
- cost。

面试回答：

```text
AI 应用的限流不仅看 QPS，还要看 token 和费用。一个请求可能消耗几千 token，所以我会做用户级和租户级 token 配额。
```

## 8. 熔断

### 是什么

当下游服务错误率或超时率过高时，暂时停止调用，快速失败。

### AI 场景

第三方模型服务异常时，如果还持续调用，会导致线程堆积和成本浪费。

### 熔断状态

- Closed：正常。
- Open：熔断。
- Half-Open：半开试探。

## 9. 降级

### 是什么

当系统压力大或依赖异常时，牺牲部分功能保证核心可用。

### AI 场景

降级方式：

- 强模型降级到小模型。
- 生成答案降级为返回检索片段。
- 关闭 Rerank。
- 返回缓存答案。
- 转人工。

## 10. 分布式事务

### 是什么

跨多个服务或资源的一致性事务。

常见方案：

- 2PC。
- TCC。
- Saga。
- 本地消息表。
- 事务消息。

### AI 场景

文档上传涉及：

- 对象存储。
- MySQL。
- MQ。
- 向量库。

不一定要强事务，常用最终一致 + 补偿。

## 11. 分布式 ID

### 是什么

在分布式系统中生成全局唯一 ID。

方案：

- UUID。
- 数据库自增。
- 雪花算法。
- 号段模式。

### AI 场景

- trace_id。
- document_id。
- chunk_id。
- conversation_id。
- task_id。

### 建议

chunk_id 要稳定，便于向量库和 MySQL 对齐。

## 12. 配置中心

### 是什么

集中管理配置，并支持动态更新。

### AI 场景

配置项：

- 模型名称。
- Prompt 版本。
- TopK。
- Rerank 开关。
- 超时时间。
- 限流阈值。
- 灰度比例。

## 13. 链路追踪

### 是什么

链路追踪用 trace_id 串起一次请求经过的所有服务。

### AI 场景

一次问答：

```text
API -> Auth -> Redis -> Vector DB -> Rerank -> Model Gateway -> LLM
```

每一步都要记录耗时，才能定位慢在哪里。

## 14. 灰度发布

### 是什么

先让少量流量使用新版本，观察稳定后再扩大。

### AI 场景

灰度对象：

- 代码。
- Prompt。
- 模型。
- Embedding。
- Rerank。
- 检索策略。

### 注意

Embedding 模型灰度复杂，因为新旧向量空间不同，可能需要双写双读。

## 15. 面试回答模板

```text
AI 应用的分布式难点在于链路长、依赖多、下游模型慢且不稳定。我会通过限流保护成本，通过超时和熔断避免依赖拖垮主服务，通过降级保证核心体验，通过 trace_id 观测每个阶段耗时，通过 MQ 和状态机保证文档索引最终一致。
```
