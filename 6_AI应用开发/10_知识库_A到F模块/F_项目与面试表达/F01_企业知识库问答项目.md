# F01 企业知识库问答项目

## 1. 项目是什么

企业知识库问答系统是让用户基于企业内部文档进行自然语言问答的 AI 应用。

它不是简单聊天机器人，而是完整 RAG 系统。

## 2. 核心目标

解决：

- 企业文档多，查找困难。
- 新员工不知道制度。
- 客服或运营重复回答问题。
- 大模型不知道企业私有知识。
- 答案需要可追溯引用。

## 3. 功能清单

用户功能：

- 登录。
- 创建知识库。
- 上传文档。
- 查看文档解析状态。
- 提问。
- 流式查看答案。
- 查看引用来源。
- 历史会话。
- 点赞点踩。

管理功能：

- 成员管理。
- 权限配置。
- 文档重建索引。
- 调用日志。
- 评测结果。

```text

```

## 5. 文档入库链路

```text
用户上传文档
  -> 后端校验文件类型和大小
  -> 原文件存对象存储
  -> MySQL 写 document(status=uploaded)
  -> MQ 发送 parse_document
  -> Parse Worker 解析、清洗、切分
  -> MySQL 写 document_chunk
  -> MQ 发送 embed_chunks
  -> Embedding Worker 批量向量化
  -> 写入向量数据库
  -> document 状态 ready
```

## 6. 在线问答链路

```text
用户提问
  -> 鉴权和限流
  -> 保存 user message
  -> Query Rewrite
  -> Query Embedding
  -> metadata 权限过滤
  -> 向量检索 + 关键词检索
  -> 合并去重
  -> Rerank
  -> 组装 RAG Prompt
  -> 模型网关调用 LLM
  -> SSE 流式返回
  -> 保存 assistant message
  -> 保存 retrieval_log 和 model_call_log
```

## 9. 技术难点 1：文档处理耗时

### 问题

PDF 解析、OCR、Embedding 都很慢，同步处理会超时。

### 方案

- 上传接口只保存文件和任务。
- MQ 异步处理。
- Worker 可水平扩容。
- 状态机记录进度。
- 失败重试和死信队列。

## 10. 技术难点 2：RAG 准确率

### 问题

召回不到或召回噪声多。

### 方案

- 优化文档解析。
- 按标题和语义切 chunk。
- 向量 + 关键词混合检索。
- Rerank。
- Query Rewrite。
- 评测集验证。

## 11. 技术难点 3：权限过滤

### 问题

用户不能检索到无权限文档。

### 方案

- chunk metadata 写入 tenant_id、kb_id、acl。
- 检索时构造 metadata filter。
- 返回引用前二次校验。
- 审计日志。

## 12. 技术难点 4：模型慢且贵

### 方案

- SSE 流式输出。
- 模型分级。
- 缓存高频问题。
- Prompt 压缩。
- 控制 TopK。
- token 成本统计。
- 用户额度限流。

## 13. 技术难点 5：稳定性

### 方案

- 模型网关统一调用。
- 超时重试。
- 熔断降级。
- 备用模型。
- RAG 召回为空时拒答。
- 全链路 trace。

## 14. 三分钟项目介绍

```text
我做的是一个企业知识库问答系统，目标是让用户能基于企业内部文档进行自然语言问答。

系统分为文档入库和在线问答两条链路。文档上传后，原文件存对象存储，MySQL 保存文档状态，然后通过 MQ 异步解析、清洗、切分并生成 Embedding，最后写入向量数据库。在线问答时，系统先做鉴权和限流，再对用户问题做 Query Rewrite 和向量化，结合 metadata 权限过滤进行向量检索和关键词检索，召回结果经过 Rerank 后拼入 Prompt，通过模型网关调用大模型，并用 SSE 流式返回答案和引用来源。

项目重点解决了文档处理异步化、RAG 检索准确率、权限过滤、模型调用稳定性和成本监控这些工程问题。
```

## 15. 前沿技术升级版项目表达

如果面试官希望听你对新技术的理解，可以在三分钟版后面补一段：

```text
这个项目第一版用 HTTP/1.1 + SSE 实现流式回答，后续如果面向移动端弱网场景，可以在网关层支持 HTTP/3/QUIC，并保留 HTTP/2 fallback。RAG 检索方面，我不会只依赖纯向量检索，而是会用 dense + sparse 的 Hybrid Search 解决金额、条款、编号这类精确词召回问题，再用 Rerank 提升上下文质量。模型输出方面，Tool Calling 和评测结果会尽量使用 Structured Outputs，避免 JSON 不稳定。工程上，我会用 OpenTelemetry 记录从网关、RAG、向量库到模型调用的 trace，并用 Feature Flag 灰度 Prompt、模型和检索策略。安全上会参考 OWASP LLM Top 10，对 Prompt Injection、越权检索、过度代理和供应链风险做防护。
```

如果面试官继续追问“还有没有更前沿的优化”，可以再接一段：

```text
如果知识库规模继续变大，我会进一步做 HNSW 参数压测、RRF 融合、向量量化和 Late Interaction 评估，用 Recall@K、MRR、NDCG 和 P95 延迟证明收益。推理服务方面，如果从第三方 API 转向私有化部署，我会让 C++ 模型网关对接 vLLM 这类 OpenAI 兼容服务，关注 PagedAttention、Continuous Batching、KV Cache、TTFT、tokens/s 和 GPU 利用率；高性能场景再评估 TensorRT-LLM 或 Triton。多模态方向，文本流用 SSE，任务状态用 WebSocket，实时语音再考虑 WebRTC 或 Realtime API。数据治理上，会做 PII 脱敏、ABAC/ReBAC、RAG metadata filter、审计日志和文档删除链路，避免模型看到不该看的数据。
```

这段话的价值是：

```text
你不是只会做基础 RAG Demo
你知道生产 AI 系统怎么继续演进
你知道新技术的优点和代价
```
