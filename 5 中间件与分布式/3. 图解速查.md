## 一、CAP 不可能三角

```
                    ┌──────────┐
                    │    C     │
                    │ 一致性    │
                    └──┬───┬───┘
                       │   │
            CP系统 ←───┘   └───→ AP系统
          (ZK/etcd)           (Eureka)
                       │   │
                 ┌─────┘   └─────┐
                 ▼               ▼
            ┌──────────┐   ┌──────────┐
            │    A     │   │    P     │
            │ 可用性    │   │ 分区容错  │
            └──────────┘   └──────────┘
                  ↑               ↑
                  └─── CA系统 ────┘
                   (单机, 不存在P)

结论: P 必须选 → 实际是 CP vs AP 的选择
      CP: 网络分区时少数节点不可用, 保证数据一致
      AP: 网络分区时所有节点可用, 可能数据不一致
```

## 四、一致性哈希环

```
普通 hash 取模 (node = key % 3):
  加一个节点 → 大部分 key 重新分配!
  
  3 节点: A=0→33%, B=33%→66%, C=66%→100%
  4 节点: A=0→25%, B=25%→50%, C=50%→75%, D=75%→100%


一致性哈希环 (0 ~ 2^32-1):
  
                2^32-1 ─┐      ┌─ 0
                         │      │
                    ┌────┴──────┴────┐
                    │                │
                    │    ┌──┐        │
              B-2   │    │A │        │  A-1
                    │    └──┘        │
                    │         ┌──┐   │
              C-1   │         │D │   │  B-1
                    │  ┌──┐   └──┘   │
                    │  │C │          │
              A-2   │  └──┘    ┌──┐  │  D-1
                    │          │B │  │
              C-2   │  ┌──┐    └──┘  │
                    │  │D │          │  A-3
              D-2   │  └──┘          │
                    │                │
                    └────────────────┘
                    
  每个物理节点有 3 个虚拟节点 (均匀分散)
  key → 顺时针找到第一个虚拟节点 → 对应物理节点
  加删节点: 只影响环上相邻的一小段 (而不是全部!)
```

## 五、分布式事务 TCC 模型

```
正常流程:
  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │   Try    │────→│ Confirm  │────→│  完成    │
  │ 预留资源  │     │ 确认提交  │     │          │
  └──────────┘     └──────────┘     └──────────┘

异常流程:
  ┌──────────┐     ┌──────────┐
  │   Try    │────→│  Cancel  │  (任一 Try 失败)
  │ 预留资源  │     │ 取消释放  │
  └──────────┘     └──────────┘

例: A 转账 100 元给 B
  Try:
    A 账户: 冻结 100 元 (余额 1000→900可用,100冻结)
    B 账户: 预留入账额度
    
  Confirm (都 Try 成功):
    A: 冻结正式扣除 (余额 900)
    B: 余额 +100
    
  Cancel (任一 Try 失败):
    A: 解冻 100 元 (余额恢复 1000)
    B: 取消预留额度

⚠️ Confirm 和 Cancel 必须幂等!
```

## 六、雪花算法 ID 结构

```
64bit 结构:
┌─┬──────────────────────┬────────────┬──────────────┐
│0│     41bit 时间戳      │ 10bit 机器  │ 12bit 序列号  │
│ │ (毫秒, 从起始时间算)   │ (1024节点)  │ (4096/ms/机器)│
└─┴──────────────────────┴────────────┴──────────────┘

性能:
  每毫秒 4096 个 → 409.6 万/s 每节点
  时间戳在前 → 整体趋势递增 (有利于 B+ 树插入)

时钟回拨处理:
  ┌─ 回拨 < 5ms    → sleep 等待时钟追上
  ├─ 回拨 < 100ms  → 切换备用 workerId
  └─ 回拨 > 100ms  → 抛异常，拒绝服务
```

## 七、令牌桶 vs 漏桶

```
漏桶 (Leaky Bucket):                令牌桶 (Token Bucket):
                                    
  请求 ──→ ┌──────┐                  令牌 ──→ ┌──────┐
  ●●●●●   │      │                  每100ms  │      │
  ●●●●    │ ~~~~ │                  放入一个  │████  │ ← 最多存 N 个令牌
  ●●●     │ ~~~~ │                  └─→      │████  │
  ●●      │ ~~~~ │                            │      │
  ──→      │ ~~~~ │ ──→ 匀速流出               └──┬───┘
           └──────┘                              │
                                                │
  溢出直接丢弃!                                  ▼
  不允许突发                             请求 ──→ ┌────┐
                                            取令牌→│放行│
                                                └────┘
                                          桶里有多少令牌
                                          就能处理多少突发

  关键区别:
    漏桶: 强制平滑, 突发流量直接丢弃
    令牌桶: 允许突发 (桶容量 = 最大突发量)
    场景: 令牌桶更适合互联网 (允许一定突发)
```

## 九、Kafka 高吞吐架构

```
Producer 批量发送               Consumer 批量拉取
  ┌──┐ ┌──┐ ┌──┐                ┌──┐ ┌──┐ ┌──┐
  │P1│ │P2│ │P3│                │C1│ │C2│ │C3│
  └┬─┘ └┬─┘ └┬─┘                └┬─┘ └┬─┘ └┬─┘
   │     │     │                  │     │     │
   │     │     │                  │     │     │
   ▼     ▼     ▼                  ▲     ▲     ▲
  ┌──────────────────────────────────────────┐
  │               Kafka Broker                │
  │                                           │
  │  Topic: orders                            │
  │  ┌──────┬──────┬──────┐                   │
  │  │Part0 │Part1 │Part2 │ ← 分区并行        │
  │  │ [0]  │ [0]  │ [0]  │                   │
  │  │ [1]  │ [1]  │ [1]  │ ← 顺序追加(append)│
  │  │ [2]  │ [2]  │ [2]  │                   │
  │  └──────┴──────┴──────┘                   │
  │     顺序写磁盘                               │
  │     ↑ Page Cache 加速                      │
  │     ↑ sendfile 零拷贝 → Consumer            │
  └──────────────────────────────────────────┘

5 大加速原因:
  ① 顺序写盘 (无随机IO)
  ② Page Cache 缓冲 (尽量内存命中)
  ③ sendfile 零拷贝 (不经过用户态)
  ④ 批量+压缩 (减少网络+磁盘压力)
  ⑤ 分区并行 (多 Partition 并行读写)
```

## 十、一句话速记卡片

| 概念 | 一句话 |
|------|--------|
| CAP | 分布式系统 P 必选，实际是 CP vs AP 的选择 |
| BASE | AP 系统的实践：基本可用 + 软状态 + 最终一致 |
| Raft | Leader选举 + 日志复制，Paxos 的简化版 |
| MQ 三大作用 | 解耦 / 削峰 / 异步 |
| 消息不丢失 | 生产者 ACK + MQ 持久化 + 消费者手动 ACK |
| 幂等 | DB 唯一约束 / Redis NX / 状态机 |
| TCC | Try 预留 → Confirm 提交 或 Cancel 回滚 |
| 雪花算法 | 时间戳 + 机器ID + 序列号，趋势递增 |
| 一致性哈希 | 映射到环 + 虚拟节点防倾斜 |
| 令牌桶 vs 漏桶 | 漏桶不允许突发，令牌桶允许 |
| 熔断 | 失败多→快速失败→半开探测→恢复或继续熔断 |
| Kafka 快 | 顺序写 + 零拷贝 + PageCache + 批量 + 分区并行 |
