# 中间件与分布式面试核心框架

> 分布式理论（CAP、BASE、一致性协议）是基础；**消息队列**和**分布式事务**是高频考点。

---

## 一、分布式基础理论

### 1.1 CAP 定理

```
        C (Consistency)
        一致性
        /        \
       /          \
      /     ❌     \     ← 只能同时满足两个!
     /   不可能三角   \
    /                \
   /                  \
  A (Availability) —— P (Partition Tolerance)
  可用性                  分区容错性
```

| 选择 | 代表系统 | 放弃什么 |
|------|---------|---------|
| **CP** | ZooKeeper、etcd | 可用性（网络分区时部分节点不可用） |
| **AP** | Eureka、Cassandra | 强一致性（可能读到旧数据） |
| CA | 传统关系型DB（单机） | 分区容错（分布式系统无法放弃P） |

> 🔥 面试重点：**分布式系统中 P 是必须要的**（网络分区不可避免），所以实际是 **CP vs AP** 的选择。

### 1.2 BASE 理论（对 CAP 的妥协）

BASE 是 **AP 系统的实践指导**：

| 缩写 | 全称 | 含义 |
|------|------|------|
| **BA** | Basically Available | 基本可用（允许部分故障，不影响核心功能） |
| **S** | Soft State | 软状态（允许数据有中间状态） |
| **E** | Eventually Consistent | 最终一致性（不要求立刻一致，最终一致就行） |

> 对比 ACID：ACID 追求强一致性，BASE 追求可用性下的最终一致性。

### 1.3 分布式一致性协议

| 协议 | 类型 | 难度 | 应用 |
|------|------|------|------|
| **2PC** | 强一致性（阻塞） | 简单 | 数据库 XA 事务 |
| **3PC** | 强一致性（非阻塞） | 中等 | 很少用 |
| **TCC** | 最终一致性 | 较难 | 分布式事务（Try-Confirm-Cancel） |
| **Paxos** | 强一致性 | 最难 | ZooKeeper (ZAB)、etcd (Raft) |
| **Raft** | 强一致性 | 中等 | etcd、Consul、TiKV |

> 🔥 面试追问：**Paxos 和 Raft 的区别？** → Raft 是 Paxos 的简化版，强调可理解性，把一致性拆成 Leader 选举 + 日志复制两个独立子问题。

## 二、消息队列 ⭐⭐

### 2.1 为什么要用消息队列？

| 作用 | 说明 | 例子 |
|------|------|------|
| **解耦** | 生产者和消费者互不知道对方 | 下单 → MQ → 发短信/扣库存/记日志 |
| **削峰** | 缓冲瞬时流量 | 秒杀：请求进 MQ → 后端慢慢处理 |
| **异步** | 非核心逻辑异步处理 | 注册后发邮件可以异步（不用等邮件发完才返回） |

### 2.2 主流 MQ 对比

| 特性 | Kafka | RocketMQ | RabbitMQ |
|------|-------|----------|----------|
| **语言** | Java/Scala | Java | Erlang |
| **吞吐量** | ⚡ 极高（百万 TPS） | 高（十万 TPS） | 一般（万 TPS） |
| **延迟** | ms 级 | ms 级 | μs 级 |
| **可靠性** | 高（副本机制） | 高（同步刷盘+主从） | 高 |
| **事务消息** | ❌ | ✅ | ❌ |
| **适用场景** | 大数据、日志、流处理 | 电商、金融（阿里系） | 通用、实时性要求高 |
| **消费模式** | pull（拉） | pull（拉） | push/pull |

> 国内大厂：Kafka 大数据管道 + RocketMQ 业务消息。

### 2.3 消息可靠性如何保证？

```
生产者 → MQ → 消费者
  │       │      │
  │       │      └─③ 消费确认 (ACK)
  │       │        手动ACK: 处理成功才确认
  │       │        自动ACK: 消费即确认(可能丢消息)
  │       │
  │       └─② MQ 自身持久化
  │         同步刷盘: 消息写磁盘才返回成功 (可靠性高,性能差)
  │         异步刷盘: 先写缓存再刷盘 (性能高,可能丢)
  │         副本机制: 消息复制到多个节点
  │
  └─① 生产者确认 (ACK)
    同步发送: 等 Broker 确认才返回 (可靠性高)
    异步发送: 发完就不管 (性能高,可能丢)
```

### 2.4 消息重复消费怎么处理？

**核心思路：消费者做幂等**。

```
方案1: 数据库唯一约束
  INSERT INTO order_log (msg_id, ...)  -- msg_id 唯一索引
  → 重复消息 → INSERT 失败 → 忽略

方案2: Redis 去重
  SET msg_id "1" NX EX 3600
  → 已存在 → 说明消费过 → 跳过

方案3: 业务状态机
  UPDATE order SET status='paid' WHERE id=100 AND status='unpaid'
  → 第二次执行 affected_rows=0 → 跳过
```

### 2.5 Kafka 核心概念速查

```
Kafka 架构:
  Producer → ┌──────────────────────┐ → Consumer
             │       Broker          │
             │  ┌──────────────────┐│
             │  │     Topic        ││
             │  │ ┌──────┬──────┐  ││
             │  │ │Part0 │Part1 │  ││
             │  │ │  ↑   │  ↑   │  ││
             │  │ │ Repl │ Repl │  ││
             │  │ └──────┴──────┘  ││
             │  └──────────────────┘│
             └──────────────────────┘

核心概念:
  Topic       → 消息分类 (类似表)
  Partition   → 分区 (并行+扩展), 每个分区内有序
  Offset      → 消息在分区内的唯一序号
  Consumer Group → 消费者组, 组内一个分区只被一个消费者消费
  ISR         → In-Sync Replicas, 同步副本集合
```

## 三、分布式事务

### 3.1 解决方案对比

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|---------|
| **2PC / XA** | 强一致 | 低（阻塞） | 低 | 传统数据库 |
| **TCC** | 最终一致 | 中 | 高 | 金融（Try-Confirm-Cancel） |
| **本地消息表** | 最终一致 | 中 | 中 | 通用 |
| **MQ 事务消息** | 最终一致 | 高 | 中 | RocketMQ |
| **Saga** | 最终一致 | 高 | 中 | 长事务、微服务 |

### 3.2 TCC (Try-Confirm-Cancel)

```
转账 100 元: A → B

Try (预留资源):
  A: 冻结 100 元 (余额不减)
  B: 预留入账额度

Confirm (确认):
  A: 冻结的 100 正式扣除
  B: 余额 +100

Cancel (取消, 任意一步失败):
  A: 解冻 100 元
  B: 取消预留额度
```

> 难点：Confirm 和 Cancel 必须**幂等**（可能被重试）。

### 3.3 MQ 事务消息（RocketMQ）

```
1. 发送 half 消息 (对消费者不可见)
2. 执行本地事务
   ├── 成功 → 提交 half 消息 → 消费者可见
   └── 失败 → 回滚 half 消息 → 消费者不可见
3. 如果 half 消息状态未知 (超时)，MQ 回查本地事务状态
```

## 四、分布式 ID 生成

| 方案 | 优点 | 缺点 |
|------|------|------|
| **UUID** | 简单，本地生成 | 太长、无序（B+树插入性能差） |
| **数据库自增** | 简单、有序 | 单点瓶颈 |
| **号段模式** | 批量取号，性能好 | 有号段浪费 |
| **雪花算法 (Snowflake)** ⭐ | 高性能、趋势递增 | 依赖时钟回拨处理 |

### 雪花算法结构

```
41bit 时间戳      10bit 机器ID    12bit 序列号
│                  │               │
├──────────────────┼───────────────┼────────┤
│      0...0       │   0...0       │ 0...0  │
│   (约69年)        │  (1024节点)   │(4096/ms)│
└──────────────────┴───────────────┴────────┘

总长 64bit = long 类型
理论峰值: 409.6 万/s 每节点
趋势递增 (非严格递增): 时间戳在高位，整体递增
```

> 🔥 面试追问：**时钟回拨怎么处理？** → ① 等到时钟追上 ② 用备用机器 ID ③ 抛异常拒绝服务。

## 五、一致性哈希

### 5.1 为什么需要一致性哈希？

```
传统 hash 取模:
  node = hash(key) % N
  N=3: A B C

  加一个节点 N=4:
  A B C D
  大部分 key 的 hash 结果都变了 → 大量缓存失效 → 缓存雪崩!

一致性哈希:
  把节点和 key 都映射到 0 ~ 2^32-1 的环上
  加节点时，只有环上一小段范围受影响
```

### 5.2 虚拟节点

```
物理节点少 → 数据倾斜 → 加虚拟节点

Node-A → 虚拟节点: A-1, A-2, A-3 (均匀分散在环上)
Node-B → 虚拟节点: B-1, B-2, B-3
Node-C → 虚拟节点: C-1, C-2, C-3

这样就均匀了！
```

## 六、负载均衡算法

| 算法 | 原理 | 适用场景 |
|------|------|---------|
| **轮询** | 依次分配 | 各节点性能一致 |
| **加权轮询** | 按权重分配 | 节点性能不一致 |
| **最少连接** | 找连接数最少的 | 长连接场景 |
| **一致性哈希** | 同 key 到同节点 | 需要会话保持 |
| **随机** | 随机选 | 简单的 |

## 七、服务治理

### 7.1 限流算法

| 算法 | 原理 | 特点 |
|------|------|------|
| **计数器** | 固定时间窗口计数 | 简单，有临界问题 |
| **滑动窗口** | 时间窗口滑动 | 平滑，比计数器好 |
| **漏桶** | 恒定速率流出 | 强制平滑，可能堆积 |
| **令牌桶** ⭐ | 恒定速率放入令牌，请求取令牌 | 允许突发流量 |

```
令牌桶:
  每隔 1/QPS 秒放入一个令牌
  桶容量 = 最大突发数
  
  请求来了 → 尝试取令牌
    ├── 有令牌 → 放行
    └── 无令牌 → 限流/等待
```

### 7.2 熔断、降级、限流 区别

| 手段 | 触发条件 | 作用 |
|------|---------|------|
| **限流** | 流量超过阈值 | 主动控制入口流量 |
| **熔断** | 下游连续失败 | 快速失败，防止级联故障 |
| **降级** | 系统压力大/故障 | 关非核心功能，保核心链路 |

## 八、RPC 框架核心

```
RPC 调用流程:
  Client                         Server
    │                              │
    │ ① 调用本地 Stub (代理)         │
    ▼                              │
  ┌──────────┐                     │
  │ 序列化    │ 对象 → 二进制        │
  ├──────────┤                     │
  │ 网络传输  │ TCP/HTTP            │
  └────┬─────┘                     │
       │ ────────────────────────→ │
       │                           ▼
       │                     ┌──────────┐
       │                     │ 网络接收  │
       │                     ├──────────┤
       │                     │ 反序列化  │ 二进制 → 对象
       │                     ├──────────┤
       │                     │ 执行方法  │
       │                     ├──────────┤
       │                     │ 序列化结果│
       │                     └────┬─────┘
       │ ◁────────────────────────  │
       ▼                           │
  ┌──────────┐                     │
  │ 反序列化  │                     │
  └──────────┘                     │
       │                           │
       ▼                           │
    返回结果
```

> 核心组件：代理层、序列化层、网络传输层、注册中心。
