# 中间件与分布式高频面试题精讲

> 分布式基础理论 + 消息队列是校招高频。每题 2-3 分钟。

---

## 分布式理论

### Q1：说说 CAP 理论，分布式系统为什么不能同时满足三者？

**CAP**：一致性(Consistency)、可用性(Availability)、分区容错性(Partition Tolerance)。

**为什么不能三者兼得**？因为网络分区（P）发生时：
- 如果保证**一致性(C)**：必须等分区恢复才能确认写操作 → 牺牲可用性(A) → **CP 系统**
- 如果保证**可用性(A)**：允许分区节点继续服务 → 可能数据不一致 → 牺牲一致性(C) → **AP 系统**

> **P 是分布式系统必须面对的**（网络故障不可避免），所以实际选择是 **CP 还是 AP**。

**典型系统**：
- CP：ZooKeeper、etcd（少数不可用时拒绝服务）
- AP：Eureka、Cassandra（允许短暂不一致）

---

### Q2：BASE 理论是什么？和 ACID 什么关系？

BASE 是 AP 系统的实践理论，是对 CAP 的妥协：

| | ACID | BASE |
|------|------|------|
| 一致性 | 强一致性 | **最终一致性** |
| 核心关注 | 数据正确 | 系统可用 |
| 代表 | 传统关系型数据库 | NoSQL、分布式系统 |

- **BA** (Basically Available)：基本可用 —— 允许部分故障
- **S** (Soft State)：软状态 —— 数据可以有中间态
- **E** (Eventually Consistent)：最终一致性 —— 不要求立刻一致

> 一句话：ACID 追求强一致，BASE 追求高可用下的最终一致。

---

### Q3：Paxos 和 Raft 有什么区别？

**核心区别**：Raft 是 Paxos 的简化版，强调"可理解性"。

| | Paxos | Raft |
|------|-------|------|
| 难度 | 极难理解 + 实现 | 较容易理解 |
| Leader | 可以有多个 proposer | 强 Leader 模型 |
| 阶段拆分 | 混在一起 | **Leader选举 + 日志复制**两个独立阶段 |
| 应用 | Chubby | etcd、Consul、TiKV |

> Raft 的核心创新：把一致性算法拆成两个清晰的子问题，外加安全约束。

---

### Q4：Raft 协议的大致流程？

```
1. Leader 选举 (Term 递增):
   节点状态: Follower → Candidate → Leader
   Candidate 发起投票，获得多数票 → 成为 Leader
   一个 Term 最多一个 Leader

2. 日志复制:
   Client → Leader → 写入本地日志
   Leader → Followers (AppendEntries RPC)
   多数 Followers 确认 → Leader 提交 → Apply 到状态机
   Followers 收到 Leader 的 commit 通知 → Apply

3. 安全性保证:
   - Leader 的日志必须包含所有已提交的日志
   - 选举时 Candidate 的日志必须不旧于多数节点
```

## 消息队列

### Q5：消息队列有什么作用？

1. **解耦**：生产者和消费者互不知道对方存在。订单系统不需要知道发短信服务的地址。
2. **削峰**：秒杀时请求先进 MQ，后端按能力消费，不会被打垮。
3. **异步**：非核心逻辑异步处理（注册→发邮件可以异步），提高响应速度。

---

### Q6：如何保证消息不丢失？

| 阶段 | 手段 |
|------|------|
| **生产者→MQ** | 发送确认(ACK)：同步发送等 Broker 确认；异步发送+回调 |
| **MQ 自身** | 同步刷盘 + 多副本(min.insync.replicas > 1) |
| **MQ→消费者** | 手动 ACK：处理完业务再确认，不自动确认 |

```
Kafka 配置:
  acks=all (所有 ISR 副本确认)
  enable.auto.commit=false (手动提交 offset)
  replication.factor ≥ 3
  min.insync.replicas ≥ 2
```

---

### Q7：消息重复了怎么办？（幂等性）

**一句话**：消费者做幂等，保证重复消费结果不变。

| 方案 | 实现 |
|------|------|
| **DB 唯一约束** | INSERT 带唯一 msg_id，重复插入失败 |
| **Redis 去重** | SET msg_id "1" NX EX 3600 |
| **状态机** | UPDATE SET status='done' WHERE status='pending' → 第二次查不到 |

> 不要依赖 MQ 的不重复投递保证，**消费者必须做幂等**。

---

### Q8：消息积压了怎么办？

```
紧急处理:
  1. 增加消费者实例（水平扩展）
  2. 临时提高消费线程数
  3. 降级非核心业务（集中资源消核心消息）

长期方案:
  1. 监控消费积压告警
  2. 消费能力不足 → 扩容
  3. 生产太快 → 限流
  4. 消费慢 → 优化消费逻辑（批量、并行处理）
```

---

### Q9：Kafka 为什么吞吐量这么高？

1. **顺序读写**：写入是追加（append），避免了随机 IO
2. **零拷贝**：sendfile 系统调用，数据不经过用户态
3. **Page Cache**：利用 OS 的页缓存，读写尽量命中内存
4. **批量+压缩**：消息批量发送，支持 GZIP/Snappy/LZ4 压缩
5. **分区并行**：多个分区可以并行读写

> **核心**：顺序写盘 ≠ 顺序读盘。磁盘顺序写可以接近内存速度（600MB/s+）。

---

### Q10：Kafka 的 ISR 是什么？

**ISR = In-Sync Replicas**（同步副本集合）。

```
Partition 有 3 个副本: Leader + Follower1 + Follower2

如果 Follower2 落后太多 (超过 replica.lag.time.max.ms):
  ISR = {Leader, Follower1}  (Follower2 被踢出)

Leader 挂了:
  新 Leader 只能从 ISR 中选
  → 保证数据不丢失 (acks=all 时有保障)
```

> ISR 是 Kafka 在数据可靠性和可用性之间的平衡机制。

## 分布式事务

### Q11：分布式事务有哪些解决方案？

| 方案 | 一致性 | 性能 | 典型场景 |
|------|--------|------|---------|
| **2PC (XA)** | 强一致 | 低（阻塞） | 同数据库跨服务 |
| **TCC** | 最终一致 | 中 | 金融（Try-Confirm-Cancel） |
| **本地消息表** | 最终一致 | 中 | 通用 |
| **MQ 事务消息** | 最终一致 | 高 | RocketMQ 场景 |
| **Saga** | 最终一致 | 高 | 长事务 |

> 校招面试说清楚 **TCC 和 MQ 事务消息**就行。

---

### Q12：2PC 的缺点是什么？

1. **同步阻塞**：第一阶段准备时，参与者锁住资源直到协调者通知提交/回滚
2. **单点故障**：协调者宕机 → 参与者一直阻塞等待
3. **数据不一致**：第二阶段部分参与者提交成功，部分失败（协调者崩溃时机尴尬）
4. **性能差**：多次网络交互 + 所有参与者的最慢者决定速度

---

### Q13：TCC 和 2PC 的区别？

| | 2PC | TCC |
|------|-----|-----|
| 粒度 | 资源层（数据库 XA） | 业务层 |
| 一致性 | 强一致 | 最终一致 |
| 性能 | 差（长事务锁资源） | 好（不长时间锁资源） |
| 实现 | 数据库/中间件支持 | 业务代码自己实现 Try/Confirm/Cancel |
| 幂等要求 | 不需要 | **必须幂等**（Confirm/Cancel 可能重试） |

## 服务治理

### Q14：限流算法有哪些？令牌桶和漏桶的区别？

| 算法 | 特点 |
|------|------|
| 计数器 | 简单，有临界问题（窗口切换瞬间） |
| 滑动窗口 | 平滑，解决临界问题 |
| **漏桶** | 恒定速率流出，**不允许突发** |
| **令牌桶** | 恒定速率放入令牌，**允许突发**（桶里有令牌就能发） |

```
漏桶: 水龙头(请求) → 桶 → 小孔匀速流出
令牌桶: 令牌匀速放入桶 → 请求来了拿令牌

关键区别:
  漏桶: 突发流量直接溢出(拒绝)
  令牌桶: 桶里有积攒的令牌就可以处理突发
```

> 实际常用令牌桶（如 Guava RateLimiter、Sentinel）。

---

### Q15：熔断、降级、限流有什么区别？

```
流量走向: 请求 → 限流(入口) → 服务调用 → 熔断(保护) → 降级(兜底)

限流:   "进来的人太多，拦掉一部分"
        控制入口流量，保护系统不过载

熔断:   "下游坏了，别调了，调也是浪费"
        快速失败，防止级联故障 (Hystrix/Sentinel)
        状态: 关闭 → 打开（熔断） → 半开（探测）

降级:   "撑不住了，关掉非核心功能"
        保核心链路，牺牲非核心功能
        例: 双11关闭退货功能，保下单
```

### Q16：服务注册与发现怎么工作的？

```
服务提供者:
  1. 启动时向注册中心注册 (IP:Port + 元数据)
  2. 定期发送心跳
  3. 下线时注销

服务消费者:
  1. 从注册中心拉取服务列表
  2. 本地缓存 + 订阅更新
  3. 负载均衡选择一个实例调用

注册中心: (ZooKeeper / Nacos / Consul / Eureka)
  存储: 服务名 → [实例列表]
  健康检查: 踢出不健康的实例
```

> Eureka AP vs ZooKeeper CP：Eureka 优先保证可用性，ZooKeeper 优先保证一致性。

## 分布式 ID

### Q17：雪花算法 (Snowflake) 的结构？时钟回拨怎么处理？

```
64bit = 1bit(保留) + 41bit(时间戳,69年) + 10bit(机器ID,1024节点) + 12bit(序列号,4096/ms)

时钟回拨处理:
  1. 短时间回拨 (< 几ms) → 等时钟追上再生成
  2. 中等回拨 → 用备用 WorkerId
  3. 大量回拨 → 抛异常，拒绝服务
  4. 美团的 Leaf 方案: 号段模式，不依赖时间戳
```

### Q18：一致性哈希是什么？为什么加虚拟节点？

**普通哈希取模**的问题：节点数变了 → 几乎所有 key 重新分配 → 缓存雪崩！

**一致性哈希**：映射到 0 ~ 2^32-1 的环，节点和 key 都落在环上，顺时针找第一个节点。加删节点只影响一小段。

**虚拟节点**：物理节点映射为多个虚拟节点分散在环上，解决数据倾斜问题。
