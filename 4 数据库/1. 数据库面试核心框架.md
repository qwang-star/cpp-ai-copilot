## 一、MySQL 架构概览

```
┌─────────────────────────────────────────────┐
│              连接器 / 连接池                    │
│          (认证、权限、连接管理)                  │
├─────────────────────────────────────────────┤
│      查询缓存 (8.0 已废弃)    │  分析器         │
│                              │ (词法/语法)     │
├──────────────────────────────┤                │
│           优化器              │   执行器        │
│    (索引选择、JOIN 顺序)       │ (调用引擎接口)   │
├─────────────────────────────────────────────┤
│              存储引擎层                        │
│   ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐    │
│   │InnoDB│  │MyISAM│  │Memory│  │Archive│    │
│   └──────┘  └──────┘  └──────┘  └──────┘    │
└─────────────────────────────────────────────┘
```

### InnoDB vs MyISAM

| 维度 | InnoDB | MyISAM |
|------|--------|--------|
| **事务** | ✅ 支持 | ❌ 不支持 |
| **行级锁** | ✅ | ❌ (表锁) |
| **外键** | ✅ | ❌ |
| **崩溃恢复** | ✅ (redo log) | ❌ |
| **MVCC** | ✅ | ❌ |
| **全文索引** | 5.6+ 支持 | ✅ |
| **count(*)** | 慢（逐行扫描） | 快（存了行数） |
| **数据存储** | 聚集索引（数据即索引） | 分开存（.MYD + .MYI） |

> 线上默认 InnoDB，MyISAM 几乎不用了。

## 二、索引 ⭐⭐⭐

### 2.1 B+ 树结构（核心）

```
                 ┌─────────┐
                 │ [30|60] │  ← 非叶子节点（只存索引）
                 └─┬───┬───┘
          ┌────────┘   │   └────────┐
          ▼            ▼            ▼
    ┌────────┐   ┌────────┐   ┌────────┐
    │[10|20] │   │[40|50] │   │[70|80] │  ← 非叶子节点
    └──┬──┬──┘   └──┬──┬──┘   └──┬──┬──┘
       │  │         │  │         │  │
    ┌──┘  └──┐   ┌──┘  └──┐   ┌──┘  └──┐
    ▼       ▼   ▼       ▼   ▼       ▼
┌──────┬──────┬──────┬──────┬──────┬──────┐
│10→R1 │20→R2 │40→R3 │50→R4 │70→R5 │80→R6 │ ← 叶子节点
│      │      │      │      │      │      │    (存数据/主键值)
└──────┴──────┴──────┴──────┴──────┴──────┘
     ◄──── 双向链表连接 ────►
```

**B+ 树特点**：
- 数据只存叶子节点
- 叶子节点用**双向链表**连接 → 范围查询极快
- 非叶子节点只存索引 → 每页能存更多索引，树更矮
- 树高通常 2-4 层 → 2-4 次 IO 就能定位数据

> 🔥 面试追问：**为什么用 B+ 树不用 B 树？** → B+ 树数据只存叶子，非叶子存更多索引→树更低；叶子有链表→范围查询 O(log n) 定位后顺序读。

### 2.2 聚集索引 vs 非聚集索引

```
聚集索引 (Clustered Index / 主键索引):
  B+ 树叶子节点存的是 ┌──────────┐
  **完整行数据**！     │ PK │ Col1 │ Col2 │ Col3 │
                      └──────────┘

非聚集索引 (Secondary Index / 二级索引):
  B+ 树叶子节点存的   ┌──────────┐
  是**主键值**！      │ idx_col │  PK  │
                      └──────────┘
  
  查到主键后 → 再去聚集索引查完整数据 → **回表**
```

### 2.3 回表与覆盖索引

```
回表 (Using index condition):
  SELECT * FROM t WHERE name='张三'
  → 走 name 索引 → 拿到主键 id=100
  → 拿着 id=100 去主键索引 → 拿到完整行
  → 两次查 B+ 树！

覆盖索引 (Using index):
  SELECT id, name FROM t WHERE name='张三'
  → 走 name 索引 → 直接拿到 id + name
  → 不需要回表！一次搞定！
  
  Extra: Using index  ← explain 看到这个就放心了
```

### 2.4 最左前缀原则

```
联合索引: INDEX(a, b, c)

有效查询（走索引）:
  WHERE a=1              ✅ 用到 a
  WHERE a=1 AND b=2      ✅ 用到 a,b
  WHERE a=1 AND b=2 AND c=3 ✅ 用到全部
  WHERE a=1 AND c=3      ✅ 用到 a (c 被跳过，用不上)

无效查询（不走索引）:
  WHERE b=2              ❌ 跳过了 a
  WHERE b=2 AND c=3      ❌ 跳过了 a
  WHERE c=3              ❌ 跳过了 a,b

范围查询在联合索引中的陷阱:
  WHERE a=1 AND b>2 AND c=3
  → 用到 a,b (c 失效，因为 b 是范围查询!)
```

### 2.5 索引失效场景

| 场景 | 原因 |
|------|------|
| `WHERE name LIKE '%三'` | 左模糊，无法用索引 |
| `WHERE age/2 = 10` | 索引列参与运算 |
| `WHERE name = CONCAT('张','三')` | 对列用函数 |
| `WHERE a=1 OR b=2` | OR 两边不是同一索引 |
| `WHERE phone = 13800138000` (phone 是 varchar) | 类型隐式转换 = 对列用函数 |
| `!=` / `NOT IN` | 通常不走索引 |
| `IS NULL` / `IS NOT NULL` | 看数据分布 |

## 三、事务 (Transaction) ⭐⭐

### 3.1 ACID

| 特性 | 含义 | 实现方式 |
|------|------|---------|
| **A** 原子性 | 要么全做，要么全不做 | **undo log** |
| **C** 一致性 | 事务前后数据满足约束 | 由其他三个保证 |
| **I** 隔离性 | 并发事务互不干扰 | **MVCC + 锁** |
| **D** 持久性 | 提交后数据不丢失 | **redo log** |

### 3.2 四种隔离级别

| 级别 | 脏读 | 不可重复读 | 幻读 | 实现 |
|------|------|-----------|------|------|
| READ UNCOMMITTED | ✅会 | ✅会 | ✅会 | 无锁 |
| READ COMMITTED | ❌ | ✅会 | ✅会 | 每次读生成 ReadView |
| REPEATABLE READ (默认) | ❌ | ❌ | ⚠️几乎解决 | 事务开始时生成 ReadView |
| SERIALIZABLE | ❌ | ❌ | ❌ | 读写都加锁 |

```
脏读:      读到别人未提交的数据（可能回滚）
不可重复读: 同一事务内两次读同一行，值不同（别人 update 提交了）
幻读:      同一事务内两次查，行数不同（别人 insert 提交了）
```

> InnoDB 的 REPEATABLE READ 用 **MVCC + 间隙锁** 基本解决了幻读。

### 3.3 MVCC (多版本并发控制)

```
核心：每行隐藏两个字段
  DB_TRX_ID  (6B) — 最后修改该行的事务 ID
  DB_ROLL_PTR (7B) — 回滚指针，指向 undo log

ReadView: 快照读时的"可见性判断"
  包含: 当前活跃事务列表 + 最小/最大事务 ID

可见性规则:
  行 trx_id < ReadView 最小 trx_id → 可见 (已提交)
  行 trx_id > ReadView 最大 trx_id → 不可见 (未来事务)
  行 trx_id 在活跃列表中 → 不可见 (未提交)

快照读 vs 当前读:
  SELECT (普通)      → 快照读 (MVCC，无锁)
  SELECT ... FOR UPDATE → 当前读 (加锁)
  UPDATE / DELETE    → 当前读 (加锁)
```

## 四、日志系统

### 4.1 redo log vs undo log vs binlog

| | redo log | undo log | binlog |
|------|---------|---------|--------|
| **层次** | InnoDB 引擎层 | InnoDB 引擎层 | MySQL Server 层 |
| **作用** | 崩溃恢复（重做） | 事务回滚 + MVCC | 主从复制 + 数据恢复 |
| **记录内容** | 物理日志（页的修改） | 逻辑日志（反向操作） | 逻辑日志（SQL 语句/行变化） |
| **写时机** | 事务执行中持续写 | 修改前写（支持回滚） | 事务提交时写 |
| **文件大小** | 固定大小，循环写 | 自动清理 | 可配置 |

### 4.2 WAL (Write-Ahead Logging)

```
先写日志，再写磁盘

更新过程:
1. 修改 Buffer Pool 中的数据页
2. 写 redo log (顺序写，快！)
3. 返回客户端"提交成功"
4. 后台线程异步把脏页刷入磁盘

崩溃恢复:
  重启 → 检查 redo log → 
  已提交的事务: 重做 (前滚)
  未提交的事务: 回滚 (undo log)
```

## 五、Redis ⭐⭐

### 5.1 五种基本数据结构

| 结构 | 底层 | 常用命令 | 场景 |
|------|------|---------|------|
| **String** | SDS | GET/SET/INCR | 缓存、计数器、分布式锁 |
| **Hash** | 压缩表/哈希表 | HSET/HGET/HGETALL | 对象缓存 |
| **List** | 快速链表 | LPUSH/RPOP/BRPOP | 消息队列、最新列表 |
| **Set** | 整数集/哈希表 | SADD/SINTER/SUNION | 去重、共同好友 |
| **ZSet** | 压缩表/跳表 | ZADD/ZRANGE/ZRANK | 排行榜、延时队列 |

### 5.2 缓存三大问题

```
缓存穿透:
  查不存在的 key → 穿到 DB
  解决: 布隆过滤器 / 缓存空值

缓存击穿:
  热点 key 过期 → 大量请求打到 DB
  解决: 互斥锁 / 永不过期 + 异步更新

缓存雪崩:
  大量 key 同时过期 → DB 瞬间被打爆
  解决: 过期时间加随机 / 多级缓存 / 限流降级
```

### 5.3 持久化：RDB vs AOF

| | RDB | AOF |
|------|-----|-----|
| 方式 | 定期快照（SAVE/BGSAVE） | 写命令追加到文件 |
| 文件 | dump.rdb | appendonly.aof |
| 恢复速度 | 快 | 慢（需要重放所有命令） |
| 数据安全 | 可能丢最后一次快照后的数据 | 更安全（每秒刷盘） |
| 文件大小 | 小（压缩过） | 大 |
| 推荐 | 都开，Redis 4.0+ 混合持久化 |

### 5.4 内存淘汰策略

| 策略 | 行为 |
|------|------|
| **noeviction** | 不淘汰，写操作直接报错 |
| **allkeys-lru** | 所有 key 中淘汰 LRU (推荐) |
| **volatile-lru** | 有过期时间的 key 中淘汰 LRU |
| **allkeys-lfu** | 所有 key 中淘汰 LFU |
| **volatile-ttl** | 有过期时间的 key 中淘汰 TTL 最短的 |
| **allkeys-random** | 随机淘汰 |

### 5.5 分布式锁

```
SET lock_key unique_value NX EX 30
  NX: 不存在才 set
  EX: 过期时间

释放时用 Lua 脚本保证原子性:
  if redis.call("get", KEYS[1]) == ARGV[1] then
      return redis.call("del", KEYS[1])
  else return 0 end

为什么不用 SETNX + DEL?
  ① 不是原子的（SETNX 过期分开设，可能宕机永不过期）
  ② DEL 可能删了别人的锁（没校验值）

RedLock (多节点):
  向 N 个 Redis 节点依次获取锁
  超过半数(N/2+1)成功 → 获取成功
  用的少，有争议
```

## 六、SQL 优化

### 6.1 EXPLAIN 关键字段

| 字段 | 关注点 |
|------|--------|
| **type** | system > const > eq_ref > ref > range > index > ALL |
| **key** | 实际使用的索引 |
| **rows** | 预估扫描行数 |
| **Extra** | Using index(覆盖索引✅) / Using filesort(排序未用索引❌) / Using temporary(临时表❌) |

### 6.2 常见优化手段

```sql
-- 1. SELECT 只取需要的列，不用 SELECT *
-- 2. WHERE 条件列建索引
-- 3. JOIN 小表驱动大表
-- 4. ORDER BY 列建索引避免 filesort
-- 5. 分页优化 (大 offset 用子查询或游标)
SELECT * FROM t WHERE id > last_id LIMIT 20;  -- 不用 OFFSET 100000

-- 6. 适当冗余减少 JOIN（反范式化）
-- 7. 批量 INSERT 代替逐条 INSERT
```
