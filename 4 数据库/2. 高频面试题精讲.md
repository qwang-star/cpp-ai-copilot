# 数据库高频面试题精讲

> 数据库面试题分 MySQL 和 Redis 两条线，每条线各 10 题。

---

## MySQL

### Q1：为什么用 B+ 树做索引，不用 B 树 / 哈希 / 二叉树？

| 数据结构 | 为什么不合适 |
|---------|------------|
| **二叉树** | 顺序插入退化成链表，树太高→IO 次数多 |
| **哈希表** | 不支持范围查询，没法排序 |
| **B 树** | 非叶子存数据→每页索引少→树更高；叶子没链表→范围查询慢 |
| **B+ 树** ✅ | 数据只存叶子→树更矮；叶子有双向链表→范围查询顺序读 |

> **一句话**：B+ 树最矮（2-4 层）+ 范围查询天然快（叶子链表）。

---

### Q2：聚集索引和非聚集索引的区别？

| | 聚集索引 | 非聚集索引（二级索引） |
|------|---------|-------------------|
| 叶子存什么 | **完整行数据** | **主键值** |
| 每表几个 | 1 个（通常是主键） | 多个 |
| 插入顺序 | 按主键顺序（乱序插入→页分裂） | 按索引列顺序 |
| InnoDB 必建 | ✅（没主键自动选唯一索引→隐式 rowid） | 按需创建 |

> **回表**：非聚集索引查到主键 → 再去聚集索引查完整数据 = 查两次 B+ 树。

---

### Q3：覆盖索引是什么？怎么用？

**一句话**：查询的列都包含在索引里，直接从索引返回，不需要回表。

```sql
-- 索引: INDEX idx_name_age (name, age)

-- 回表 (2次B+树查找)
SELECT * FROM users WHERE name='张三';

-- 覆盖索引 (1次B+树查找)
SELECT name, age FROM users WHERE name='张三';
```

> EXPLAIN 里 Extra 显示 `Using index` 就说明用了覆盖索引。

---

### Q4：联合索引的最左前缀原则？

`INDEX(a, b, c)` 相当于建了 `(a)`、`(a,b)`、`(a,b,c)` 三个索引。

```
✅ WHERE a=1              → 用到 a
✅ WHERE a=1 AND b=2      → 用到 a,b  
✅ WHERE a=1 AND c=3      → 用到 a (c 断了)
❌ WHERE b=2              → 没用到（跳过了 a）
❌ WHERE b=2 AND c=3      → 没用到

⚠️ WHERE a=1 AND b>2 AND c=3
   → 用到 a,b, c 失效（b 是范围查询）
```

> **建联合索引的顺序**：区分度高的放前面，等值查询的放前面，范围查询的放后面。

### Q5：哪些情况会导致索引失效？

1. **左模糊** `LIKE '%abc'`（右模糊 `LIKE 'abc%'` 可以）
2. **对索引列用函数** `WHERE YEAR(create_time) = 2024`
3. **对索引列做运算** `WHERE age + 1 = 20`
4. **隐式类型转换** `WHERE phone = 13800138000`（phone 是 varchar，会转成数字比较）
5. **OR 条件** 两边不是同一索引（可以用 UNION 替代）
6. **NOT IN / !=** 通常不走（看优化器判断）
7. **联合索引不满足最左前缀**

### Q6：事务的隔离级别有哪些？MySQL 默认是什么？

| 级别 | 脏读 | 不可重复读 | 幻读 |
|------|------|-----------|------|
| READ UNCOMMITTED | ✅ | ✅ | ✅ |
| READ COMMITTED | ❌ | ✅ | ✅ |
| **REPEATABLE READ** (默认) | ❌ | ❌ | ⚠️ 几乎解决 |
| SERIALIZABLE | ❌ | ❌ | ❌ |

- **脏读**：读到别人未提交的数据 → RC 以上解决
- **不可重复读**：同一行两次读不一样 → RR 以上解决
- **幻读**：同一个条件结果行数不同 → RR用MVCC+间隙锁解决

---

### Q7：MVCC 是怎么实现读已提交和可重复读的？

**核心区别在 ReadView 的生成时机**：

```
READ COMMITTED:
  每次 SELECT 都生成新的 ReadView
  → 能看到别的事务提交的数据 → 不可重复读

REPEATABLE READ:
  第一次 SELECT 时生成 ReadView，之后复用
  → 全事务用同一个快照 → 可重复读

MVCC 可见性判断:
  行 trx_id < ReadView.min_trx → 可见 (已提交的老事务)
  行 trx_id > ReadView.max_trx → 不可见 (未来事务)
  行 trx_id 在活跃列表中 → 不可见 (未提交)
```

### Q8：redo log、undo log、binlog 的区别？

| | redo log | undo log | binlog |
|------|---------|---------|--------|
| 层 | InnoDB | InnoDB | Server |
| 作用 | 崩溃恢复，保证持久性 | 回滚 + MVCC | 主从复制 + 数据恢复 |
| 内容 | 物理（对页的修改） | 逻辑（反向 SQL） | 逻辑（SQL / 行数据） |
| 写时机 | 事务进行中就写 | 修改数据前写 | 事务提交时写 |
| 循环写 | ✅ 固定大小 | ❌ | ❌ |

> **崩溃恢复**：先 redo log 前滚已提交事务，再 undo log 回滚未提交事务。

### Q9：SQL 执行很慢，怎么排查？

```
1. EXPLAIN 看执行计划
   → type 是不是 ALL (全表扫描)?
   → key 用没用上索引?
   → rows 预扫描行数是不是很大?
   → Extra 有没有 Using filesort / Using temporary?

2. 检查是否索引失效（看 Q5）

3. 检查锁等待
   SHOW ENGINE INNODB STATUS;

4. 检查是否返回了过多数据
   → 加 LIMIT

5. 检查表数据量
   → 考虑分库分表 / 分区

6. 慢查询日志
   SET slow_query_log = 1;
   SET long_query_time = 2;
```

### Q10：分库分表怎么做？有哪些问题？

**拆分方式**：
- **垂直拆分**：按业务模块拆（用户表→用户库，订单表→订单库）
- **水平拆分**：按某字段（如 user_id）hash 到多个表/库

**核心问题**：
1. **跨库 JOIN** → 避免，或用全局表/冗余字段
2. **分布式事务** → 最终一致性（本地消息表/RocketMQ）
3. **全局 ID** → 雪花算法 / 号段模式
4. **跨分片排序/分页** → 各分片取 N 条 → 汇总后二次排序

## Redis

### Q11：Redis 为什么这么快？

1. **纯内存操作** → 大部分操作微秒级
2. **单线程**（命令处理） → 无锁、无上下文切换、无竞争
3. **IO 多路复用** → 一个线程处理多个连接（epoll）
4. **高效数据结构** → SDS/跳表/压缩表，专为 Redis 设计

> Redis 6.0+ 引入多线程处理网络 IO，但**命令执行仍是单线程**。

---

### Q12：Redis 的过期删除策略？

**惰性删除 + 定期删除** 组合：

```
惰性删除:
  访问 key 时检查是否过期 → 过期就删
  缺点：没人访问的过期 key 一直占内存

定期删除:
  每隔 100ms 随机抽一批 key 检查 → 过期比例高就继续抽
  缺点：CPU 消耗

两者配合: 懒惰删除清热点 key，定期删除清角落里的过期 key
```

---

### Q13：Redis 内存淘汰策略有哪些？

| 策略 | 适用场景 |
|------|---------|
| **noeviction** | 缓存绝对不能丢（不太现实） |
| **allkeys-lru** ⭐ | 通用缓存，最近最少用淘汰 |
| volatile-lru | 只淘汰有过期时间的 |
| **allkeys-lfu** | 热点数据场景 |
| volatile-ttl | 快过期的先淘汰 |
| allkeys-random | 简单的随机淘汰 |

> **推荐 allkeys-lru**，最通用。

---

### Q14：RDB 和 AOF 的区别？怎么选？

| | RDB | AOF |
|------|-----|-----|
| 方式 | 定期快照（子进程 bgsave） | 写命令追加 |
| 数据安全 | 可能丢最后一次快照后的 | 更安全（每秒 fsync） |
| 恢复速度 | **快**（二进制加载） | 慢（重放命令） |
| 文件大小 | 小 | 大（需要重写压缩） |
| 影响性能 | fork 子进程时短暂卡顿 | 持续写盘 |

> **建议：都开**。Redis 4.0+ 混合持久化，RDB 做全量备份，AOF 做增量。

### Q15：缓存穿透、击穿、雪崩怎么区分和解决？

```
          请求 Redis
              │
              ▼
         ┌────────┐
         │ 有数据?  │
         └───┬─┬──┘
       有数据 │ │ 没数据 ──→ 穿透：根本不在缓存/DB 中
             │ │            解决：布隆过滤器 / 缓存空值
             ▼ │
         返回     │
                  ▼
            ┌──────────┐
            │ key 存在? │
            └───┬───┬──┘
           存在  │   │ 不存在 → 穿透
                 │
                 ▼
            ┌──────────────┐
            │ 大量 key 同时 │ → 雪崩：大量 key 同时过期
            │   过期?       │   解决：过期时间加随机值
            └───┬──────┬───┘
           不是  │      │ 是
                 │      └──→ DB 被打爆
                 ▼
            ┌──────────┐
            │ 单个热点   │ → 击穿：一个热点 key 过期
            │  key 过期? │   解决：互斥锁 / 永不过期
            └──────────┘
```

### Q16：Redis 分布式锁怎么实现？

```bash
# 加锁
SET lock:order:1001 unique_uuid NX EX 30

# 释放（Lua 脚本保证原子性）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**三个关键点**：
1. **SET NX EX** — 原子加锁 + 设过期（防死锁）
2. **unique_value** — 释放时校验值（防误删别人的锁）
3. **Lua 脚本释放** — 查 + 删原子操作

> ⚠️ 单节点 Redis 锁在 master 宕机时可能丢失。对一致性要求高的用 RedLock 或 ZooKeeper。

### Q17：Redis 主从同步原理？

```
全量同步:
  Slave → PSYNC ? -1 → Master
  Master → 执行 BGSAVE 生成 RDB → 发给 Slave
  Slave 加载 RDB
  Master 把期间的写命令发给 Slave (replication buffer)

增量同步:
  Slave → PSYNC replid offset → Master
  Master 从 repl_backlog 中找 offset 之后的命令 → 发给 Slave
  
  关键: repl_backlog 是环形缓冲区，offset 太旧可能得全量同步
```

### Q18：Redis 集群模式有哪几种？

| 模式 | 特点 | 缺点 |
|------|------|------|
| **主从** | 读写分离，数据冗余 | 手动切换，不自动故障转移 |
| **哨兵** | 自动故障转移 | 不能水平扩展（所有数据全量） |
| **Cluster** ⭐ | 分片+高可用，16384 slot | 不支持跨 slot 多 key 操作 |

```
Cluster 分片:
  key → CRC16(key) % 16384 → slot → 对应节点
  每个节点负责一部分 slot
```

### Q19：Redis 的 bigkey 有什么问题？怎么找？

**问题**：
- 读写 bigkey 会阻塞 Redis（单线程！）
- 删除 bigkey 可能卡住很久（4.0+ 用 UNLINK 异步删除）
- 主从复制时传输大 RDB 会占满带宽

**排查**：
```bash
redis-cli --bigkeys       # 统计每种数据结构的最大 key
redis-cli MEMORY USAGE key  # 看具体 key 的内存占用
```

**优化**：
- String 拆成多个小 key
- Hash 按 field 拆分
- 大 List 考虑用 Stream 代替

### Q20：Redis 和 MySQL 怎么保证双写一致性？

**最终一致性方案**（推荐）：
```
写操作:
  1. 更新 MySQL
  2. 删除 Redis 缓存（不要更新缓存！）
  
读操作:
  1. 读 Redis
  2. miss → 读 MySQL → 写 Redis

延迟双删（更保险）:
  1. 先删缓存
  2. 更新 MySQL
  3. 延迟几百ms → 再删一次缓存
  目的：防止并发读写了旧数据进缓存

Canal 方案（最高级）:
  MySQL binlog → Canal → MQ → 更新/删除 Redis
  异步，最终一致性，代码无侵入
```

> **为什么是删缓存不是更新缓存？** → 更新缓存可能被并发请求覆盖；删除后由下次读请求重建，天然幂等。
