## 一、B+ 树结构全景

```
                     ┌──────┐
                     │ Root │  ← 1 次磁盘 IO
                     └──┬───┘
              ┌─────────┼─────────┐
              ▼         ▼         ▼
         ┌──────┐  ┌──────┐  ┌──────┐
         │ 内部  │  │ 内部  │  │ 内部  │  ← 2 次 IO (树高 3)
         │ 节点  │  │ 节点  │  │ 节点  │
         └──┬───┘  └──┬───┘  └──┬───┘
       ┌────┼────┐     │        │
       ▼    ▼    ▼     ▼        ▼
    ┌────┬────┬────┬────┬────┬────┐
    │ R1 │ R2 │ R3 │ R4 │ R5 │ R6 │  ← 3 次 IO (叶子层)
    │    │    │    │    │    │    │     存完整数据 + 双向链表
    └────┴────┴────┴────┴────┴────┘
    ◄──────── 双向链表, 范围扫描 O(k) ──────►

特点:
• 非叶子节点: 只存索引key + 子节点指针 → 每个节点存更多,树更矮
• 叶子节点: 存完整数据 (或主键值) + 双向链表 → 范围查询顺序读
• 树高: 千万级表也才 3-4 层 → 3-4 次 IO 定位数据
```

## 二、聚集索引 vs 非聚集索引 (回表示意图)

```
  主键索引 (聚集索引)               二级索引 (非聚集索引)
  INDEX(PK)                       INDEX(name)
  
      [50|100]                        [Alice|Bob]
       ╱    ╲                          ╱      ╲
     [1-50] [51-100]              [Alice,Cat] [Bob,Dog]
     ╱    ╲   ...                   ╱    ╲
   ┌──────┬──────┐              ┌────────┬────────┐
   │PK=1  │PK=10 │              │Alice,1 │Bob,51  │
   │name  │name  │              │Cat,7   │Dog,88  │
   │age   │age   │              └───┬────┴───┬────┘
   │...完整行│...完整行│               │        │
   └──────┴──────┘              回表!│   回表!│
                                    ▼        ▼
                              ┌──────┐  ┌──────┐
                              │PK=1  │  │PK=51 │
                              │完整行 │  │完整行 │
                              └──────┘  └──────┘

非聚集索引查不到全部列 → 用主键回聚集索引 → 二次查找
                          ↑
              覆盖索引: SELECT id,name FROM t WHERE name='Alice'
              索引中已有 id+name → 不需要回表! 只查一次!
```

## 三、事务隔离级别与问题对照

```
隔离级别                   脏读    不可重复读   幻读
─────────────────────────────────────────────────
READ UNCOMMITTED          ●         ●         ●
    ↓  ("读未提交")
    │   ┌──── 解决脏读 ────┐
    ▼   ▼                 │
READ COMMITTED            ○         ●         ●
    ↓  ("读已提交")
    │         ┌── 解决不可重复读 ──┐
    ▼         ▼                  │
REPEATABLE READ (默认)     ○         ○       ⚠️(几乎)
    ↓  ("可重复读")
    │                     ┌───── 解决幻读 ──────┐
    ▼                     ▼                    ▼
SERIALIZABLE              ○         ○         ○
  ("串行化")
─────────────────────────────────────────────────

● = 会发生  ○ = 不会发生  ⚠️ = MVCC+间隙锁几乎解决

实现差异:
  RC  → 每次 SELECT 生成新 ReadView
  RR  → 事务开始时生成 ReadView，全程复用
```

## 四、MVCC 工作流程

```
事务 T1 (id=100)                      事务 T2 (id=200)
│                                       │
│  BEGIN                                │  BEGIN
│  SELECT * FROM t WHERE id=1           │  
│  → name='Alice' (快照: trx<100)        │
│                                       │  UPDATE t SET name='Bob' WHERE id=1
│                                       │  → 旧值→undo log
│                                       │  → 行 DB_TRX_ID=200
│                                       │  COMMIT
│                                       │
│  SELECT * FROM t WHERE id=1           │
│  → 行 trx_id=200                      │
│  → ReadView.active=[100]             │
│  → 200 > max? 不在active?            │
│     RC: 生成新 ReadView, 200 不在列表中→可见→'Bob'
│     RR: 复用旧 ReadView, 200 > max→不可见
│          → 查 undo log → 找到 trx_id<100 的版本 → 'Alice'
│                                       │

行隐藏列:
┌────┬──────┬──────┬────────┬──────────────┐
│ id │ name │ age  │DB_TRX_ID│ DB_ROLL_PTR  │
├────┼──────┼──────┼────────┼──────────────┤
│ 1  │ Bob  │ 25   │  200   │ → undo log   │
└────┴──────┴──────┴────────┴──────────────┘
```

## 五、Redo Log 工作原理 (WAL)

```
更新一行数据:

  ① 修改 Buffer Pool 中的页
    ┌──────────────────┐
    │  Buffer Pool      │
    │  ┌──────────┐    │
    │  │ 数据页     │◄── 在内存中修改
    │  │ id=1→Bob  │    │
    │  └──────────┘    │
    └───────┬──────────┘
            │
  ② 同时写 Redo Log 缓冲区
    ┌────────▼─────────┐
    │ Redo Log Buffer   │ ← 顺序写入，速度极快
    │ "页X 偏移Y→Bob"   │
    └────────┬─────────┘
            │ 事务提交时 fsync
            ▼
    ┌──────────────────┐
    │ Redo Log 文件     │ ← 磁盘，持久化
    │ ib_logfile0/1    │
    └──────────────────┘
            │
  ③ 返回客户端 "提交成功"  ← 不用等数据页刷盘!
            │
  ④ 后台线程异步刷脏页 ──→ 磁盘数据文件 (.ibd)

崩溃恢复:
  重启 → 检查 Redo Log → 
  已提交事务 → 重做（apply redo）→ 数据恢复
  未提交事务 → undo log 回滚
```

## 六、Redis 缓存三大问题图解

```
缓存穿透:                        缓存击穿:
  请求 key="x"                    请求 key="hot"
    │                                │
    ▼                                ▼
  Redis: 不存在                    Redis: 已过期
    │                                │
    ▼                                ▼
  MySQL: 也不存在                  MySQL: 有数据
    │                                │
    ▼                大量并发请求同时间打过来!
  每次都穿到DB!         ┌──────┬─────┼─────┬──────┐
                       ▼      ▼     ▼     ▼      ▼
  解决:                全部查 MySQL → DB 瞬间过载!
  ① 布隆过滤器         解决:
  ② 缓存空值            ① 互斥锁 (只有第一个请求查DB)
  (短过期时间)           ② 热点key永不过期+异步更新

缓存雪崩:
  时间线: ────┬────────┬────────┬────────┬────→
         key1过期  key2过期  key3过期
         (同时/接近同时大量 key 过期)
                        │
                        ▼
              MySQL 瞬间接收海量请求
                        │
                        ▼
                    DB 过载/宕机

  解决:
  ① key 过期时间加随机值 (TTL + random(0,300s))
  ② 多级缓存 (本地缓存 + Redis)
  ③ 限流 + 降级
```

## 七、Redis 分布式锁时序

```
客户端 A                           Redis                 客户端 B
  │                                  │                     │
  │ SET lock order:1001 uuid_A NX EX 30                    │
  │ ─────────────────────────────→   │                     │
  │ OK (获取成功)                     │                     │
  │                                  │                     │
  │ 执行临界区...                     │                     │
  │                                  │                     │
  │                                  │  SET lock order:1001 uuid_B NX EX 30
  │                                  │ ◄───────────────────
  │                                  │  (nil) 获取失败
  │                                  │                     │
  │                                  │         B 等待/重试...
  │                                  │                     │
  │ eval "if get(KEYS[1])==ARGV[1]   │                     │
  │       then del(KEYS[1]) end"     │                     │
  │ ─────────────────────────────→   │                     │
  │ 释放锁                            │                     │
  │                                  │                     │
  │                                  │  SET lock order:1001 uuid_B NX EX 30
  │                                  │ ◄───────────────────
  │                                  │  OK (获取成功)
  │                                  │                     │

关键: ① SET NX EX 原子加锁+过期  ② value用唯一标识防误删  ③ Lua原子释放
```

## 八、Redis 集群分片 (16384 slot)

```
                    ┌──────────┐
                    │  Client  │
                    └────┬─────┘
                         │ CRC16(key) % 16384 = slot
                         ▼
              ┌─────────────────────┐
              │    slot → node 映射  │
              └──┬──────┬───────┬───┘
                 │      │       │
      slot       ▼      ▼       ▼
      0-5460  ┌────┐ ┌────┐ ┌────┐
              │Node│ │Node│ │Node│
      5461-   │ A  │ │ B  │ │ C  │  3 主 3 从
      10922   └──┬─┘ └──┬─┘ └──┬─┘
                 │      │      │
      10923-  ┌──┴─┐ ┌──┴─┐ ┌──┴─┐
      16383   │Slave│ │Slave│ │Slave│
              └────┘ └────┘ └────┘

  特性:
  • 共 16384 个槽 (slot)
  • 每个节点负责一段 slot 范围
  • key → CRC16 → slot → 对应节点
  • 客户端缓存 slot 映射，减少重定向
  • MOVED: 永久重定向 | ASK: 临时重定向 (slot 迁移中)
```

## 九、SQL 执行顺序 vs 书写顺序

```
书写顺序:                          实际执行顺序:
SELECT DISTINCT                    1. FROM     (确定表)
  column,                          2. ON       (JOIN 条件)
  AGG_FUNC(column)                 3. JOIN    (关联)
FROM table                         4. WHERE   (过滤)
  JOIN other_table                 5. GROUP BY (分组)
  ON join_condition                6. HAVING   (分组后过滤)
WHERE condition                    7. SELECT   (选择列)
GROUP BY column                    8. DISTINCT (去重)
HAVING condition                   9. ORDER BY (排序)
ORDER BY column                    10. LIMIT   (限制行数)
LIMIT n;
```

> 核心：**WHERE 在 GROUP BY 之前，HAVING 在 GROUP BY 之后**。WHERE 过滤行，HAVING 过滤组。

## 十、一句话速记卡片

| 概念 | 一句话 |
|------|--------|
| B+ 树 | 数据只存叶子 + 叶子有双向链表 → 快在范围查询 |
| 聚集索引 | 叶子存完整行 → 一张表只有一个 |
| 回表 | 二级索引查到主键 → 再去主键索引查完整数据 |
| 覆盖索引 | 查询列都在索引里 → 不回表 → Extra: Using index |
| 最左前缀 | 联合索引从最左列开始匹配，中间不能断 |
| MVCC | 每行有版本号 + ReadView 判断可见性 |
| redo log | 先写日志再写磁盘，崩溃恢复靠它 |
| 缓存穿透 | 查不存在的 key → 布隆/缓存空值 |
| 缓存击穿 | 热点 key 过期 → 互斥锁 |
| 缓存雪崩 | 大量 key 同时过期 → 过期时间加随机 |
| SET NX EX | Redis 分布式锁加锁原子操作 |
| CRC16 % 16384 | Redis Cluster 分片算法 |
