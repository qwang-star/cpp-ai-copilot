# 后端项目常见设计模式和本项目对应关系

这份文档用表格把常见后端设计模式和当前 C++ AI Copilot 项目对应起来。

重点不是“我用了很多模式”，而是：

```text
我知道每种模式解决什么问题。
我知道当前项目哪里已经体现了它。
我知道哪些模式是未来才需要引入的。
```

## 1. 当前已经用到的模式和思想

| 模式 / 思想 | 当前项目对应代码 | 解决的问题 | 当前状态 |
|---|---|---|---|
| 工厂方法 | `HttpResponse::json` | 创建 JSON 响应时隐藏细节 | 已使用 |
| 回调机制 | `Handler = std::function<...>` | Router 不直接依赖具体业务函数 | 已使用 |
| 路由表 | `std::map<std::string, Handler>` | 用表管理 path/method 到 Handler 的映射 | 已使用 |
| 委托 | `application.cpp -> handle_chat_request` | 路由层不写业务 | 已使用 |
| 单一职责 | `chat_service`、`api_response`、`error_code` | 每个模块只做一类事 | 已使用 |
| 封装 | `HttpRequest`、`HttpResponse`、`Router` | 隐藏 HTTP 原始文本和路由细节 | 已使用 |
| 分层架构 | application / service / response / error_code | 控制复杂度，便于迁移框架 | 已使用 |

## 2. 未来会用到的模式

| 模式 | 未来用在哪里 | 为什么需要 |
|---|---|---|
| 适配器模式 | Drogon Controller 适配现有业务层 | 框架请求对象和业务对象不一样 |
| 策略模式 | RAG 检索策略、模型供应商策略 | 多种算法/供应商可替换 |
| 责任链模式 | 鉴权、限流、日志、参数校验中间件 | 请求要经过多个处理步骤 |
| 仓储模式 Repository | MySQL 文档、用户、会话访问 | 业务层不直接写 SQL |
| 外观模式 Facade | RagService 对外隐藏检索、Rerank、Prompt 细节 | 对 Controller 暴露简单接口 |
| 观察者 / 事件模式 | 文档上传后触发解析、Embedding、入库 | 异步任务解耦 |
| 模板方法 | 统一模型调用流程：构造请求、发送、解析、记录日志 | 多模型共享流程 |

## 3. 工厂方法在项目中的位置

当前：

```cpp
HttpResponse::json(200, body)
```

它封装了：

```text
状态码
reason phrase
Content-Type
Connection
body
```

未来产品版中也会有类似函数：

```text
ApiResponse::success(data)
ApiResponse::error(code, message)
```

也就是：

```text
创建响应对象的细节，不散落在每个业务接口里。
```

## 4. 回调机制在项目中的位置

当前：

```cpp
using Handler = std::function<HttpResponse(const HttpRequest&)>;
```

这让 Router 可以保存各种处理函数。

比如：

```cpp
router.get("/health", [](const HttpRequest&) { ... });
router.post("/api/v1/chat", [](const HttpRequest& request) { ... });
```

Router 不知道业务是什么，只知道：

```text
我找到 Handler 后可以调用它。
```

这就是回调。

## 5. 委托在项目中的位置

当前：

```cpp
router.post("/api/v1/chat", [](const HttpRequest& request) {
    return handle_chat_request(request);
});
```

`application.cpp` 把 chat 请求委托给 `chat_service.cpp`。

这带来的好处是：

```text
application.cpp 保持很小
chat_service 可以单独测试
后续换 Drogon 时业务逻辑更容易复用
```

## 6. 适配器模式未来怎么出现

toy 版现在的业务函数是：

```cpp
HttpResponse handle_chat_request(const HttpRequest& request);
```

Drogon 里可能是：

```cpp
void asyncHandleHttpRequest(
    const drogon::HttpRequestPtr& request,
    std::function<void(const drogon::HttpResponsePtr&)>&& callback);
```

两个接口不一样。

未来可以写：

```text
DrogonChatController
  -> 从 drogon::HttpRequestPtr 取 body
  -> 转成业务层需要的数据
  -> 调用 ChatService
  -> 把业务响应转成 drogon::HttpResponsePtr
```

这就是适配器模式：

```text
用一层转换连接两个不兼容的接口。
```

## 7. 策略模式未来怎么出现

RAG 查询时可能有多种方案：

```text
方案 A：只用向量检索
方案 B：关键词 + 向量混合检索
方案 C：召回后再 Rerank
```

如果写成一堆 `if/else`：

```cpp
if (mode == "vector") ...
else if (mode == "hybrid") ...
else if (mode == "rerank") ...
```

后面会乱。

更好的方式是：

```text
RetrieverStrategy
  VectorRetriever
  HybridRetriever
  RerankRetriever
```

业务层只依赖统一接口：

```text
retrieve(query)
```

这就是策略模式。

## 8. 责任链未来怎么出现

真实后端请求通常不是直接进业务逻辑。

它会经过：

```text
请求日志
鉴权
权限校验
限流
参数校验
业务处理
响应日志
```

这些步骤可以形成责任链：

```text
Request
  -> LoggingFilter
  -> AuthFilter
  -> RateLimitFilter
  -> ValidateFilter
  -> Controller
```

Drogon 的 Filter / Middleware 思路就接近这个。

## 9. Repository 模式未来怎么出现

现在还没有数据库。

以后接 MySQL 后，不应该在 `ChatService` 里直接写 SQL：

```cpp
// 不推荐
db.execSqlSync("select * from conversations ...");
```

更好的做法：

```text
ConversationRepository
DocumentRepository
UserRepository
```

业务层调用：

```cpp
conversation_repository.save_message(...);
```

这样业务层关心业务，不关心 SQL 细节。

## 10. 面试回答模板

如果面试官问：

```text
你这个项目里用了哪些设计模式？
```

可以这样回答：

```text
我没有为了套模式而套模式，而是在项目演进中自然用了几个设计思想。

第一，HttpResponse::json 是工厂方法，封装 JSON 响应对象的创建。

第二，Router 使用 std::function 作为 Handler，本质是回调机制；Router 只负责 method + path 到 Handler 的分发，不关心具体业务。

第三，application.cpp 把 /api/v1/chat 委托给 chat_service，实现了路由层和业务层分离。

第四，我把响应格式和错误码拆成 api_response 和 error_code，体现了单一职责和分层架构。

后续迁移 Drogon 时，会自然引入适配器模式；做 RAG 多检索策略时，会引入策略模式；接 MySQL 后会用 Repository 隔离 SQL 和业务逻辑。
```

这比单纯背诵“工厂、单例、策略”更像真实项目经验。
