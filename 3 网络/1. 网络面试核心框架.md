## 二、TCP vs UDP

### 2.1 核心对比（必背）

| 维度 | TCP | UDP |
|------|-----|-----|
| **连接** | 面向连接（三次握手） | 无连接 |
| **可靠性** | 可靠（确认+重传+序号） | 不可靠（发完不管） |
| **顺序** | 有序（序号保证） | 无序 |
| **速度** | 较慢 | 快 |
| **头部开销** | 20 字节 | 8 字节 |
| **流量/拥塞控制** | ✅ 有 | ❌ 无 |
| **适用场景** | HTTP、文件传输、邮件 | 视频直播、DNS、游戏 |
| **比喻** | 打电话（先接通再说话） | 寄明信片（投了就完） |

### 2.2 TCP 报文头部结构

```
 0      4      8      12     16     20     24     28     32
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│      源端口 (2B)       │     目的端口 (2B)              │
├───────────────────────┼───────────────────────────────┤
│                    序列号 Seq (4B)                     │
├───────────────────────────────────────────────────────┤
│                   确认号 Ack (4B)                      │
├──────┬───────────────┬───────────────────────────────┤
│数据偏移│   保留+标志位  │           窗口大小             │
│ (4b)  │   (6b+6b)    │           (2B)                │
├──────┴───────────────┼───────────────────────────────┤
│   校验和 (2B)         │        紧急指针 (2B)           │
└──────────────────────┴───────────────────────────────┘

标志位: URG | ACK | PSH | RST | SYN | FIN
```

## 三、三次握手 ⭐⭐⭐

### 3.1 完整时序

```
  客户端 (主动打开)              服务端 (被动打开)
       │                              │
       │       ① SYN, Seq=x           │  CLOSED → LISTEN
       │ ──────────────────────────→  │
   SYN_SENT                          │  LISTEN → SYN_RCVD
       │                              │
       │    ② SYN+ACK, Seq=y, Ack=x+1 │
       │ ◁──────────────────────────  │
  ESTABLISHED                        │
       │                              │
       │       ③ ACK, Seq=x+1, Ack=y+1│
       │ ──────────────────────────→  │
  ESTABLISHED                        │  SYN_RCVD → ESTABLISHED
       │                              │
```

### 3.2 为什么是三次，不是两次？

**核心原因**：防止**旧的重复 SYN 连接请求**到达服务端导致建立无效连接。

```
场景：客户端发了 SYN(seq=90)，网络延迟没到。
      客户端超时重发 SYN(seq=100)，连接建立完成后关闭。
      此时旧的 SYN(seq=90) 到达服务端。
      
  两次握手：服务端直接回 SYN+ACK 进入 ESTABLISHED
           → 客户端不理（认为自己没发请求）
           → 服务端空等，浪费资源！
  
  三次握手：服务端回 SYN+ACK 后等待客户端的 ACK
           → 客户端发现不对，回 RST
           → 服务端关闭，不会浪费资源
```

> 🔥 面试还有另一个角度：**TCP 是全双工的**，需要双方都确认自己能发能收。两次握手只能确认一方。

## 四、四次挥手 ⭐⭐⭐

### 4.1 完整时序

```
  客户端 (主动关闭)              服务端 (被动关闭)
       │                              │
       │     ① FIN, Seq=u, Ack=v      │  ESTABLISHED
  FIN_WAIT_1                          │
       │ ──────────────────────────→  │  → CLOSE_WAIT
       │                              │
       │     ② ACK, Seq=v, Ack=u+1    │
  FIN_WAIT_2                          │
       │ ◁──────────────────────────  │  (可能还有数据要发)
       │                              │
       │     ③ FIN, Seq=w, Ack=u+1    │
       │ ◁──────────────────────────  │  → LAST_ACK
  TIME_WAIT                           │
       │                              │
       │     ④ ACK, Seq=u+1, Ack=w+1  │
       │ ──────────────────────────→  │
  (等待 2MSL)                         │  → CLOSED
       │                              │
```

### 4.2 TIME_WAIT 为什么等 2MSL？

1. **确保最后一个 ACK 能到达服务端** → 如果丢失，服务端会重发 FIN，客户端可以重发 ACK
2. **让旧连接的所有报文在网络中消失** → 防止影响新连接（2MSL ≈ 2 分钟）

> MSL = Maximum Segment Lifetime，报文最大生存时间（通常 30s-2min）。

### 4.3 为什么挥手需要四次？

TCP 是全双工的，**一方关闭只表示自己不再发了**，另一方可能还有数据要发。

- FIN → ACK：客户端说"我不发了"
- 服务端可能还在发剩余数据
- FIN → ACK：服务端说"我也不发了"

> 所以中间两个 ACK 和 FIN 是分开的 → 四步。有时候服务端没有数据要发，②③ 可以合并（三次挥手）。

### 4.4 TCP 状态转换图（精简版）

```
                    ┌──────────┐
                    │  CLOSED  │
                    └────┬─────┘
             被动打开    │    主动打开
         ┌──────────────┼──────────────┐
         ▼              │              ▼
     ┌──────┐           │          ┌──────────┐
     │LISTEN│           │          │ SYN_SENT │
     └──┬───┘           │          └────┬─────┘
        │ SYN_RCVD◄─────┼──────────────┘
        │    │           │
        └────┼───────────┘
             ▼
     ┌─────────────┐
     │ ESTABLISHED │
     └──┬───────┬──┘
      主动关闭  │  被动关闭
        ▼      │      ▼
   FIN_WAIT_1  │  CLOSE_WAIT
        │      │      │
   FIN_WAIT_2  │      ▼
        │      │  LAST_ACK
        ▼      │      │
    TIME_WAIT  │      ▼
        │      │   CLOSED
        └──────┘
      (2MSL后)
```

## 五、TCP 可靠传输机制

### 5.1 核心机制

| 机制 | 作用 |
|------|------|
| **序列号 + 确认应答** | 每个字节编号，接收方确认收到的 |
| **超时重传** | 超过 RTO 未收到 ACK → 重发 |
| **快速重传** | 收到 3 个重复 ACK → 立即重发（不等超时） |
| **滑动窗口** | 流量控制，控制发送速度 |
| **拥塞控制** | 控制网络负载，避免网络拥塞 |

### 5.2 超时重传 vs 快速重传

```
超时重传:
  发送方 --> 等 RTO 超时 --> 没收到 ACK --> 重发
  缺点: 慢！RTO 至少几百毫秒

快速重传 (Fast Retransmit):
  发送方收到 3 个重复 ACK (Dup ACK) --> 立即重发缺失的包
  不需要等超时！
```

### 5.3 滑动窗口

```
发送方窗口:
  [已发送已确认| 已发送未确认 | 可发送未发送 | 不可发送 ]
               ◄── 发送窗口 ──►

接收方窗口:
  [已接收已确认| 可接收未接收 | 不可接收 ]
               ◄── 接收窗口 ──►

窗口大小 = min(接收方通告窗口, 拥塞窗口)
```

## 六、拥塞控制 ⭐⭐

### 6.1 四大算法

```
            ┌── 慢启动 (SS) ──┐
            │  cwnd 指数增长   │
            │  (1→2→4→8...)   │
            └────────┬────────┘
                     │ 达到 ssthresh
                     ▼
            ┌── 拥塞避免 (CA) ──┐
            │  cwnd 线性增长   │
            │  (+1 每 RTT)     │
            └──┬──────────┬────┘
               │          │
         3 Dup ACK    超时(RTO)
               │          │
               ▼          ▼
        ┌── 快速重传 ──┐  ┌── 超时重传 ──┐
        │ ssthresh=cwnd/2│  │ ssthresh=cwnd/2│
        │ cwnd=ssthresh │  │ cwnd=1 (回慢启动)│
        └───────┬───────┘  └───────────────┘
                ▼
        ┌── 快速恢复 ──┐
        │ cwnd 从 ssthresh│
        │ 线性增长       │
        └───────────────┘
```

> cwnd = 拥塞窗口，ssthresh = 慢启动阈值

### 6.2 一句话总结

| 算法 | 触发 | 动作 |
|------|------|------|
| **慢启动** | 连接建立 / 超时后 | cwnd 从 1 开始指数增长 |
| **拥塞避免** | cwnd ≥ ssthresh | cwnd 线性增长 |
| **快速重传** | 收到 3 个重复 ACK | 立即重传，不等超时 |
| **快速恢复** | 快速重传后 | cwnd 减半，进入拥塞避免 |

## 七、HTTP ⭐⭐

### 7.1 HTTP 版本演进

| 版本 | 连接 | 特点 |
|------|------|------|
| **HTTP/1.0** | 短连接 | 每次请求都要新开 TCP 连接 |
| **HTTP/1.1** | 持久连接 + 管道化 | Keep-Alive + 可连续发多个请求（但响应要按顺序） |
| **HTTP/2.0** | 多路复用 | 一个连接并发多个请求/响应，二进制分帧，头部压缩 |
| **HTTP/3.0** | QUIC (UDP) | 0-RTT 握手，解决 TCP 队头阻塞 |

### 7.2 HTTP/1.1 队头阻塞 (Head-of-Line Blocking)

```
HTTP/1.1 管道化:
  请求1 ──等待响应1──→
  请求2 ──必须等响应1──→  (请求2明明可以更快返回)
  请求3 ──必须等响应1,2──→

HTTP/2 多路复用:
  请求1 ──┐
  请求2 ──┤ 并发 ! 响应可以乱序
  请求3 ──┘
```

### 7.3 HTTP 常见状态码

| 状态码 | 含义 | 场景 |
|--------|------|------|
| 200 OK | 成功 | GET 请求成功 |
| 201 Created | 创建成功 | POST 创建资源 |
| 301 | 永久重定向 | 旧域名→新域名 |
| 302 | 临时重定向 | 登录后跳转 |
| 304 | 未修改 | 缓存有效 |
| 400 | 请求错误 | 参数格式不对 |
| 401 | 未认证 | 没登录 |
| 403 | 禁止访问 | 没权限 |
| 404 | 未找到 | URL 不对 |
| 500 | 服务器错误 | 代码崩了 |
| 502 | 网关错误 | 代理转发失败 |
| 503 | 服务不可用 | 过载/维护 |

### 7.4 GET vs POST

| | GET | POST |
|------|-----|------|
| 语义 | 获取资源 | 提交/创建资源 |
| 参数位置 | URL 查询字符串 | 请求体 |
| 长度限制 | URL 长度限制（浏览器约 2KB） | 无限制 |
| 缓存 | 可缓存 | 默认不缓存 |
| 幂等性 | ✅ 幂等 | ❌ 不幂等 |
| 安全性 | 参数暴露在 URL | 参数在请求体中 |

> **幂等**：多次执行同样的请求，结果相同。GET/PUT/DELETE 幂等，POST 不幂等。

## 八、HTTPS ⭐

### 8.1 HTTPS = HTTP + TLS/SSL

```
HTTP (明文)              HTTPS (加密)
  80 端口                  443 端口
  ┌──────┐               ┌──────┐
  │ HTTP │               │ HTTP │
  ├──────┤               ├──────┤
  │ TCP  │               │ TLS  │ ← 加密层
  ├──────┤               ├──────┤
  │  IP  │               │ TCP  │
  └──────┘               ├──────┤
                          │  IP  │
                          └──────┘
```

### 8.2 TLS 1.2 握手（简化版）

```
  客户端                          服务端
    │                                │
    │ ① ClientHello                  │
    │  (支持的加密套件 + 随机数R1)     │
    │ ─────────────────────────────→ │
    │                                │
    │ ② ServerHello + 证书 + 公钥    │
    │  (选择加密套件 + 随机数R2)       │
    │ ◁───────────────────────────── │
    │                                │
    │ ③ 验证证书 + 生成Pre-Master    │
    │   用公钥加密 Pre-Master         │
    │   生成会话密钥                  │
    │ ─────────────────────────────→ │
    │                                │
    │ ④ 私钥解密 Pre-Master          │
    │   生成同样的会话密钥             │
    │   握手完成，对称加密通信         │
    │ ◀═════ 对称加密通信 ═════════▶ │
```

> TLS 1.3 简化为 1-RTT（甚至 0-RTT），握手更快。

### 8.3 HTTPS 为什么安全？

| 威胁 | 防护 |
|------|------|
| 窃听 | **加密**（对称加密通信内容） |
| 篡改 | **完整性校验**（MAC/哈希） |
| 冒充 | **证书 + CA**（验证身份） |

## 九、DNS 域名解析

```
浏览器输入 www.example.com
    │
    ▼ ① 浏览器缓存
    │
    ▼ ② OS 缓存 (hosts文件)
    │
    ▼ ③ 本地 DNS 服务器 (递归查询)
    │
    ▼ ④ 根域名服务器 → 返回 .com 的 DNS
    │
    ▼ ⑤ .com 顶级域名服务器 → 返回 example.com 的 DNS
    │
    ▼ ⑥ example.com 权威 DNS → 返回 IP 地址
    │
    ▼ ⑦ 浏览器用 IP 发起 HTTP 请求
```

> DNS 默认走 UDP 端口 53。区域传送时走 TCP。
