# 网络高频面试题精讲

> 网络面试题高度模式化，答案标准。每题控制在 2-3 分钟。

---

## TCP 核心

### Q1：TCP 三次握手的过程？为什么需要三次？

**过程**：
```
C → S: SYN, Seq=x                    (我想和你连接)
S → C: SYN+ACK, Seq=y, Ack=x+1       (好的，我也想和你连接)
C → S: ACK, Seq=x+1, Ack=y+1         (确认，连接建立)
```

**为什么三次**：
1. **防止旧重复连接**：如果有旧的 SYN 延迟到达服务端，两次握手下服务端会直接 ESTABLISHED，浪费资源。第三次握手让客户端有机会拒绝。
2. **全双工确认**：TCP 是双向的，需要双方都确认自己能发能收。两次握手只能确认客户端→服务端单向。

---

### Q2：四次挥手的过程？TIME_WAIT 为什么等 2MSL？

**过程**：
```
C → S: FIN, Seq=u             (我不发了)
S → C: ACK, Seq=v, Ack=u+1    (知道了)
S → C: FIN, Seq=w, Ack=u+1    (我也不发了)
C → S: ACK, Seq=u+1, Ack=w+1  (知道，拜拜)
C 进入 TIME_WAIT 等 2MSL
```

**TIME_WAIT 等 2MSL 的原因**：
1. 确保最后一个 ACK 到达服务端（如果丢了，服务端重发 FIN，客户端还能回 ACK）
2. 让旧连接的残留报文在网络中消失，不影响新连接

> MSL ≈ 30s-60s，2MSL ≈ 1-2 分钟。

---

### Q3：为什么挥手需要四次，握手只要三次？

TCP 是全双工。握手时 SYN+ACK 可以合并（一次发两个标志位）。但挥手时，**一方 FIN 表示"我不发了"，另一方可能还有数据要发**，所以 ACK 和 FIN 不能合并 → 需要四次。

---

### Q4：TCP 怎么保证可靠传输？

| 机制 | 说明 |
|------|------|
| **序列号 + 确认** | 每个字节有编号，接收方确认 |
| **超时重传** | 超过 RTO 没收到 ACK → 重发 |
| **快速重传** | 收到 3 个重复 ACK → 立即重发（不等超时） |
| **滑动窗口** | 流量控制，接收方告知自己能收多少 |
| **拥塞控制** | 四算法控制发送速率 |
| **校验和** | 检测数据是否损坏 |

---

### Q5：TCP 拥塞控制的四个算法是什么？

1. **慢启动**：cwnd 从 1 MSS 开始，每 RTT 翻倍（指数增长），直到 ≥ ssthresh
2. **拥塞避免**：cwnd 每 RTT 加 1（线性增长）
3. **快速重传**：收到 3 个重复 ACK → 立即重传丢失包
4. **快速恢复**：快速重传后，ssthresh = cwnd/2，cwnd = ssthresh，进入拥塞避免

### Q6：TCP 粘包问题是什么？怎么解决？

**现象**：TCP 是字节流协议，发送方多次发送的数据在接收方可能粘在一起。

**原因**：TCP 不保留消息边界，只是字节流。

**解决**：
- **定长消息**
- **分隔符**（如 `\r\n`，类似 HTTP）
- **消息头 + 消息体**（头部声明长度）
- 使用更高层协议（HTTP、Protobuf）

> UDP 不会有粘包问题，因为 UDP 以数据报为单位，有天然消息边界。

---

### Q7：TCP Keep-Alive 是什么？

TCP 内置的心跳机制。连接空闲一定时间后，发送探测包确认对端是否还活着。

```
默认配置 (Linux):
  tcp_keepalive_time   = 7200s (2小时空闲后开始探测)
  tcp_keepalive_intvl  = 75s   (探测间隔)
  tcp_keepalive_probes = 9     (探测次数)
```

> 实际应用层通常会自己实现心跳（如 WebSocket ping/pong），因为 TCP Keep-Alive 时间太长了。

## UDP

### Q8：UDP 的应用场景有哪些？

| 场景 | 为什么用 UDP |
|------|-------------|
| DNS | 单次查询，包小，TCP 握手开销太大 |
| 音视频直播 | 允许少量丢包，不允许卡顿 |
| 在线游戏 | 低延迟 > 可靠性 |
| DHCP | 广播发现服务器 |
| QUIC (HTTP/3) | 在 UDP 上实现可靠传输 |

## HTTP

### Q9：HTTP/1.1 相比 1.0 有哪些改进？

| 特性 | 1.0 | 1.1 |
|------|-----|-----|
| 连接 | 短连接（每次请求新建 TCP） | **持久连接** (Keep-Alive) |
| Host 头 | ❌ | ✅ （一台 IP 多个域名） |
| 管道化 | ❌ | ✅ （但响应必须有序，队头阻塞） |
| 缓存 | Expires | **Cache-Control** + ETag |
| 断点续传 | ❌ | ✅ Range 请求 |
| 分块传输 | ❌ | ✅ Chunked |

---

### Q10：HTTP/2 相比 1.1 有哪些改进？

1. **多路复用**：一个 TCP 连接同时发送多个请求/响应（解决队头阻塞）
2. **头部压缩 (HPACK)**：相同头部字段只发一次
3. **二进制分帧**：不再用文本，用二进制帧传输
4. **服务端推送 (Server Push)**：服务端主动推送客户端需要的资源
5. **流优先级**：指定资源加载优先级

---

### Q11：HTTP/3 为什么用 QUIC (UDP) 而不用 TCP？

1. **0-RTT 握手**：QUIC 基于 UDP，握手极快（甚至 0-RTT）
2. **解决 TCP 队头阻塞**：TCP 层面丢包会阻塞所有 HTTP/2 流；QUIC 各流独立
3. **连接迁移**：切换网络（WiFi→4G）不需要重建连接（基于 Connection ID）
4. **内置加密**：TLS 1.3 内置于 QUIC

---

### Q12：HTTP 和 HTTPS 的区别？

| | HTTP | HTTPS |
|------|------|-------|
| 端口 | 80 | 443 |
| 加密 | 明文 | TLS/SSL 加密 |
| 证书 | 不需要 | 需要 CA 证书 |
| 速度 | 快 | 慢（多一次 TLS 握手） |
| SEO | 普通 | 搜索引擎优先 |

---

### Q13：HTTPS 握手过程中，对称加密的密钥是怎么安全交换的？

**非对称加密交换对称密钥**：
1. 服务端发送证书（含公钥）
2. 客户端生成 Pre-Master Secret，用**公钥加密**后发送
3. 只有服务端有**私钥**能解密
4. 双方用 Pre-Master Secret + 之前的随机数生成**会话密钥**（对称密钥）
5. 之后用会话密钥**对称加密**通信

> **为什么不用非对称加密一直通信？** → 太慢了。非对称加解密比对称加解密慢 100-1000 倍。

### Q14：状态码 301 和 302 的区别？

- **301 Moved Permanently**：永久重定向。浏览器会缓存，下次自动跳新地址。搜索引擎会更新收录。
- **302 Found**：临时重定向。浏览器不缓存，每次请求都先访问原地址。搜索引擎保留原地址。

> 登录后用 302 跳回，改用 HTTPS 后用 301。

### Q15：PUT 和 PATCH 的区别？

| | PUT | PATCH |
|------|-----|-------|
| 语义 | 完整替换 | 部分更新 |
| 幂等性 | ✅ 幂等 | ❌ 不幂等 |

```
PUT /users/1  {name:"张三", age:20, email:"..."}  → 全部替换
PATCH /users/1 {age:21}  → 只更新 age 字段
```

## DNS

### Q16：浏览器输入 URL 后发生了什么？（经典连环题 ⭐⭐⭐）

```
1. URL 解析 → 提取域名
2. DNS 解析 → 域名 → IP
   (浏览器缓存 → OS hosts → 本地DNS → 根DNS → ...)
3. TCP 三次握手 → 建立连接
4. TLS 握手（如果是 HTTPS）→ 加密连接
5. 发送 HTTP 请求
6. 服务端处理 → 返回 HTTP 响应
7. 浏览器解析 HTML → 渲染页面
8. TCP 四次挥手 → 关闭连接
```

> 这道题能考察面试者对整个网络栈的理解，是经典的综合题。

---

### Q17：DNS 走 TCP 还是 UDP？

**默认 UDP**（端口 53），因为单次查询包小（< 512 字节），TCP 握手开销太大。

**以下情况走 TCP**：
- 响应超过 512 字节（UDP 包大小限制）
- DNS 区域传送（AXFR）

## 综合

### Q18：Cookie、Session、Token 的区别？

```
Cookie: 浏览器里的小数据块，自动携带
  - 存储在浏览器，每次请求自动发送
  - 大小限制 4KB
  - 受同源策略限制

Session: 服务端存储的用户状态
  - Session ID 存 Cookie 中
  - 服务端查找对应用户数据
  - 问题：分布式下需要 Session 共享（Redis）

JWT Token: 自包含的令牌
  - 无状态，不需服务端存储
  - 包含用户信息 + 签名
  - 适合分布式/微服务
  - 问题：无法主动失效（发了就有效直到过期）
```

### Q19：正向代理和反向代理的区别？

| | 正向代理 | 反向代理 |
|------|---------|---------|
| 代理谁 | 代理**客户端** | 代理**服务端** |
| 隐藏谁 | 隐藏客户端 IP | 隐藏服务端 IP |
| 用途 | 翻墙、访问内网 | 负载均衡、缓存、安全 |
| 例子 | VPN、Shadowsocks | **Nginx**、HAProxy |

```
正向代理:  客户端 → 代理 → Google
反向代理:  用户 → Nginx → 多个后端服务
```

### Q20：SYN 泛洪攻击是什么？怎么防御？

**攻击原理**：攻击者发送大量 SYN 但不完成握手，服务端为每个 SYN 分配资源进入 SYN_RCVD 状态，资源耗尽 → 拒绝服务。

**防御**：
- **SYN Cookie**：不立即分配资源，用算法在 SYN+ACK 中编码信息，只有收到第三次 ACK 才分配
- 缩短 SYN 超时时间
- 增加最大半连接数
- 防火墙限流
