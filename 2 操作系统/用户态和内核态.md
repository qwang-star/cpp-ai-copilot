这个过程进入服务器之后要进行区分的事情：

用户态：你的服务器程序自己能干的事  
内核态：必须让操作系统帮忙干的事
# 1. 为什么要分用户态和内核态？

因为有些东西太危险，不能让普通程序随便碰。

比如：

```
网卡磁盘物理内存文件系统进程调度TCP/IP 协议栈
```

如果你的程序可以随便操作这些东西，那一个 bug 就可能把整个系统搞崩。

所以操作系统规定：

```
普通应用程序运行在用户态操作系统内核运行在内核态
```

用户态权限低，内核态权限高。

你可以把它想成：

```
用户态 = 普通员工内核态 = 系统管理员
```

普通员工不能直接进机房拔网线、改磁盘、操作内存。

他只能提交申请：

```
请帮我读一下 socket请帮我打开这个文件请帮我把这些数据发到网络
```

这个“提交申请”的动作，就是：

```
系统调用 syscall
```

你上传的 OS 框架里也提到，用户态程序通过 `syscall` 指令进入内核态，然后由内核执行对应函数，再返回用户态。

# 2. 放到你的 Web 服务器例子里

你写的服务器程序大概是：

```cpp
int main() {    start_server();}
```

这个程序本身运行在：

```
用户态
```

它可以做一些普通计算，比如：

```
解析字符串
判断 URL 是不是 /a.jpg
维护用户连接状态
拼 HTTP 响应头
调用业务逻辑
```

这些都可以在用户态完成。

但是只要它要做这些事：

```
接受用户连接
读取用户请求
打开 a.jpg 文件
读取 a.jpg 内容
把 a.jpg 发回浏览器
```

就必须找操作系统内核帮忙。

因为这些涉及：

```
网络
文件系统
磁盘
网卡
内核缓冲区
```

# 3. 用户态和内核态到底在哪里切换？

比如服务器想读用户请求：

```cpp
read(conn_fd, buf, sizeof(buf));
```

表面上看这是一个 C 函数。

但它背后发生的是：

```
服务器程序在用户态
  ↓
调用 read()
  ↓
触发 syscall
  ↓
CPU 从用户态切换到内核态
  ↓
内核检查这个 socket 有没有数据
  ↓
把数据从内核 socket 缓冲区拷贝到用户态 buf
  ↓
返回用户态
  ↓
服务器程序继续执行
```

所以：

```
read / write / open / accept / sendfile
```

这些都不是普通函数，它们都会让程序进入内核态。

---

# 4. 用户态和内核态的核心区别

你先记这个表：

| 角度        | 用户态            | 内核态                |
| --------- | -------------- | ------------------ |
| 谁在运行      | 你的服务器程序        | 操作系统内核             |
| 权限        | 低              | 高                  |
| 能不能直接操作硬件 | 不能             | 能                  |
| 能不能访问任意内存 | 不能             | 能                  |
| 主要做什么     | 业务逻辑、解析请求、管理状态 | 网络、磁盘、内存、文件系统、调度   |
| 怎么进入      | 普通代码执行         | 通过系统调用 / 中断 / 异常进入 |

一句话：

> **用户态负责业务，内核态负责资源。**
# 5. 服务器怎么把 a.jpg 发回浏览器？

这里有两种方式。

## 方式一：普通 read + write

你可以写：

```cpp
int file_fd = open("/var/www/html/a.jpg", O_RDONLY);
char buf[4096];
while (true) {
    int n = read(file_fd, buf, sizeof(buf));
    if (n <= 0) break;
    write(conn_fd, buf, n);
}
```

这里发生了什么？
## 第一步：open 文件

```cpp
open("/var/www/html/a.jpg", O_RDONLY);
```

进入内核态：

```
用户态服务器程序
  ↓ 
open syscall内核态
  ↓
文件系统查找 /var/www/html/a.jpg
  ↓
返回 file_fd用户态
```

`file_fd` 是操作系统给这个文件的编号。
## 第二步：read 文件

```cpp
read(file_fd, buf, sizeof(buf));
```

进入内核态：

```
用户态
  ↓ 
read syscall内核态
  ↓
内核看 Page Cache 里有没有 a.jpg 数据
  ├── 有：直接从内存拿
  └── 没有：从服务器磁盘读进 Page Cache
  ↓
把数据拷贝到用户态 buf  ↓返回用户态
```

这里注意：

```
Page Cache 在内核管理的内存里
buf 在你的服务器程序用户态内存里
```

所以普通 `read` 会有一次：

```
内核态 → 用户态
```

的数据拷贝。
## 第三步：write 到 socket

```cpp
write(conn_fd, buf, n);
```

再次进入内核态：

```
用户态
  ↓ 
write syscall内核态
  ↓
把用户态 buf 拷贝到内核 socket 缓冲区
  ↓
TCP/IP 协议栈处理
  ↓
网卡驱动通知网卡发送
  ↓
返回用户态
```

这里又有一次：

```
用户态 → 内核态
```

的数据拷贝。

# 6. 普通 read + write 的完整路径

你可以这样记：

```
a.jpg 在服务器磁盘
  ↓
内核把它读到 Page Cache
  ↓
read 把数据从内核态拷贝到用户态 buf
  ↓
write 把数据从用户态 buf 拷贝回内核 socket 缓冲区
  ↓
内核 TCP/IP 协议栈处理
  ↓
网卡发送给浏览器
```

画成数据流：

```
服务器磁盘
  ↓
内核 Page Cache
  ↓ 
read 拷贝用户态 buf
  ↓ 
write 拷贝内核 Socket Buffer
  ↓
网卡
  ↓
浏览器
```

这里最关键的是：

> 普通方式里，文件数据先从内核态拷贝到用户态，又从用户态拷贝回内核态。

这就是为什么它不够高效。

因为服务器程序其实没有必要真正“看一眼”图片内容。

它只是想把 `a.jpg` 原封不动发给浏览器。
# 7. 高性能服务器怎么做？sendfile

高性能 Web 服务器常用：

```
sendfile(conn_fd, file_fd, &offset, count);
```

它的意思是：

> 内核，你直接把这个文件发到这个 socket，不用把文件内容搬到我的用户态 buffer 里。

于是路径变成：

```
服务器磁盘  ↓内核 Page Cache  ↓内核 Socket Buffer  ↓网卡  ↓浏览器
```

也就是：

```
文件数据尽量留在内核态不再经过用户态 buf
```

这就是**零拷贝 Zero-Copy**相关思想。

注意，“零拷贝”不是一点拷贝都没有，而是：

> **减少用户态和内核态之间没必要的数据拷贝。**

# 8. read + write 和 sendfile 对比

## 普通方式

```
磁盘
  ↓
内核 Page Cache
  ↓ 
拷贝到用户态用户态 buf
  ↓ 
拷贝回内核态内核 Socket Buffer
  ↓
网卡
```

问题：

```
数据绕了一圈用户态
```

但是你的服务器并不需要修改图片内容。

---

## sendfile 方式

```
磁盘
  ↓
内核 Page Cache
  ↓
内核 Socket Buffer
  ↓
网卡
```

优点：

```
文件数据不需要进入用户态减少拷贝减少 CPU 开销提高吞吐量
```

所以像 Nginx 这种发静态文件的服务器，就很适合用 `sendfile`。

# 9. 用户态 / 内核态在这条链路里的位置

你可以把整个请求分成两层：

```
用户态：服务器程序的业务逻辑
内核态：操作系统帮你处理网络、文件、内存、磁盘、网卡
```

完整流程：

```
浏览器发请求
  ↓
网卡收到数据
  ↓
【内核态】网络协议栈处理 TCP 数据
  ↓
【内核态】数据放入 socket 接收缓冲区
  ↓
【用户态】epoll_wait 返回，服务器知道 fd 可读
  ↓
【用户态】服务器调用 read
  ↓
【内核态】内核把请求数据拷贝给用户态 buffer
  ↓
【用户态】服务器解析 HTTP 请求，发现要 /a.jpg
  ↓
【用户态】服务器调用 open/read 或 sendfile
  ↓
【内核态】内核查文件系统 / Page Cache / 磁盘
  ↓
【内核态】内核把文件数据送到 socket 发送缓冲区
  ↓
【内核态】TCP/IP 协议栈封装数据
  ↓
【内核态】网卡驱动让网卡发送
  ↓
浏览器收到图片
```

你会发现：

> 真正碰网络、磁盘、文件系统、网卡的地方，基本都在内核态。  
> 你的服务器程序主要负责决定“我要干什么”。

# 10. 用一个生活例子理解

你可以把服务器程序想成餐厅服务员。

用户说：

```
我要一张 a.jpg
```

服务员，也就是用户态程序，能做的是：

```
听懂用户要什么
判断菜单上有没有
决定去仓库拿
决定送回给谁
```

但是服务员不能自己进仓库、不能自己开物流车。

仓库管理员和物流系统就是内核态。

普通 read + write 像这样：

```
服务员：我要 a.jpg
仓库管理员：给你
服务员：我再交给物流发出去
```

sendfile 像这样：

```
服务员：仓库管理员，你直接把 a.jpg 交给物流发给用户
```

所以 sendfile 更快，因为中间少倒手。

# 11. 你现在应该这样理解用户态和内核态

你的原理解：

> 前十步是用户态，把用户需求读取进来，然后缓存里有没有需要的数据，没有需要去磁盘里读取，找到之后需要把数据发回给浏览器，怎么发回，涉及到用户态和内核态。

修正成：

> **服务器程序的业务逻辑运行在用户态，比如解析请求、判断用户要 `/a.jpg`。但读取 socket、访问文件、从磁盘加载数据、通过网卡发送数据，都必须通过系统调用进入内核态。普通 read/write 会让数据在内核态和用户态之间来回拷贝；sendfile 可以让文件数据主要停留在内核态，直接从 Page Cache 发送到 socket，减少拷贝。**

# 12. 最适合你现在记忆的版本

你就背这段：

```
用户态是我的服务器程序运行的地方，负责业务逻辑；
内核态是操作系统运行的地方，负责网络、磁盘、文件系统、内存和网卡。

服务器想读用户请求，要通过 read 进入内核态；
服务器想打开 a.jpg，要通过 open 进入内核态；
服务器想把 a.jpg 发回用户，要通过 write 或 sendfile 进入内核态。

普通 read/write 会让数据从内核拷到用户态，再从用户态拷回内核；
sendfile 让数据尽量留在内核里，直接从文件缓存发到 socket，所以更适合高性能 Web 服务器。
```

核心一句话：

> **用户态决定“我要做什么”，内核态负责“真正操作系统资源去完成它”。**
