先把“我要写一个高性能 Web 服务器”这件事讲成一个新手能听懂的故事。

你现在要实现的需求是：

> 用户访问 `http://server.com/a.jpg`，服务器从磁盘读取图片，然后通过网络发给用户。

这里的“磁盘”先明确一下：

**是服务器的磁盘，不是用户电脑的磁盘。**

更准确地说，是：

> `server.com` 这台服务器机器上，操作系统能访问到的文件系统里的 `a.jpg`。

它可能是服务器本地 SSD，也可能背后挂了云盘、网络盘，但在操作系统八股里，我们先把它理解成：

```text
服务器本机磁盘 / 文件系统里的 a.jpg
```

---

# 一、先把整个故事讲顺

你不要一开始想“进程线程、epoll、TLB、零拷贝”这些词。

先想：

> 我是程序员，我要写一个服务器，它要接客、分工、找文件、发文件。

整个流程可以先粗暴理解成：

```text
写程序
  ↓
启动程序，OS 创建进程
  ↓
进程里创建线程 / worker
  ↓
服务器监听用户连接
  ↓
多个用户同时来了，用 IO 多路复用管理连接
  ↓
某个用户请求 /a.jpg
  ↓
服务器去磁盘 / 文件系统找 a.jpg
  ↓
把图片内容读到内存 / 内核缓冲区
  ↓
通过网络 socket 发回用户
  ↓
继续等待下一个请求
```

这就是主线。

八股知识不是凭空来的，它们都是为了解决这条主线里的具体问题。

---

# 二、第一步：我先得写一个程序

你先写一个服务器程序，比如：

```cpp
int main() {
    start_server();
}
```

然后编译成一个可执行文件：

```bash
g++ server.cpp -o server
```

现在磁盘上有一个文件：

```text
./server
```

注意，这时候它还只是一个**静态文件**，还没运行。

就像你写了一份菜谱，但厨房还没开火。

---

# 三、第二步：服务启动，操作系统创建进程

你执行：

```bash
./server
```

这时操作系统会把这个程序加载起来，创建一个**进程**。

## 新手理解

**进程 = 正在运行的程序。**

也就是说：

```text
磁盘上的 ./server 文件
  ↓ 运行起来
内存里的 server 进程
```

它开始拥有自己的资源：

```text
代码
内存
文件描述符
打开的文件
socket
堆
栈
```

所以面试里说：

> 进程是操作系统分配资源的基本单位。

你可以先别背那么抽象，换成人话：

> 操作系统要管理一个正在运行的程序，就给它分一套资源包，这个资源包就叫进程。

这一步对应的八股词：

```text
Process / 进程 / 地址空间 / 文件描述符 FD
```

---

# 四、第三步：服务器不能只服务一个人，所以需要并发

如果服务器一次只能处理一个用户，会发生什么？

假设用户 A 请求一张大图片，服务器正在慢慢发。

这时用户 B、C、D 也来了。

如果没有并发，就会变成：

```text
先服务 A
A 完了再服务 B
B 完了再服务 C
C 完了再服务 D
```

这太慢了。

所以服务器需要同时处理很多事情。

最容易想到的方法是：

```text
来一个用户，开一个线程处理
```

## 线程是什么？

**线程 = 进程内部的一条执行流。**

一个进程里可以有多个线程。

你说得很好，可以先用包含关系理解：

```text
一个进程
  ├── 线程 1
  ├── 线程 2
  ├── 线程 3
  └── 线程 4
```

进程像一个公司，线程像公司里的员工。

公司拥有办公室、电脑、文件柜，这些是进程资源。

员工在里面干活，这些员工就是线程。

所以面试说：

> 线程是 CPU 调度的基本单位。

新手版理解：

> CPU 真正轮流执行的是线程，不是抽象的“整个公司”。

这一步对应的八股词：

```text
Thread / 线程 / 栈 / 寄存器 / CPU 调度
```

---

# 五、第四步：只靠“一个连接一个线程”会出问题

现在你有多线程了，看起来能并发了。

但如果用户很多呢？

比如：

```text
10000 个用户连接
```

如果你给每个连接都创建一个线程：

```text
10000 个连接 → 10000 个线程
```

问题就来了。

## 会有什么问题？

第一，线程太多，占内存。

每个线程都有自己的栈。

```text
线程 1 栈
线程 2 栈
线程 3 栈
...
线程 10000 栈
```

第二，CPU 切换开销大。

CPU 一会儿跑线程 A，一会儿跑线程 B，一会儿跑线程 C。

切来切去也要成本，这叫**上下文切换**。

第三，很多连接其实没事干。

比如用户连接着服务器，但他暂时没发新请求。

你给它单独配一个线程，线程大部分时间都在等。

这很浪费。

---

# 六、第五步：于是出现 IO 多路复用

这个时候才引出 **IO 多路复用**。

你不要先背 select、poll、epoll，先理解它解决的问题：

> 我不想给每个用户都安排一个专属线程。  
> 我想让一个线程同时盯着很多用户连接。  
> 谁有事，我就处理谁。

这就是 IO 多路复用。

可以想象成一个前台服务员：

```text
用户 A 坐着没说话
用户 B 坐着没说话
用户 C 举手说：我要 a.jpg
用户 D 坐着没说话

服务员只处理 C
```

也就是说：

```text
一个线程
  ↓
同时监听很多 socket
  ↓
哪个 socket 有事件
  ↓
就处理哪个 socket
```

这一步对应的八股词：

```text
IO Multiplexing / select / poll / epoll / fd / socket
```

其中 Linux 高性能服务器常用的是：

```text
epoll
```

新手版记忆：

> epoll 就像一个事件通知器。  
> 你先告诉它：“这些连接你帮我盯着。”  
> 后面它告诉你：“这几个连接有事了。”

[[socket是什么]]

# 七、第六步：用户请求 /a.jpg，服务器怎么知道？

用户浏览器访问：

```text
http://server.com/a.jpg
```

本质上会建立一个 TCP 连接。

在服务器程序里，这个连接会表现成一个东西：

```text
socket fd
```

fd 是 **File Descriptor，文件描述符**。

你可以先理解成：

> 操作系统给每个打开的资源发的编号。

比如：

```text
fd 3 -> 监听 socket
fd 4 -> 用户 A 的连接
fd 5 -> 用户 B 的连接
fd 6 -> 用户 C 的连接
```

epoll 监听这些 fd。

当用户 C 发来了 HTTP 请求，epoll 会告诉服务器：

```text
fd 6 可读了
```

服务器就去读：

```cpp
read(fd6, buffer, size);
```

这一步对应：

```text
fd / socket / epoll_wait / read
```

---

# 八、第七步：read 不是普通函数，它会进入内核

这里开始进入操作系统核心。

你的程序运行在**用户态**。

但网络、磁盘、文件系统这些东西归操作系统内核管。

所以你不能自己直接读网卡、读磁盘。

你必须请求操作系统帮你。

这就叫：

```text
系统调用 syscall
```

比如：

```cpp
read(fd, buf, size);
write(fd, buf, size);
accept(...);
sendfile(...);
```

这些表面上像函数，本质上会进入内核。

新手版理解：

> 用户程序不能直接摸硬件。  
> 想用网络、磁盘、文件，就要通过系统调用找内核帮忙。

这一步对应八股词：

```text
用户态 / 内核态 / syscall / read / write / accept
```
- 系统调用是用户程序与操作系统内核之间的接口，必须通过特权模式进入内核态才能访问硬件。
    
- 用户态程序无法直接操作硬件设备，需通过系统调用请求内核进行相应的操作，确保安全性。
    
- 进入内核态后，操作系统可以执行特权操作，如设备驱动程序的调用，确保资源的有效管理。
---

# 九、第八步：服务器去找 a.jpg

现在服务器知道用户要：

```text
/a.jpg
```

它就要去服务器的文件系统里找这个文件。

路径可能是：

```text
/var/www/html/a.jpg
```

这里再次明确：

```text
这是服务器上的文件，不是用户电脑上的文件。
```

如果文件已经在内存缓存里，系统可能不用真的读磁盘。

如果不在缓存里，才需要从磁盘读。

这里会涉及一个很重要的概念：

```text
Page Cache
```

新手版理解：

> 操作系统会把最近读过的文件内容缓存在内存里。  
> 如果下次还要读同一个文件，就可以直接从内存拿，不一定每次都碰磁盘。

所以流程可能是：

```text
请求 a.jpg
  ↓
先看内存里的 Page Cache 有没有
  ↓
有：直接用
  ↓
没有：从服务器磁盘读入内存
```

这一步对应八股词：

```text
文件系统 / 磁盘 IO / Page Cache / 缓冲区
```

# 十、第九步：内存为什么又冒出来了？

因为程序不可能直接拿着“磁盘上的文件”发给用户。

数据通常要进入内存，CPU / 网卡才能进一步处理。

但你的程序看到的内存也不是物理内存，而是**虚拟内存**。

这里先别急着深入页表。

你先这样理解：

> 每个进程都以为自己有一整片独立内存。  
> 但真实物理内存由操作系统统一管理。  
> 虚拟内存就是操作系统给进程制造的“独立内存假象”。

为什么要这样？

因为如果没有虚拟内存，一个进程可能乱改另一个进程的内存。

那系统就乱套了。

所以虚拟内存的初心是：

```text
隔离
安全
方便管理
```

这一步对应八股词：

```text
Virtual Memory / 虚拟内存 / 虚拟地址 / 物理地址 / 页表 / TLB
```

---

# 十一、第十步：如果 a.jpg 对应的数据页不在内存呢？

假设程序要访问某块数据，但这块数据还没加载到物理内存。[[磁盘、物理内存、虚拟内存三者区别与联系|相当于没有虚拟内存到物理内存映射到那张页表]]

这时会发生：

```text
缺页中断 Page Fault
```

新手版理解：

> 程序要用某页数据，发现内存里没有。  
> CPU 暂停一下，让操作系统去把这页数据弄进来。  
> 弄进来之后，程序继续执行。

这不一定是错误。

它可能只是正常的“按需加载”。

这一步对应八股词：

```text
Page Fault / 缺页中断 / 页面置换 / LRU / Clock
```

---

# 十二、第十一步：服务器把 a.jpg 发回用户

传统方法是：

```cpp
read(file_fd, buf, size);
write(socket_fd, buf, size);
```

新手理解：

```text
先把文件读到用户程序的 buffer
再把 buffer 写到 socket
```

但这样会有很多数据拷贝。

高性能服务器喜欢用：

```text
sendfile
```

它属于**零拷贝**相关技术。

这里的“零拷贝”不是完全没有拷贝。

而是：

> 尽量减少数据在用户态和内核态之间来回搬运。 [[用户态和内核态|高性能服务器怎么做？sendfile]]

对于发静态文件，比如 `a.jpg`，sendfile 很适合。

大概流程是：

```text
服务器磁盘 / Page Cache
  ↓
内核直接把数据送到 socket
  ↓
网卡发给用户
```

这一步对应八股词：

```text
Zero-Copy / sendfile / mmap / DMA / Socket Buffer
```

---

# 十三、第十二步：多线程共享数据，会不会出错？

你的服务器可能有共享数据：

```text
连接队列
任务队列
缓存
日志对象
访问计数器
内存池
```

多个线程同时改这些东西，就可能出错。

比如访问计数：

```cpp
count++;
```

看似一行，其实可能是：

```text
读 count
加 1
写回 count
```

两个线程同时操作，结果可能错。

所以需要锁。

这一步对应：

```text
Mutex / RWLock / Semaphore / Condition Variable
```

新手版理解：

> 只要多个线程要改同一个东西，就要考虑同步。  
> 锁的作用就是保证关键区域同一时间只有一个线程进去。

如果锁乱用，还会引出：

```text
死锁
```

比如：

```text
线程 A 拿着锁 1，等锁 2
线程 B 拿着锁 2，等锁 1
```

两个人互相等，谁也走不了。

这一步对应八股词：

```text
临界区 / 互斥锁 / 条件变量 / 信号量 / 死锁
```

# 十四、现在把主线重新串一次

你现在可以这样理解整件事：

```text
我写了一个 server 程序
  ↓
运行它，操作系统创建 server 进程
  ↓
进程里面可以有多个线程负责干活
  ↓
但用户很多，不能一个连接一个线程
  ↓
所以用 epoll 让少量线程监听大量连接
  ↓
某个用户请求 /a.jpg，epoll 通知这个 socket 可读
  ↓
线程调用 read，进入内核读取请求
  ↓
服务器解析请求，发现要服务器磁盘上的 a.jpg
  ↓
操作系统先看 Page Cache，没有就从磁盘读
  ↓
数据进入内核缓冲区 / 内存
  ↓
服务器用 sendfile / write 把数据通过 socket 发回用户
  ↓
网卡把数据发出去
  ↓
连接关闭或保持 Keep-Alive，服务器继续等下一个事件
```

这才是你应该先记住的“故事”。

# 十五、再把专业知识填进去

现在我们把八股词塞回每一步。

| 步骤     | 新手理解           | 填进去的专业词                 |
| ------ | -------------- | ----------------------- |
| 写程序    | 先有一个服务器可执行文件   | 编译、可执行文件                |
| 启动服务   | 程序跑起来，OS 给它资源  | 进程 Process、地址空间、FD      |
| 并发执行   | 不能只服务一个用户      | 线程 Thread、栈、寄存器         |
| 管理连接   | 不能一个连接一个线程     | IO 多路复用、epoll、fd        |
| 谁来运行   | CPU 要决定先跑哪个线程  | CPU 调度、CFS、上下文切换        |
| 进入内核   | 网络和文件都要找内核     | 系统调用、用户态、内核态            |
| 读取文件   | 找服务器磁盘上的 a.jpg | 文件系统、磁盘 IO、Page Cache   |
| 使用内存   | 数据要进内存才能处理     | 虚拟内存、页表、TLB             |
| 页不在内存  | 用到的数据还没加载      | 缺页中断、页面置换               |
| 发回用户   | 把图片通过网络发出去     | socket、sendfile、零拷贝、DMA |
| 保护共享数据 | 多线程不能乱改同一份数据   | mutex、读写锁、信号量、死锁        |

你上传的 OS 框架里也把这些分成了进程线程、虚拟内存、同步互斥、死锁、IO 多路复用、零拷贝、CPU 调度、系统调用与中断这几类；其中 C++ 后端面试里尤其高频的是**内存管理**和**IO 多路复用**。

# 十七、你现在先记这个版本

最简主线：

```text
程序启动成进程，
进程里有线程干活；
用户连接太多，所以用 epoll 管理；

再通过 socket / sendfile 发回用户；
多线程共享数据时要加锁；
整个过程中 CPU 调度线程，内存靠虚拟内存管理。
```
