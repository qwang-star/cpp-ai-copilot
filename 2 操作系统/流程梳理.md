## 3 服务器创建监听socket
服务器程序启动后，即为server进行，创建一个监听socket，并绑定端口
```cpp

bind(listen_fd, server_ip:80);
listen(listen_fd);
```
这里发生了什么？

```
socket()  创建一个网络通信端点
bind()    把它绑定到 IP:端口listen()
开始监听有没有用户连接
```

>Socket 是应用程序接入网络通信的端点。口语化：socket 是服务器程序接入网络的“插口”

监听 socket 的作用是：

```
站在门口等用户来连接
```

## 5 服务器accept，得到连接socket
监听 socket 发现有人来了，服务器调用：

```cpp
conn_fd = accept(listen_fd);
```

这时候会得到一个新的 socket：

```
连接 socket
```

现在服务器里至少有两类 socket：

```
listen_fd：监听 socket，负责等新用户
conn_fd：连接 socket，负责和某个具体用户通信
```
比如：

```
fd 3：监听 socket，负责接新用户
fd 4：用户 A 的连接 socket
fd 5：用户 B 的连接 socket
fd 6：用户 C 的连接 socket
```

人话理解：

> 监听 socket 是门口迎宾，连接 socket 是某个顾客坐下之后对应的一张桌子。

## 7 引入epoll IO多路复用

为了解决“连接很多但线程不能太多”的问题，引出：
```
IO 多路复用
```

核心思想：
> **一个线程同时监听很多 socket fd，哪个 fd 有事件，就处理哪个。**

Linux 高性能服务器常用：

```
epoll
```
- epoll的优势在于其支持边缘触发和水平触发，适应不同的应用场景，优化资源使用。

服务器会把这些 fd 注册给 epoll：

```
fd 3：监听 socket
fd 4：用户 A socket
fd 5：用户 B socket
fd 6：用户 C socket
```

然后调用：

```
epoll_wait(...)
```

epoll 帮你盯着：

```
fd 3 有没有新连接？
fd 4 有没有用户请求？
fd 5 有没有用户请求？
fd 6 有没有用户请求？
```

人话理解：

> epoll 就像前台总管，一个人盯着所有桌子，谁举手就通知服务员。

对应八股知识：

```
IO 多路复用
select / poll / epoll
epoll_ctl
epoll_wait
fd
LT / ET
非阻塞 IO
```
- **能够直接访问磁盘数据，无需内核介入**：这个说法是错误的。IO多路复用并不意味着用户空间可以直接访问磁盘数据。实际上，所有的IO操作仍然需要通过系统调用与内核进行交互，内核负责管理硬件资源和调度IO请求。因此，内核介入是不可避免的。
    
- **仅适用于TCP连接，UDP无法使用**：这个说法也是错误的。虽然IO多路复用在TCP连接中应用广泛，但它同样适用于UDP连接。通过IO多路复用，程序可以同时处理多个UDP数据包的接收和发送，只要相应的socket被正确设置。因此，IO多路复用并不局限于TCP连接。

## 10 线程调用 read，进入内核读取用户请求
服务器线程看到 fd 5 可读，于是调用：

```cpp
read(conn_fd, buf, sizeof(buf));
```

这里就涉及你刚才卡住的：

```
用户态 / 内核态
```

你的服务器程序运行在：

```
用户态
```

但是 socket、网络、网卡这些东西由操作系统内核管理。

所以服务器程序不能直接读网卡，也不能直接拿 socket 里的数据。

它必须通过：

```
系统调用 syscall
```

进入内核态，请操作系统帮忙读。

真实过程是：

```
服务器程序在用户态
  ↓
调用 read
  ↓
通过 syscall 进入内核态
  ↓
内核从 socket 接收缓冲区拿到 HTTP 请求
  ↓
把请求数据拷贝到用户态 buf
  ↓
返回用户态
  ↓
服务器程序继续执行
```

人话理解：

> 你的程序是普通员工，socket 数据在内核管理员手里。你要读数据，必须向管理员申请。

对应八股知识：

```
用户态
内核态
系统调用 syscall
read
socket buffer
用户态 buffer
数据拷贝
```

## 14 操作系统先看Page Cache（内核文件缓存）

现在服务器要读 `a.jpg`。

操作系统不会每次都直接读磁盘。

它会先看：

```
Page Cache 里有没有 a.jpg 的内容
```

Page Cache 是什么？

> **操作系统用物理内存做的文件缓存。**

也就是说：

```
磁盘：长期保存 a.jpg
物理内存：临时缓存正在用的 a.jpg 数据
Page Cache：内核用内存缓存文件内容
```

如果之前有人访问过 `a.jpg`，它可能已经在 Page Cache 里。

那么这次就不用读磁盘，直接从内存拿。

人话理解：

> 仓库里最常用的货，可能已经提前放到前台货架上了，不用每次都跑仓库深处拿。

对应八股知识：

```
Page Cache
物理内存
磁盘 IO
文件缓存
缓存命中
缓存未命中
```

## 15 如果Page Cache没有，就从服务器磁盘读入物理内存

如果 Page Cache 没有 `a.jpg` 的数据，操作系统就要从服务器磁盘读。
流程：

```
服务器磁盘
  ↓
读入物理内存
  ↓
形成 Page Cache
```
在这个文件读取场景里，基本可以这么理解：

> **文件数据长期在磁盘上；如果 Page Cache 没有，就需要从磁盘读到物理内存。**

但是要注意：

```
虚拟内存不是另一块真正的内存
虚拟内存是进程看到的地址空间
物理内存才是 CPU 真正能直接访问的内存条
磁盘是长期保存数据的地方
```

三者关系：

```
磁盘 = 仓库，长期保存 a.jpg
物理内存 = 工作台，CPU 能直接处理的数据在这里
虚拟内存 = 程序看到的地址地图，最终要映射到物理内存
```

人话理解：

> a.jpg 平时躺在仓库，真正要发货时，先搬到工作台上。

对应八股知识：

```
磁盘
物理内存 RAM
虚拟内存
Page Cache
缺页中断 Page Fault
页面置换
LRU / Clock
```

## 16 发送文件
#### 方式 A：普通 read + write

代码类似：

```cpp
read(file_fd, user_buf, size);
write(conn_fd, user_buf, size);
```
数据路径：

```
服务器磁盘
  ↓
内核 Page Cache
  ↓ read：拷贝到用户态 buffer
用户态 buffer
  ↓ write：拷贝回内核 socket buffer
内核 socket buffer
  ↓
网卡
  ↓
浏览器
```

这里的问题是：

```
数据从内核到用户态，又从用户态回到内核
```

多拷贝了一圈。

人话理解：

> 仓库管理员先把货给服务员，服务员再交给快递员。中间服务员其实没必要拿这个货。

对应八股知识：

```
read
write
用户态 / 内核态切换
数据拷贝
socket buffer
```
#### 方式 B：高性能 sendfile

高性能 Web 服务器更喜欢：

```cpp
sendfile(conn_fd, file_fd, &offset, count);
```

意思是：

> 内核，你直接把这个文件发到这个 socket，不要先拷贝到我的用户态 buffer。

数据路径：

```
服务器磁盘
  ↓
内核 Page Cache
  ↓
内核 socket buffer
  ↓
网卡 DMA 发送
  ↓
浏览器
```

这就是零拷贝相关思想。

注意：

> ==零拷贝不是完全没有拷贝，而是减少用户态和内核态之间没必要的数据拷贝。

人话理解：

> 服务员只下命令：把 a.jpg 从仓库直接交给快递发出去。服务员不亲自搬货。

对应八股知识：

```
sendfile
零拷贝 Zero-Copy
DMA
Page Cache
Socket Buffer
TCP/IP 协议栈
```

## 19 多线程共享数据，需要同步
整个服务器运行过程中，多个线程可能会共享一些东西：

```
连接队列
任务队列
日志
缓存
访问计数器
内存池
```

如果多个线程同时修改，就可能出错。

比如：

```cpp
count++;
```

看起来一行，其实可能是：

```
读 count
加 1
写回 count
```

两个线程同时操作，结果可能错。

所以需要：

```
锁 / 同步机制
```

比如：

```
mutex
读写锁
条件变量
信号量
```

但锁用错了可能死锁。

比如：

```
线程 A 拿着锁 1，等锁 2
线程 B 拿着锁 2，等锁 1
```

人话理解：

> 多个服务员抢同一本账本，需要规定谁能写；否则账会乱。锁就是账本的使用规则。

对应八股知识：

```
互斥锁 mutex
读写锁 RWLock
条件变量 condition_variable
信号量 semaphore
临界区

```
