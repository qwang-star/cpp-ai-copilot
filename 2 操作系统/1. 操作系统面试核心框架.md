# 操作系统面试核心框架

> C++ 后端面试中，**内存管理** 和 **IO 多路复用** 是操作系统最高频考点。

---

## 一、进程与线程

### 1.1 核心区别（必背）

| 维度       | 进程 (Process)             | 线程 (Thread)                    |
| -------- | ------------------------ | ------------------------------ |
| **定义**   | 资源分配的基本单位                | CPU 调度的基本单位                    |
| **资源**   | 独立地址空间、文件描述符、信号处理        | 共享进程地址空间，独有栈+寄存器               |
| **通信**   | 管道/FIFO/消息队列/共享内存/Socket | 共享内存（需同步）                      |
| **切换开销** | 大（切换页表、TLB 刷新）           | 小（切换栈和寄存器）                     |
| **崩溃影响** | 一个进程崩溃不影响其他              | 一个线程崩溃可能拖垮整个进程                 |
| **系统调用** | fork() / exec()          | pthread_create() / std::thread |

### 1.2 上下文切换

```
进程切换步骤：
  1. 保存当前进程的寄存器、PC、栈指针 → PCB
  2. 刷新 TLB（切换页表）
  3. 加载新进程的 PCB → 恢复寄存器、PC、栈指针
  4. 返回用户态，继续执行

线程切换（同进程内）：
  1. 保存当前线程上下文（寄存器、栈指针）
  2. 加载新线程上下文（不需要刷新 TLB）
  3. 继续执行
```

> 🔥 面试追问：**进程切换为什么比线程切换慢？** → 因为需要切换页表 + 刷新 TLB。

### 1.3 进程间通信（IPC）

| 方式 | 特点 | 场景 |
|------|------|------|
| **管道 (Pipe)** | 半双工，父子进程 | shell 管道 `ls \| grep` |
| **命名管道 (FIFO)** | 有名字，无关进程 | 简单 C/S |
| **消息队列** | 异步，报文为单位 | 解耦通信 |
| **共享内存** | ⚡ 最快，需同步 | shmget/shmat，配合信号量 |
| **信号量 (Semaphore)** | 同步锁，PV 操作 | 生产者-消费者 |
| **Socket** | 跨网络，通用 | 网络通信 |

> 🔥 面试追问：**共享内存为什么最快？** → 数据不经过内核拷贝，直接读写同一块物理内存。

## 二、虚拟内存 ⭐

### 2.1 为什么需要虚拟内存？

1. **隔离** — 每个进程有独立地址空间，一个进程不能访问另一个进程的内存
2. **安全** — 用户态不能直接访问内核态内存
3. **扩展** — 程序可以使用比物理内存更大的地址空间（交换到磁盘）

### 2.2 分页机制（Paging）

```
虚拟地址 = 页号 | 页内偏移
         │
         ▼ 页表（MMU）
物理地址 = 页框号 | 页内偏移

页大小：通常 4KB（Linux x86_64）
```

### 2.3 页表结构

| 级别 | 结构 | 说明 |
|------|------|------|
| 单级页表 | 连续数组 | 32 位：4MB 页表 / 进程，浪费 |
| 多级页表 | 树形结构 | 64 位：四级页表（PGD→PUD→PMD→PTE），按需分配 |
| **TLB** | CPU 高速缓存 | Translation Lookaside Buffer，缓存最近使用的页表项 |

> 🔥 面试追问：**TLB 是什么？进程切换时 TLB 怎么办？** → TLB 是页表的 CPU 缓存。进程切换时通常需要 **刷新 TLB**（flush），现代 CPU 有 **ASID** 标记避免全量刷新。

### 2.4 缺页中断（Page Fault）

```
程序访问虚拟地址 → 查 TLB（未命中）→ 查页表
  ├── 页在物理内存 → 更新 TLB → 继续
  └── 页不在物理内存 → 缺页中断
       ├── 内存有空闲 → 从磁盘读取 → 映射 → 继续
       └── 内存已满 → 页面置换算法 → 换出一页 → 换入新页 → 继续
```

### 2.5 页面置换算法

| 算法              | 思想             | 缺点            |
| --------------- | -------------- | ------------- |
| **OPT**         | 淘汰未来最远使用的页     | 理论最优，无法实现     |
| **LRU**         | 淘汰最近最少使用的页     | 实现复杂，需要硬件支持   |
| **Clock (NRU)** | 近似 LRU，环形扫描访问位 | Linux 实际使用    |
| **FIFO**        | 淘汰最早进入的页       | 可能有 Belady 异常 |

> 🔥 面试追问：**LRU 怎么实现？** → 双向链表 + 哈希表（就是 LRU Cache），O(1) 操作。

## 三、进程同步与互斥

### 3.1 临界区问题满足条件

1. **互斥** — 同一时刻只有一个进程进入临界区
2. **前进** — 临界区空闲时，想进入的进程必须能进入
3. **有限等待** — 每个进程等待时间是有限的（不能饿死）

### 3.2 锁的层次（从轻到重）

| 锁类型 | 忙等？ | 适用场景 | C++ 对应 |
|--------|--------|---------|---------|
| **自旋锁** | ✅ 忙等 | 临界区极短（几行代码） | `std::atomic_flag` |
| **互斥锁** | ❌ 阻塞 | 临界区较长 | `std::mutex` |
| **读写锁** | ❌ 阻塞 | 读多写少 | `std::shared_mutex` |
| **信号量** | ❌ 阻塞 | 控制并发数 | `std::counting_semaphore` (C++20) |
| **条件变量** | ❌ 阻塞 | 等待条件满足 | `std::condition_variable` |

> 🔥 面试追问：**自旋锁 vs 互斥锁？** → 自旋锁不切换上下文（忙等），适合临界区极短的情况；互斥锁会让出 CPU（阻塞），适合临界区较长的情况。

### 3.3 经典同步问题

**生产者-消费者**（信号量解法）：
```
empty = N    // 空槽位数
full = 0     // 已用槽位数
mutex = 1    // 缓冲区互斥

producer:                consumer:
  P(empty)                 P(full)
  P(mutex)                 P(mutex)
  // 生产                  // 消费
  V(mutex)                 V(mutex)
  V(full)                  V(empty)
```

**读者-写者**（读写锁）：
- 读读不互斥，读写互斥，写写互斥
- 变体：读者优先 vs 写者优先 vs 公平

## 四、死锁

### 4.1 四个必要条件（缺一不死锁）

1. **互斥** — 资源只能独占
2. **持有并等待** — 持有资源的同时等待新资源
3. **不可抢占** — 资源不能被强制剥夺
4. **环路等待** — 进程间形成循环等待链

### 4.2 处理策略

| 策略 | 思想 | 方法 |
|------|------|------|
| **鸵鸟** | 假装没死锁 | 重启大法（实际最常用） |
| **预防** | 破坏四条件之一 | 一次性分配 / 有序分配 |
| **避免** | 分配前预判 | 银行家算法 |
| **检测** | 图算法检测环路 | 资源分配图 + 死锁检测算法 |

> 🔥 面试追问：**怎么看 Linux 有没有死锁？** → 进程状态 `D`（不可中断睡眠）长时间不消失；用 `pstack` + `strace` 看卡在哪里。

## 五、IO 模型 ⭐⭐

### 5.1 五种 IO 模型

| 模型          | 阻塞                      | 描述                         |
| ----------- | ----------------------- | -------------------------- |
| **阻塞 IO**   | ✅                       | 全程等待数据准备好                  |
| **非阻塞 IO**  | ❌ 轮询                    | 立即返回，没数据返回 EAGAIN          |
| **IO 多路复用** | ✅ 阻塞在 select/poll/epoll | 一个线程监听多个 fd                |
| **信号驱动 IO** | ❌                       | 数据就绪时发 SIGIO 信号            |
| **异步 IO**   | ❌                       | 内核完成所有操作后通知（Linux AIO 不成熟） |

### 5.2 select / poll / epoll 对比 ⭐⭐⭐

| 维度           | select          | poll         | epoll                 |
| ------------ | --------------- | ------------ | --------------------- |
| **数据结构**     | 固定长度数组（默认 1024） | 链表           | 红黑树 + 就绪链表            |
| **fd 上限**    | 1024（可改，但性能下降）  | 无上限          | 无上限                   |
| **拷贝次数**     | 每次调用都拷贝 fd_set  | 每次调用都拷贝      | **只需一次拷贝**（epoll_ctl） |
| **就绪 fd 获取** | 遍历所有 fd，O(n)    | 遍历所有 fd，O(n) | **只返回就绪 fd，O(1)**     |
| **触发模式**     | 水平触发            | 水平触发         | 水平触发 + **边沿触发**       |
| **适用场景**     | 少量连接            | 少量连接         | **高并发（C10K/C100K）**   |

### 5.3 水平触发 (LT) vs 边沿触发 (ET)

```
LT（Level Triggered）— 默认模式
  fd 就绪 → 通知 → 如果你没读完 → 下次继续通知
  优点：编程简单，不会漏数据
  缺点：可能重复通知

ET（Edge Triggered）— 高效模式
  fd 从"未就绪"变"就绪" → 通知一次（只有一次！）
  优点：减少系统调用，高效
  缺点：必须非阻塞 IO + 循环读直到 EAGAIN，编程复杂
```

> 🔥 面试追问：**epoll 为什么快？** → ① fd 只拷贝一次（epoll_ctl），不重复拷贝；② 就绪链表直接返回有事干的 fd，不需要遍历全部；③ mmap 映射内核空间减少拷贝开销。

### 5.4 Reactor 与 Proactor

| 模式 | 事件通知 | 数据处理 | 代表 |
|------|---------|---------|------|
| **Reactor** | 数据就绪后通知 | 自行读写 | epoll + 非阻塞 IO |
| **Proactor** | 内核完成读写后通知 | 内核完成 | IOCP (Windows) |

> Linux 下主流是 **Reactor 模式**（epoll），因为 Linux AIO 不够成熟。

## 六、零拷贝技术 ⭐

### 6.1 传统文件传输（4 次拷贝）

```
硬盘 →(DMA) 内核缓冲区 →(CPU) 用户缓冲区 →(CPU) Socket缓冲区 →(DMA) 网卡
  ① DMA拷贝    ② CPU拷贝       ③ CPU拷贝        ④ DMA拷贝
```

### 6.2 sendfile（2 次拷贝）

```
硬盘 →(DMA) 内核缓冲区 →(CPU,只传元数据) Socket缓冲区 →(DMA) 网卡
  ① DMA拷贝        ② 只传描述信息（不拷贝数据）      ③ DMA拷贝

sendfile(fd_out, fd_in, &offset, count);
```

### 6.3 mmap + write（3 次拷贝）

```
硬盘 →(DMA) 内核缓冲区 →(mmap映射) 用户/内核共享 →(CPU) Socket缓冲区 →(DMA) 网卡
  ① DMA拷贝        ② 无拷贝（共享映射）    ③ CPU拷贝      ④ DMA拷贝
```

| 方案 | 拷贝次数 | CPU 参与 | 适用 |
|------|---------|---------|------|
| 传统 read/write | 4 次（2 DMA + 2 CPU） | 大量 | 普通 |
| mmap + write | 3 次（2 DMA + 1 CPU） | 减少 | Kafka 生产者 |
| **sendfile** | 2 次（2 DMA）| 极少 | Nginx、静态文件 |

> 🔥 面试追问：**Kafka 为什么快？** → 使用 mmap + sendfile 零拷贝技术，数据不经过用户态。

## 七、CPU 调度

### 7.1 调度算法速查

| 算法 | 思想 | 特点 |
|------|------|------|
| FCFS | 先来先服务 | 简单，长作业友好，短作业可能饿死 |
| SJF | 最短作业优先 | 平均等待时间最小，但需要预知执行时间 |
| RR | 时间片轮转 | 公平，交互性好，时间片太小→切换开销大 |
| 优先级 | 按优先级调度 | 低优先级可能饿死，需要老化机制 |
| CFS | 完全公平调度 | Linux 默认，红黑树维护 vruntime |

> 🔥 面试追问：**CFS 怎么保证公平？** → 维护每个进程的 `vruntime`（虚拟运行时间），每次选 vruntime 最小的进程运行。红黑树 O(log n) 查找。

## 八、中断与系统调用

```
用户态程序
  │
  │ int 0x80 / syscall 指令（系统调用）
  ▼
内核态 ─── 查系统调用表 ──→ 执行内核函数 ──→ 返回用户态

软中断 ≠ 硬中断：
  硬中断：硬件信号（网卡收包、键盘按下），异步
  软中断：软件触发（系统调用、除零异常），同步
```
