# 操作系统高频面试题精讲

> 每题控制在 3 分钟内回答。先一句话给出结论，再展开 2-3 句细节。

---

## 进程与线程

### Q1：进程和线程的区别？

**一句话**：进程是资源分配单位，线程是 CPU 调度单位。

**展开**：
- 进程有独立地址空间，线程共享进程地址空间
- 进程切换要切换页表、刷新 TLB，线程切换只需切换栈和寄存器
- 进程间通信需要 IPC（管道/共享内存等），线程间直接共享内存（需同步）
- 一个进程崩溃不影响其他进程，线程崩溃可能拖垮整个进程

---

### Q2：进程切换发生了哪些事？

1. 保存当前进程上下文（寄存器、PC、栈指针）到 PCB
2. 切换到新进程的页表 → **刷新 TLB**
3. 从新进程的 PCB 恢复上下文
4. 返回用户态继续执行

> 追问：**线程切换少做了什么？** → 不切换页表，不刷新 TLB。

---

### Q3：进程间通信有哪些方式？哪种最快？

| 方式 | 快慢 | 原因 |
|------|------|------|
| 管道/FIFO | 慢 | 需要内核中转 |
| 消息队列 | 较慢 | 报文需要拷贝 |
| Socket | 慢（本地也走协议栈） | 网络协议栈开销 |
| **共享内存** | ⚡ 最快 | 直接读写同一物理内存，无内核拷贝 |

> **共享内存最快**，因为数据根本不需要经过内核空间拷贝。用 `shmget/shmat` 映射同一块物理内存即可。

---

### Q4：协程是什么？和线程有什么区别？

**一句话**：协程是用户态的"轻量级线程"，由程序自己调度而非操作系统。

| 维度 | 线程 | 协程 |
|------|------|------|
| 调度者 | 操作系统内核 | 用户态调度器 |
| 切换开销 | 系统调用 + 上下文切换 | 函数调用级（几条指令） |
| 内存 | 内核栈 (8MB) | 用户栈 (KB 级) |
| 数量 | 数千个 | 数十万个 |
| 抢占 | 抢占式 | 协作式（主动让出） |

> Go 的 goroutine、Python 的 asyncio、C++20 的 coroutine 都是协程实现。

## 内存管理

### Q5：为什么要用虚拟内存？

1. **隔离** — 每个进程有独立的地址空间，互不干扰
2. **安全** — 用户态无法访问内核态内存
3. **扩大** — 程序可以使用比物理内存更大的地址空间（配合磁盘交换）
4. **简化** — 编译器和链接器不用关心物理内存布局

---

### Q6：malloc 1KB 和 mmap 1MB，底层有什么区别？

| 操作 | 底层分配方式 |
|------|------------|
| `malloc(1024)` (小内存) | **brk()** 系统调用 → 调整数据段末尾 → 在堆上分配 |
| `malloc(1MB)` (大内存) | **mmap()** 系统调用 → 映射一块匿名内存区域 |

> Linux 默认阈值 128KB：小于用 brk，大于用 mmap。
> brk 分配的内存不能在中间释放，容易产生碎片。
> mmap 分配的内存可以独立 munmap 释放。

---

### Q7：什么是内存碎片？内部碎片 vs 外部碎片？

- **内部碎片**：已分配的内存块中未使用的部分（分配 30 字节，实际给了 64 字节）
- **外部碎片**：空闲内存总量足够但都不连续，无法分配给需要大块内存的请求

**解决方案**：
- 内部碎片 → 内存池、slab 分配器（内核）
- 外部碎片 → 紧凑（compaction）、分页机制

---

### Q8：缺页中断 (Page Fault) 的完整流程？

```
1. CPU 访问虚拟地址 → 查 TLB → 未命中
2. 查页表 → 页表项标记为"不在内存"
3. CPU 触发缺页异常 → 进入内核缺页中断处理
4. 内核查找空闲页框
   ├── 有空闲 → 从磁盘读入页面 → 更新页表 → 返回
   └── 无空闲 → 选一页换出（页面置换算法）
        └── 如果页面脏（修改过）→ 写回磁盘
            └── 换入新页 → 更新页表 → 返回
5. CPU 重新执行触发缺页的指令
```

---

### Q9：TLB 是什么？为什么重要？

**一句话**：TLB (Translation Lookaside Buffer) 是页表的 CPU 高速缓存，存储最近用过的虚拟地址→物理地址映射。

- 没有 TLB 时，每次内存访问需要**先查页表** → 变 2 次访问，性能减半
- TLB 命中率通常 > 99%
- 进程切换时需要**刷新 TLB**（除非 CPU 支持 ASID/PCID 标记）

## 死锁

### Q10：死锁的四个必要条件？怎么预防？

| 条件 | 破解方法 |
|------|---------|
| ① 互斥 | 不能破坏（资源天生互斥） |
| ② 持有并等待 | **一次性分配**所有资源 |
| ③ 不可抢占 | 允许抢占（设置超时，强制回收） |
| ④ 环路等待 | **有序分配**（所有进程按同一顺序申请资源） |

> 实际工程中最常用的是**鸵鸟策略** + 超时重试 + 监控告警。

---

### Q11：银行家算法是什么？

一种**死锁避免**算法。在每次资源分配前，先模拟分配后是否处于安全状态（存在一个安全序列让所有进程都能完成）。

```
假设有进程 P0, P1, P2
资源总量: A=10, B=5, C=7

分配前检查: 
  Available + 各进程的 Max-Allocation 是否足够
  → 找到安全序列 (P1, P3, P0, P2) → 可以分配
  → 找不到安全序列 → 拒绝分配
```

> 实际很少用（需要预知最大需求，开销大）。

## IO 多路复用

### Q12：select、poll、epoll 的区别？（最高频 ⭐⭐⭐）

**一句话对比**：select 和 poll 每次都要传全部 fd（O(n)），epoll 只传一次（O(1)）。

| | select | poll | epoll |
|------|--------|------|-------|
| fd 限制 | 1024 | 无上限 | 无上限 |
| fd 拷贝 | 每次拷贝全部 | 每次拷贝全部 | **只拷贝一次**（epoll_ctl） |
| 就绪检测 | 遍历全部 O(n) | 遍历全部 O(n) | **就绪链表 O(1)** |
| 触发方式 | LT | LT | **LT + ET** |
| 适用 | 几十个连接 | 几百个 | **数万连接（C10K）** |

---

### Q13：epoll 为什么这么快？（常问 ⭐⭐⭐）

1. **fd 只拷贝一次** — `epoll_ctl()` 把 fd 注册到内核红黑树，之后 `epoll_wait` 不需要再传 fd
2. **就绪链表直接返回** — 内核把就绪 fd 加入一个双向链表，`epoll_wait` 只返回就绪的 fd（不用遍历全部）
3. **mmap 映射** — epoll 用 mmap 在用户态和内核态间共享一块内存，减少拷贝

> 一句话：**选 epoll 不选 select/poll 的三个理由：fd 只拷贝一次、只返回就绪的 fd、支持 ET 模式。**

---

### Q14：LT 和 ET 的区别？ET 下怎么编程？

| | LT（水平触发）| ET（边沿触发）|
|------|------------|------------|
| 通知频率 | 就绪就通知（可重复） | 从"未就绪→就绪"时通知一次 |
| 编程难度 | 简单 | 较难 |
| 性能 | 有冗余通知 | 高效 |

**ET 模式下的正确姿势**：
```c
// 1. fd 必须设为非阻塞
fcntl(fd, F_SETFL, O_NONBLOCK);

// 2. 读的时候循环读到 EAGAIN
while (1) {
    n = read(fd, buf, sizeof(buf));
    if (n < 0) {
        if (errno == EAGAIN) break;  // 读完
        // 真正的错误处理
    }
    // 处理数据
}
```
> 为什么必须非阻塞？因为 ET 只通知一次，如果读阻塞了，可能永远错过新数据。

---

### Q15：Reactor 和 Proactor 的区别？

| | Reactor | Proactor |
|------|---------|---------|
| 事件通知 | 数据**就绪**了，你来读 | 数据**读完**了，这是结果 |
| 数据读写 | 应用程序自己做 | 内核完成 |
| 代表 | **epoll** (Linux) | IOCP (Windows) |
| 模式 | 同步非阻塞 | 真正异步 |

> Linux 下主流的网络库（libevent、libuv、muduo）都是 Reactor 模式。

## 零拷贝

### Q16：零拷贝是什么？为什么需要？

**一句话**：避免数据在用户态和内核态之间来回拷贝，减少 CPU 参与。

传统文件发送需要 **4 次拷贝**（2 次 DMA + 2 次 CPU）。

| 方案 | 拷贝次数 | CPU 参与 |
|------|---------|---------|
| 传统 read+write | 4 次 | 2 次 |
| mmap+write | 3 次 | 1 次 |
| **sendfile** | **2 次** | **0 次**（全是 DMA） |

> **场景**：Nginx 用 `sendfile` 发送静态文件，Kafka 用 `mmap` + `sendfile` 实现零拷贝。

## 锁与同步

### Q17：自旋锁和互斥锁的区别？

| | 自旋锁 | 互斥锁 |
|------|--------|--------|
| 等不到锁时 | 忙等（while 循环） | 阻塞等待（让出 CPU） |
| CPU 消耗 | 高 | 低（不用占着 CPU 空转） |
| 上下文切换 | 不切换 | 切换（开销大） |
| 适用 | 临界区**极短**（几行代码） | 临界区较长 |

> **口诀**：短临界区用自旋锁（省去切换开销），长临界区用互斥锁（不占着 CPU 浪费）。

---

### Q18：读写锁适用于什么场景？

**读多写少**的场景。允许多个读者同时持有锁，但写者是互斥的。

| | 读锁 | 写锁 |
|------|------|------|
| 读锁 + 读锁 | ✅ 兼容 | ❌ 互斥 |
| 读锁 + 写锁 | ❌ 互斥 | ❌ 互斥 |
| 写锁 + 写锁 | ❌ 互斥 | ❌ 互斥 |

> C++17: `std::shared_mutex`，`shared_lock`（读），`unique_lock`（写）。

## 文件系统

### Q19：硬链接和软链接（符号链接）的区别？

| | 硬链接 (Hard Link) | 软链接 (Symbolic Link) |
|------|-------------------|----------------------|
| 本质 | 同一个 inode 的多个目录项 | 指向路径的单独文件 |
| 跨文件系统 | ❌ 不能 | ✅ 可以 |
| 删原文件后 | 硬链接仍然可用 | 软链接失效（悬空） |
| 命令 | `ln f1 f2` | `ln -s f1 f2` |

> `f1` 和 `f2` 是硬链接的话，它们共享同一个 inode。删除 `f1` 只是减少引用计数，inode 还在。

---

### Q20：inode 是什么？

**一句话**：inode 是文件系统存储文件元数据的结构体，每个文件一个。

**包含**：文件大小、权限、时间戳、数据块指针。**不包含**：文件名（文件名在目录项里）。

> 为什么 `ls -l` 很快但 `du` 很慢？→ `ls` 只读 inode 的 size 字段，`du` 要遍历所有数据块。
